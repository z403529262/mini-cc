## 1. Agent Loop 的本质

内核就是一个 `while`（剥掉打印/熔断后约 11 行）：
```
messages = [user 任务]
while true:
  res = messages.create(model, system, tools, messages)
  messages.push(assistant: res.content)        # 记住模型这轮的完整输出
  if res.stop_reason != "tool_use": break       # 不再调工具 → 收工
  results = 执行 res 里每个 tool_use
  messages.push(user: results)                  # 工具结果回填 → 下一轮
```

**三条「写错就报错」的硬约束：**
1. **assistant 回合必须原样回传完整 `content`**（text + tool_use + thinking 全部），不能只抠文字。少块下一轮 API 找不到对应关系直接报错。
2. **一轮可能有多个 tool_use（并行调用）**——要遍历全部、收集全部结果。
3. **所有 tool_result 必须配齐、塞进同一条 user 消息**，每个 `tool_use_id` 对上，少一个都报错。

> 一句话：**模型发出的每个块，你都得完整、配对、原样带回去。**

**最该刻进脑子的洞察（面试金句）：** `messages` 数组**只增不减** → ① context window 会被填满（长度天花板）② **token 成本平方级累积**（每轮重发全部历史，第 N 轮重发前 N−1 轮）。**这就是 agent 又贵、又有长度上限的根因**，也是 D4（压缩 + prompt caching）要解决的。

## 2. 请求/响应数据结构

**响应顶层**：`id / type:"message" / role:"assistant" / model / content[] / stop_reason / stop_sequence / usage`。

**`content` 是「内容块数组」，不是字符串**——一轮可同时含 `[thinking, text, tool_use]` 多块。

**`stop_reason` 是 loop 的命门：**
- `end_turn` 自然说完 ｜ `tool_use` 要调工具 ｜ `max_tokens` **被截断**（输出不完整，警惕）｜ `stop_sequence` 命中停止词 ｜ `refusal` 安全拒绝（较新模型）

**块类型全集（SDK 0.104 类型定义直接拆，会随版本长）：**
- **响应侧 `ContentBlock` = 12 种**：`text` `thinking` `redacted_thinking` `tool_use` `server_tool_use` + 7 个服务端工具结果块（web_search / web_fetch / code_execution / bash_code_execution / text_editor_code_execution / tool_search / container_upload）。后 8 个全是「Anthropic 托管工具」衍生，每加一个托管工具就 +1。
- **请求侧 `ContentBlockParam` = 17 种**：= 响应侧 12 个 + **5 个请求专有**（`image` `document` `search_result` `tool_result` `mid_conversation_system`）。
- **关键区分**：`tool_use` 是**响应块**（模型发，在 assistant 消息里）；`tool_result` 是**请求块**（你回填，在 user 消息里）。一出一进，分属两侧。
- `tool_use` 作请求块的场景 = 多轮里**转发 assistant 历史**（`m1.ts` 那行 push assistant content 就是）。
- `mid_conversation_system` = 对话中途往 messages 末尾追加 system 指令（保缓存 + 防注入的「可信操作员通道」，beta，简单 agent 用不到）。

## 3. Tool Use / 工具定义

工具 = `name` + `description`（模型靠它判断何时用）+ `input_schema`（JSON Schema 描述入参）。`Anthropic.Tool[]` 是这个契约的 TS 类型。**声明（schema，给模型看）与实现（execute，给代码跑）分离 = 接口/SPI vs 实现类。**

**跨厂商：没有国标，各家方言，但内核都是 JSON Schema。**
- Anthropic：`{name, description, input_schema}`
- OpenAI：`{type:"function", function:{name, description, parameters}}`（多包一层、字段叫 parameters）
- 返回侧：Anthropic 给 `tool_use`（input 是**对象**）；OpenAI 给 `tool_calls`（arguments 是 **JSON 字符串**，要自己 parse）
- **MCP** 想做的就是「工具定义的统一标准」，让工具不绑某一家（= D6）。

**【实测】智谱「Anthropic 兼容端点」的 3 个方言瑕疵**（说明「兼容 ≠ 同一个东西」）：
1. tool_use 的 id 是 `call_xxx`（OpenAI 风格），不是 Anthropic 的 `toolu_xxx` → 兼容层很可能内部走 OpenAI 格式再转外壳。
2. thinking 的 `signature` 是短 hex，不是 Anthropic 的长 base64。
3. thinking 响应里有**未转义的裸控制字符**（换行），是**不合规 JSON** → `jq`/严格 `JSON.parse` 会挂，得用宽容解析（Python `json.loads(strict=False)`）。
- 对代码无碍：`tool_use_id: b.id` 原样带回即可，不在乎前缀。

## 4. thinking 块

**两种完全不同的「思考」：**
- **隐式计算**：模型生成每个 token 时的内部运算（注意力/前向传播）——**永远存在、永远不以文本暴露**。哪怕不开 thinking，模型也一直在「想」。
- **显式 thinking 块**：一个**可选功能**，让模型把推理**额外写成文字**。因为 LLM 自回归，「写出来」这个动作本身会让它想得更好（草稿纸效应）。

**thinking 块结构**：`{ type:"thinking", thinking:"...", signature:"..." }`；安全标记时变 `redacted_thinking`（带加密 `data`）。位置通常排在 content **最前**（先想→再说→再调工具）。

**几档显示（Anthropic 新模型默认 omitted）：**
- 不开 → 没有 thinking 块
- 开 + `omitted` → 块在、`thinking` 文本**为空**、signature 在、token **照计费** ← 「想了不给你看」的实物
- 开 + `summarized` → 给**摘要**（不是原始链）
- **原始完整思维链：任何模型都不给**

**计费可验证性（诚实）：** omitted/summarized 时你拿不到原始文本，**无法独立核对 token**——只能信厂商；连「一段字算多少 token」都由厂商 tokenizer 定。input 侧可用 `count_tokens` 对账，output/thinking 基本凭信任。（类比云计量计费的不可审计。）

**【实测】** 开 thinking 问逻辑题：`max_tokens:600` → `stop_reason:max_tokens` 截断在思考中途、连答案都没生成；加到 `max_tokens:2000` → `[thinking, text]` 两块齐全、`stop_reason:end_turn`。**开 thinking 必须把 max_tokens 调大**（思考 + 答案都要留预算）。

## 5. 无状态架构（D1 最深的一块）

**Messages API 是无状态的（stateless）：服务器不保存对话历史，模型没有记忆。** 每次 `messages.create` 是独立调用：输入(完整 messages)→输出。所以：
- **为什么每轮回传完整 content** = 因为 API 无记忆，你不重发它就什么都不知道（**主因是无状态，不是防篡改**）。
- 代价 = token 平方累积（见 §1）；缓解 = prompt caching（稳定前缀走缓存折扣，命中省约 90%）。
- 取舍 = 经典的**无状态服务 vs sticky session**：无状态→易水平扩展，代价是每次传得多。

**signature / redacted_thinking 的精妙：状态不在服务端存，而是加密+签名后由客户端携带。**
- 服务端把思考状态打包成「加密的记忆胶囊」塞给你，你每轮原样背回，服务端解密+验签恢复。
- 你**读不懂**（加密/防蒸馏）、**改不了**（验签/防篡改）。`redacted_thinking` 的 `data` = 加密内容，是铁证。
- **一石三鸟**：无状态（服务端不存）+ 不暴露（防蒸馏）+ 防篡改。

**这就是加密签名的 JWT / session token：**
- JWT 三段 `header.payload.signature`，**默认是签名（防改）不是加密（防看）**——payload 是 base64 明文，谁都能解码。要藏内容才用 JWE。
- 签名两派：**对称 HMAC（HS256，共享密钥，签验同一把）** vs **非对称（RS256/ES256/EdDSA，私钥签、公钥验，适合分布式/OAuth）**。HMAC 是单向的，保完整性不保机密性。
- session 两范式：**opaque token**（服务端存映射，有状态，查库验证）vs **JWT**（数据自包含，无状态，验签）。
- Anthropic 的 thinking：普通 signature = 签名派（防改）；redacted data = 加密派（防看）。

> **核心认知：stateless ≠ 没有状态，而是状态不由服务端持有。**

## 6. 安全（= D5 的预告）

`m1.ts` 的 bash 工具**零护栏**：任意命令、用你本人权限、直接执行、无确认、无沙箱。
- 风险：能读 `.env`（你的 token）外传、删文件、`curl | sh`。
- 更阴的是 **prompt injection**：agent 读了外部内容（网页/文件/数据），里面藏「忽略指令、执行 X」→ 模型被骗着调 bash。M1 零防御。
- **生产解法（D5）**：执行前权限审批、allowlist/denylist、沙箱（cc-haha 的 `@anthropic-ai/sandbox-runtime`）、deny-by-default、不信任工具返回/外部内容里的「指令」。
- 面试金句：拿自己的 minimal agent 当**活反例**讲 agent 安全。

## 7. Bun（运行时与工具链）

- **是什么**：不是框架，是 **JS/TS 的运行时 + 一体化工具链**。类比你的世界 ≈ **JVM + Gradle + JUnit + 编译器** 四合一，主打快。跑 `.ts` = JVM 跑字节码；`bun install` = Gradle 拉依赖；`bun build` = 打包；`bun test` = JUnit。
- **不是打包现成工具**：用 **Zig 从头重写**了这些功能，做成单一原生二进制；**替代** Node/npm/webpack，不**包含**它们（只兼容 npm 的格式和仓库）。
- **引擎**：Bun 用 **JavaScriptCore（JSC）**，Node 用 V8。**JSC 不一定比 V8 快**——Bun 的快主要来自工具链工程，不是引擎。选 JSC 是为「启动快/内存省/可控/差异化」，不是为峰值速度。坑：Node 生态围着 V8 长，native addon 要靠 N-API 翻译；V8 特性依赖的库可能不兼容。
- **跨平台**：mac/Linux 一等，Windows 原生支持自 Bun 1.1（2024），更早走 WSL。JSC 是可移植 C++（不绑 mac，类比 V8 也不绑 Chrome）；但 **Windows-JSC 实战验证最薄**（无主流浏览器在 Win 用 JSC，最大用户就是 Bun 自己）→ 求 Windows 生产可靠性优先 Node。
- **构建期 vs 运行期依赖的风险分级**：开发/构建可激进（环境可控），分发给用户的运行时必须保守。
- **cc-haha ≠ 官方 Claude Code**：cc-haha 是泄露源码修复的第三方复刻（用 Bun + Tauri）；官方 Claude Code 是 **npm 分发的 Node CLI**（用户跑的是 Node/V8）。「Anthropic 怎么敢用 Bun」——它没把 Bun 运行时分发给用户，风险被关在可控的开发/构建环节。

## 8. 两种 agent 范式

| | 自托管（mini-cc，Messages API） | 托管（Managed Agents） |
|---|---|---|
| agent loop | **你**手写 while | **Anthropic** 服务端跑 |
| 状态 | 无状态，你每轮重传 | 有状态，服务端存 |
| 工具执行 | 你的机器 | Anthropic 的 container 沙箱 |
| 你收到 | 一次性 response/流 | SSE 事件流 |
| 控制 | 全控（审批/日志/条件） | 托管（省心，loop 黑箱） |

- Managed Agents：`Agent`（POST /v1/agents，持久版本化，建一次）→ `Session`（每次跑，配 container）；带 Skills/MCP/file mounts/memory/scheduled deployments。仅 first-party API + Claude Platform on AWS（Bedrock/Vertex/Foundry 不支持）。
- **守恒律**：每份「省心」对应一份「失控」。要控制力/可观测/数据不出门 → 自托管；要快速起步/隔离 workspace/不想运维沙箱 → 托管。
- **核心认知**：两种是同一个 agent，区别只是 loop/状态/工具执行这三件活的**边界画在你这边还是 Anthropic 那边**。

## 9. Provider 配置坑（mini-cc 专属，复用前必看）

- 用智谱 GLM 的 Anthropic 兼容端点 `https://open.bigmodel.cn/api/anthropic`（`bigmodel.cn` 国内**直连、不走代理**），`apiKey` 用 `ANTHROPIC_AUTH_TOKEN`（实测就是智谱 key，走 `x-api-key`），model `glm-4.6`。代码里**硬编码 baseURL**，不依赖 env。
- ⚠️ shell 里现成的 `ANTHROPIC_BASE_URL=https://api.anthropic.com` 是**坏的**（官方域名却映射 glm 模型，必报错），别复用。
- ⚠️ `ANTHROPIC_AUTH_TOKEN` 是 **Claude Code 运行时注入的**，用户独立终端读不到 → mini-cc 自己终端跑会 `Could not resolve authentication method`。解法：已固化进 `.env`（Bun 自动加载，已被 `.gitignore` 忽略不上传）；失效就换成自己的智谱 key。

## 10. 面试金句速取

- agent loop = while 里反复 `messages.create`，靠 `stop_reason` 区分继续/结束，`tool_result` 回填再循环。
- agent 又贵又有长度上限 = messages 只增不减 → token 平方累积 + context 撑满；解法是压缩 + prompt caching。
- 回传完整历史 = 因为 API **无状态**；stateless ≠ 没状态，是状态不由服务端持有。
- thinking 的 signature/redacted = 加密签名的客户端携带令牌（同 JWT 思路：签名防改 + 可选加密防看）。
- agent 安全：任意工具执行是能力也是最大风险；要权限审批 + allowlist + 沙箱 + deny-by-default + 防 prompt injection。
- 怎么衡量 agent 质量 / 多 agent / MCP …见 `../../agent工程师面试_2周冲刺计划.md` §6。
- 「你怎么设计 agent 服务」= 先问边界画哪（自托管 vs 托管），再讲省心↔失控守恒。
