> 定位:我们的深度全在 Anthropic 形态一侧,但面试官很可能是 OpenAI 视角或框架视角。这份卡补横向坐标系——**知道全景,"手写"才读作选择而不是无知**。
> 事实核验日:2026-07-06(联网核验,非凭记忆;标 ★ 的数字面试前一天建议再查一次)。

## 一、OpenAI vs Anthropic API 对照

| 维度 | Anthropic Messages | OpenAI Chat Completions | OpenAI Responses |
|---|---|---|---|
| 状态 | 无状态,历史自己带 | 无状态 | **可有状态**:server 存对话(Conversations 对象) |
| system | 顶级 `system` 字段 | system/developer role 消息 | `instructions` 参数 / Prompt 对象 |
| 工具结果回填 | `tool_result` block 放**user 消息里** | `role:"tool"` 独立消息 | `function_call_output` item |
| 强制工具 | `tool_choice:{type:"tool",name}` | `tool_choice` 指定 function | 同类机制 |
| 结构化输出 | tool 强制 schema / output_config | `response_format` json_schema strict | text.format |
| 内置 server 工具 | web search / code execution / computer use | 基本无 | deep research / **MCP** / computer use / file search |
| 推理 | thinking block(可 interleaved) | o 系 reasoning_effort | reasoning token **跨轮保留**(GPT-5 起) |
| caching | 显式 `cache_control` 打点 | 自动前缀缓存 | 自动前缀缓存 |

**三条主线叙事(比表格更重要)**:
1. **OpenAI 三条产品线的迁移**:Chat Completions(存量最大)→ Responses(2025-03 推出,**主推,token 用量已超过 Chat Completions**)→ Assistants API **2026-08-26 停服**(2025-08-26 官宣,一年缓冲)。概念映射:Assistants→Prompts(dashboard 版本化)、Threads→Conversations(存 item 不只存消息)、Runs→Responses。
2. **范式在收敛**:两家都往"server 状态 + 内置工具 + agent 原语"走;MCP 已被 OpenAI 接入 Responses 内置工具 → 事实标准(Anthropic 发起、对手采纳,这是协议成功的铁证)。
3. **我的立场答法**:"我主力用的是智谱 GLM 的 **Anthropic 兼容端点**——Messages API 形态本身成了兼容标准,这就是我选它深挖的原因之一。OpenAI 侧的差异我清楚(上表),迁移是形态转换不是重学。"

**追问防守**:
- *"Responses 的有状态和你手带 messages 哪个好?"* —— server 状态省心但锁供应商、调试黑盒;自管 messages 麻烦但可压缩、可持久化、可跨供应商。我的 compact/caching 全套都建立在"历史我可控"上——**上下文工程的前提是你拥有上下文**。
- *"Anthropic 为什么坚持无状态单端点?"* —— 哲学差异:把编排留给开发者(所以才有 Claude Code/Agent SDK 这条产品线),OpenAI 把编排收进平台。

## 二、Agent 框架 landscape + "为什么手写"

**报菜名(2026 主流七家,LangChain 官方 2026-06 也在横评这个名单)**:

| 框架 | 一句话定位 | 记忆钩子 |
|---|---|---|
| **LangGraph** | 图状态机:节点/边/checkpoint,持久化与 human-in-loop 是核心卖点 | 我的 loop 就是它的退化形态:节点=轮,边=stop_reason 分支,checkpoint=messages 可序列化 |
| **OpenAI Agents SDK** | 轻量三概念:agents / handoffs / guardrails(Swarm 的生产化后继) | handoff ≈ 我的 task 工具换了个方向:平级转交 vs 父子派遣 |
| **Microsoft Agent Framework** | AutoGen + Semantic Kernel 合并后的企业系 | 名字直译:微软把两个实验框架合成一个生产框架 |
| **CrewAI** | role-based 多 agent 协作剧本(crew=角色分工) | orchestrator-worker 的"角色扮演"封装 |
| **Claude Agent SDK** | 把 Claude Code 内核(loop/工具/权限/子 agent)直接开放成 SDK | **mini-cc 就是它的手写最小版——这是我的原生地盘** |
| **Dify** | 开源 LLMOps 平台:可视化编排+RAG+插件生态(私有化部署首选) | 平台派,见下节 |
| **Coze Studio** | 字节 2025-07 开源的零代码 agent 平台(Go 微服务);配套 CozeLoop 管 tracing/评估 | 国内面试大概率被问 |

**"为什么手写不用框架?"——正面三段答法(此题必被问,答好是加分答不好是减分)**:
1. **目的不同**:我的目标是吃透内核,而框架帮你做掉的事(loop、压缩、权限、子 agent)恰恰就是面试和生产中真正难的事。手写完我能说清每一层为什么存在——这是用框架十次也得不到的。
2. **我承认框架的真价值**:状态持久化(LangGraph checkpointer)、可观测、团队协作约定、被验证过的边角处理。这些我在 mini-cc 里都有对应的最小实现,所以我知道它们各自在解决什么问题——**评价框架的资格来自实现过它解决的问题**。
3. **生产选型不宗教化**:小团队快速验证 → 平台(Dify/Coze);重定制强工程 → 代码框架(LangGraph / Agents SDK);深度绑 Claude 生态 → Claude Agent SDK。我按"团队水平 × 定制深度 × 生态绑定"选,不按信仰选。

**国内平台题("和 Coze/Dify 比,自研价值在哪?")**:
- 格局:Coze Studio(开源后,零代码上手最快)/ Dify(开源 LLMOps,私有化部署最强)/ 百炼(阿里云生态)/ 千帆(百度)/ 元器(腾讯)——低代码平台已经把"搭一个能用的 bot"打成白菜价。
- 答法:**agent 是产品核心就自研,是提效工具就上平台**。平台的天花板在:深度定制(改 loop 行为/上下文策略)、调试黑盒(坏在哪一轮看不见)、成本精控(caching/压缩策略不可自定义)、数据与供应商锁定。我做的这套(压缩/缓存/权限/eval)正是平台黑盒里的东西——**用平台的人遇到天花板时,需要我这种知道盒子里是什么的人**。

## 三、协议:MCP vs A2A

| | MCP | A2A |
|---|---|---|
| 解决什么 | agent ↔ 工具/数据(纵向能力接入) | agent ↔ agent(横向跨 agent 协作) |
| 发起与治理 | Anthropic 2024-11 发布;OpenAI/Google 均已采纳 | Google 2025-04 发布,捐给 Linux Foundation |
| 2026 现状★ | 事实标准,两大对手阵营都内置 | 一周年(2026-04):**150+ 组织**,Google/Microsoft/AWS 云平台集成,供应链/金融/保险/ITOps 有生产部署 |
| 核心概念 | tools/resources/prompts,stdio/HTTP 传输 | Agent Card(能力发现)、任务生命周期、跨厂商语义协商 |
| 我的锚点 | **手写过两端**(client 130 行 + server),D6 全套 | 报菜名 + 概念,没实现过(诚实说) |

一句话关系:**互补不竞争**——MCP 管"一个 agent 能用什么",A2A 管"多个 agent 怎么互相找到并派活"。我的 M7 子 agent 是进程内父子调用;A2A 把这件事标准化到跨进程、跨厂商——概念上就是"把 task 工具的 prompt 下发/结论回收,变成有服务发现的网络协议"。

## 四、Agent benchmark 名录卡

> 用法三步:报名字 → 说各测什么 → 套 D9 三问(谁做的/对照公平吗/饱和了吗)。★数字核验于 2026-07-06,面试前一天再查一次榜。

| benchmark | 测什么 | 2026 锚点★ |
|---|---|---|
| **SWE-bench Verified** | 500 个人工筛选的真实 GitHub issue 修复(编码 agent 金标准) | 头部 88~95%(Opus 4.8 88.6%,前沿模型 95%)→ **接近饱和** |
| **OSWorld(-Verified)** | 桌面 GUI computer use(短程) | 头部 ~85%,前三名差距 1.6 分 → 也近饱和 |
| **OSWorld 2.0**(2026 新) | 108 个长程真实工作流:人类中位 1.6 小时、平均 318 次工具调用 | **榜首仅 20.6% 完成率**(Opus 4.8 max thinking) |
| **τ-bench / τ²** | 客服场景:工具调用 + 政策规则遵守 + 模拟用户多轮 | "设计客服 agent"题的现成评估依据 |
| **GAIA** | 通用助手:多步推理 + 工具组合 | 通用 agent 常引 |
| **WebArena / BrowseComp** | 浏览器 agent:真实网站任务 / 深度检索 | 浏览器赛道 |
| **Terminal-Bench** | 终端环境任务 | CLI agent 赛道 |

**最值钱的一段话(饱和叙事,直接背)**:
"SWE-bench Verified 和 OSWorld 旧榜 2026 年都逼近饱和了——头部 88~95%。但同一批模型放到 OSWorld 2.0 的长程任务上,**榜首只有 20.6%**。这个落差说明两件事:一,榜单饱和不等于能力到顶,是题目到顶了(D9 说的'数据集挑没挑'反过来用);二,当前 agent 的真实短板不在单步操作,而在**长程**——OSWorld 2.0 总结的失败模式是:丢约束、漏掉中途到达的新信息、瞎猜而不是问用户、跳过验证。这四条正好是我在 mini-cc 里用 maxTurns 熔断、权限门问人、eval 判据设计去对抗的东西——玩具尺度和前沿尺度,病是同一种。"
