# D8 — 评估与可观测：mini-cc × cc-haha 对照

> 双线螺旋第 8 站。一句话：**eval 把"凭感觉"变成"有数字"。**
> 配套代码：`demo/eval-tasks.ts`(5 任务集) · `demo/eval.ts`(harness + LLM-judge)

---

## 第一部分：是什么 / 为什么

### 1. 是什么

**eval(评估)** = 给 agent 一组**固定任务** + 每个任务的**成功判据**，批量跑、统计**通过率 / 轮数 / token**。说白了就是 agent 版的「单元测试 + 性能基准」。

### 2. 为什么（含反例）

没有 eval，agent 开发就是**凭感觉**。最典型的反例:你改了一句 system prompt、或为省钱把模型从 Sonnet 换成 Haiku——**自以为优化了**,但某几类任务的成功率悄悄从 90% 掉到 70%,**没有 eval 你根本发现不了**,直到线上炸了。

> **eval 是 agent 工程从「手艺」变「工程」的分界线。** 面试高频:"你怎么衡量一个 agent 好不好""改 prompt 怎么防退化""该不该换模型省钱"——答不出 eval,暴露你只做过 demo、没做过生产。

### 3. agent 评估难在哪（面试深水区）

传统单测是 `输入 → 确定输出`,一行 `assert` 搞定。agent 有**三个特性**让它没这么简单:

1. **非确定性**——同一个任务,模型这次先 `glob` 再 `read`、下次直接 `grep`,**过程不一样**。
2. **多种正确答案**——"把函数改个名"有好几种合理改法,没唯一标准。
3. **多轮 + 有副作用**——它真的改文件、跑命令,判据得检查这些副作用。

于是判据判「任务有没有**达成**」(看最终文件状态 / 输出),而非「过程一不一样」。两条路(都得会讲):
- **程序化断言**:查文件状态 / 输出关键词。**确定、快、免费**,但只能判"结果对不对"。
- **LLM-as-judge**:让另一个 LLM 按 rubric 打分。能判**"质量好不好"**,但**慢、贵、judge 本身也会错**。

## 第二部分：mini-cc 怎么做

### 4. 任务集：判「达成」不判「过程」（`demo/eval-tasks.ts`）

5 个任务覆盖 write / edit / glob+grep / mcp / task,把 M2–M7 的能力都 eval 到。每个 `{ name, dir, prompt, setup?, check }`,关键在 `check` 用**宽松匹配**对抗非确定性:

```ts
{
  name: "T2-edit",
  prompt: `把 ${t2}/calc.ts 里的加法改成减法，其余不变。`,
  setup: () => { freshDir(t2); writeFileSync(.../calc.ts, "return a + b") },
  check: () => {                                   // ← 查文件最终状态，不管它怎么改的
    const code = read(join(t2, "calc.ts"))
    return { pass: code.includes("a - b") && !code.includes("a + b"), ... }
  },
}
```

每个任务 `setup` 先 `freshDir` 清空自己的 `demo/scratch/eval-tX/` ——**eval 必须可重复**,残留会让结果不可信。

### 5. harness：runAgent 的第二次复用（`demo/eval.ts`）

eval 要"批量跑 N 个 agent 并聚合指标",正好就是 D7 抽出的 `runAgent`(可注入 client + 返回 `turns`/`usage`)的用武之地——**继 task 之后,再证一次抽象对了**:

```ts
for (const task of tasks) {
  task.setup?.()
  const result = await runAgent({
    client, model, system, tools: allTools,
    messages: [{ role: "user", content: task.prompt }],  // 每任务独立 messages，互不污染
    approve: () => true, maxTurns: 12, verbose: VERBOSE,
  })
  const verdict = task.check(result.text)
  rows.push({ name, pass: verdict.pass, turns: result.turns, tokens: result.usage.fresh + result.usage.cached })
}
// 报告：通过率 + 平均轮数 + 总 token —— 最朴素的成本可观测
```

### 6. LLM-judge 演示：程序化判「有没有」、judge 判「好不好」

对 T1 额外跑一个裸 LLM(不带工具)按 rubric 打分,和程序化断言对照:

```ts
async function judge(taskDesc, artifact) {
  const res = await client.messages.create({  // 裸 create，无工具
    system: "你是严格的代码评审。按 rubric 打 1-5 分，只输出一行：分数/5 — 理由。",
    messages: [{ role: "user", content: `任务：${taskDesc}\n产出：${artifact}\nrubric：正确性/类型/可读性` }],
  })
  return ...
}
```

### 7. 验证（端到端真 LLM）

`bun run demo/eval.ts` →

```
✅ T1-write    2 轮 ·   2345 tok   greet 函数 + Hello 都在
   🧑‍⚖️ LLM-judge：5/5 — 正确实现 + 类型标注 + 简洁
✅ T2-edit     3 轮 ·   3556 tok   加法→减法
✅ T3-search   3 轮 ·   3551 tok   命中 a.ts/c.ts
✅ T4-mcp      3 轮 ·   3478 tok   结果含 24
✅ T5-task     2 轮 ·   2354 tok   子 agent glob 数出 3
通过率 5/5 · 平均轮数 2.6 · 总 token 15284
```

**「判达成不判过程」的实证**:T5 子 agent 这次用 `glob "**/*.ts"` 数出 3,下次可能用 `grep -c`,过程不固定;check 只认"输出含 3"。这正是 agent 评估区别于传统单测的核心。(T1 的双重判据也成了:程序化判"有没有"、judge 判"好不好"。)

## 第三部分：cc-haha 对照（重心不一样）

### 8. 一个反直觉发现：cc-haha 的"评估"不在 agent 任务 eval

我本以为真 CC 会有一套"agent 任务通过率"的 eval。**翻下来发现没有**——它的"评估与可观测"是**两个别的维度**:① 运行时成本可观测 ② CI 质量门(传统覆盖率/冒烟)。**为什么?见 §12,这是本站最值钱的洞见。**

### 9. 运行时成本可观测：`cost-tracker.ts` + `modelCost.ts`

mini-cc 的报告只数"总 token";真 CC 是工业版——**把每次 API 调用的 usage 按模型单价换算成 USD,实时累计,还追踪时长和代码增删行**:

- `cost-tracker.ts`(381 行)一组 `getTotal*`([:11-22](openSource/cc-haha/src/cost-tracker.ts:11)):`getTotalCostUSD` / `getTotalInputTokens` / `getTotalOutputTokens` / `getTotalCacheReadInputTokens` / `getTotalCacheCreationInputTokens` / `getTotalWebSearchRequests` / `getTotalDuration` / `getTotalLinesAdded` / `getTotalLinesRemoved` / `getTotalToolDuration`——**token、钱、时间、活动量(代码增删行)全维度**(⚠️ 代码行是「活动量」不是「产出/价值」,详见 §17 的纠正)。
- 跨会话持久化:`getStoredSessionCosts`([:115](openSource/cc-haha/src/cost-tracker.ts:115))、`restoreCostStateForSession`([:158](openSource/cc-haha/src/cost-tracker.ts:158))——关掉重开,成本接着算。
- `modelCost.ts`(231 行)分档单价表:`COST_TIER_3_15`(Sonnet $3/$15,[:36](openSource/cc-haha/src/utils/modelCost.ts:36))、`COST_TIER_15_75`(Opus4 $15/$75)、`COST_TIER_5_25`(Opus4.5 $5/$25);算账走 `getModelCosts`([:144](openSource/cc-haha/src/utils/modelCost.ts:144))→`calculateUSDCost`([:177](openSource/cc-haha/src/utils/modelCost.ts:177))。**单价含 5 维**:input / output / `promptCacheWriteTokens`(3.75)/ `promptCacheReadTokens`(0.3)/ webSearch——**cache read 0.3 vs input 3 = 1/10 价,正好印证 D4 的 cache 经济学**。

### 10. CI 质量门：`scripts/quality-gate/`

这是传统软件工程的回归门禁,跑在 PR 上(`.github/workflows/pr-quality.yml`):

- **三模式**:`bun run quality:gate --mode <pr|baseline|release>`([index.ts:40](openSource/cc-haha/scripts/quality-gate/index.ts:40))→`runQualityGate`([:73](openSource/cc-haha/scripts/quality-gate/index.ts:73))。
- **覆盖率基线对比(防退化)**:`coverage.ts` 把当前覆盖率和 `coverage-baseline.json` 比,改动行覆盖率低于阈值就 fail——`"changed-lines: coverage X% is below minimum Y%"`([coverage.ts:564](openSource/cc-haha/scripts/quality-gate/coverage.ts:564))。这就是"防退化"落到代码。
- **按改动区域选检查**:`pr-quality.yml` 的 `change-policy` 按改了 desktop/server/adapter/docs 决定跑哪些 lane——不是每次全量跑。
- 多 provider 冒烟(`provider-smoke`)、desktop 冒烟(`desktop-smoke`)。

### 11. quarantine：怎么处理 flaky 测试而不让它腐烂

一个很有面试讲头的细节(`quarantine.ts`)。flaky(时好时坏)的测试,**既不能让它阻塞 CI,又不能无脑删/skip 任其腐烂**。cc-haha 的做法是登记到隔离区,每条带问责信息:

```ts
type QuarantineEntry = {
  id; path; reason;          // 哪个测试、为什么隔离
  owner;                     // 谁负责修（问责）
  reviewAfter;               // 复查日期（不能无限期赖着）
  exitCriteria;              // 满足什么才能放出来
}
```

**有 owner、有期限、有退出标准**——这是成熟工程对"技术债"的管理姿势,不是 `it.skip` 一删了之。

### 12. 为什么 cc-haha 不用"agent 任务 eval"当 CI 门？（本站最值钱）

因为 **agent 任务 eval 是非确定性的**(§3),做成 CI 硬门会**频繁 flaky**——模型今天这么走通过、明天那么走失败,门禁天天红,团队就麻木了。所以 CC 把两类质量**分开治**:

| 质量类型 | 确定性 | 怎么治 |
|---|---|---|
| 工具 / 代码 / 类型 / 覆盖率 | **确定** | 程序化 CI 硬门(quality-gate) |
| agent 的决策质量 | **非确定** | 运行时可观测(cost-tracker)+ 人工 + 离线 eval(不当硬门) |

**这正好解释了我 mini-cc 为什么把 check 写得"宽松/判达成不判过程"——就是在跟非确定性搏斗。** cc-haha 更进一步:干脆不让非确定的东西进硬门禁。

## 第四部分：评估分层框架（统一 mini-cc 与 cc-haha）

把两边收口成一张图——**"agent 评估"不是一种东西,要按「确定性」分层**:

| 层 | 对象 | 手段 | 谁做了 |
|---|---|---|---|
| 确定性门禁 | 工具/代码/覆盖率 | 程序化断言、覆盖率基线 | cc-haha quality-gate;mini-cc 程序化 check |
| 离线 agent eval | 任务通过率 | 固定任务集 + 宽松判据(容忍 flaky,**不当硬门**) | **mini-cc eval(本站)** |
| 开放质量 | 输出好不好 | LLM-judge + rubric | mini-cc judge 演示;真 CC 内部 eval |
| 运行时可观测 | 成本/性能/产出 | token→USD 累计、时长、代码行 | **cc-haha cost-tracker**;mini-cc"总 token"雏形 |
| 兜底 | 一切 | 人工 | 都靠 |

mini-cc 点亮了「离线 eval + LLM-judge + token 雏形」;cc-haha 强在「确定性 CI 门 + 运行时成本可观测」。**两者拼起来,才是完整的"评估与可观测"。**

## 第五部分 · 追问深挖沉淀（可观测全貌 + 两处自我纠正）

> 这部分是 D8 主体之外、由一连串追问挖出的干货,含**两处对前文的纠正**——把"概念上该有"误说成"cc-haha 做了"、把 activity 误说成 ROI。沉淀于此,免得带着错的说法上面试。

### 13. 可观测 ≠ 成本：三支柱 + agent 全景

cost-tracker 只是可观测的一角。**Observability 三支柱**:
- **Metrics(指标)**:数值聚合(成本/延迟/错误率/token)——cost-tracker 在这。
- **Logs(日志)**:离散事件——cc-haha 几十种 `tengu_*` 事件(安全检查 42 处最多 / API 成败 / 压缩失败 / 工具错误 / 会话恢复 / 用户反馈…)。
- **Traces(追踪)**:一次请求的完整调用链——见 §14。

agent 特有、最该观测的三块:**行为轨迹(trace)、缓存健康、循环健康**。下面逐个看 cc-haha 的实际实现。

### 14. trace 的实际实现：OTel 三层 span 树（`telemetry/sessionTracing.ts`）

cc-haha 真用 OpenTelemetry SDK(`package.json` 有 `@opentelemetry/*`)。一棵**三层 span 树**:
```
claude_code.interaction       根:一次"用户请求→响应"周期(startInteractionSpan:176)
  └─ claude_code.llm_request   子:每次 LLM 调用(:274，:316 挂到父)
       └─ claude_code.tool      孙:每个工具调用(:466)
       └─ tool_blocked_on_user  连"卡等用户审批"都有 span(:526)
```
- **粒度=结构化字段,不是堆文字**:llm_request span 记 model/speed/各类 token/success/statusCode/**attempt(第几次重试)**/modelOutput/thinkingOutput。span 像数据库行——可按 tool/model **聚合查询**(p95 耗时、重试率),不是 `print` 大段文本。
- **隐私分层(关键):默认只报元数据,全文逐项 opt-in + 截断**:
  - user prompt 默认 `<REDACTED>`,要 `OTEL_LOG_USER_PROMPTS`(:206)
  - 工具内容默认不记,要 `OTEL_LOG_TOOL_CONTENT`(:739)
  - system / 输出 / thinking 受 `isEnhancedTelemetryEnabled`(:126) 门控
  - 即使开了也 `truncateContent` 截断(:764,超长加 `_truncated`)
  - → **默认你的代码/对话一个字不上报,只报 token 数/时长/tool 名/成败**。这就是企业敢开遥测的原因。
- **回放=导出到后端看 span 树**:OTLP(`instrumentation.ts:99`)→ Jaeger/Tempo;Perfetto 本地时间线;BigQuery metrics;`BatchSpanProcessor` 批量发。**区分两种回放**:trace 回放(看调用链/哪步慢错,诊断**结构**)vs `session.jsonl` 回放(重放对话内容,诊断**决策**)。

### 15. 缓存健康的实际实现：多维 hash root-cause（`promptCacheBreakDetection.ts`）

不只"命中率掉了报警",而是**定位是什么炸了前缀**。记录上次请求的多维 hash,这次逐项 diff:`systemHash` / `toolsHash` / `cacheControlHash`(catch scope/TTL 翻转) / **`perToolHashes`(每个工具单独 hash)** / `globalCacheStrategy` / `betas` / `effort`…。最精彩的是 `perToolHashes` 的注释带线上数据:**"77% 的 tool break 是'工具没增减但某个工具描述变了'(BQ 2026-03-22),AgentTool/SkillTool 嵌动态列表所以易变"**,断了还写 `cache-break-XXXX.diff` 文件。这是 D4 break detection 的真身:**可观测到 root-cause 级**——不是"缓存掉了",是"因为 AgentTool 描述里 agent 列表变了"。

### 16. 循环健康 + 打转检测：业界两档（含自我纠正①）

**自我纠正①**:我前面说 cc-haha"监控原地打转"——**错了**。grep 下来 cc-haha **只有 `maxTurns` 硬熔断**(`QueryEngine.ts:870`,撞顶 yield 一个带成本快照的 error),**没有打转检测**。

"防打转"业界分**两档**:
- **第一档·硬上限**:步数封顶(LangGraph `recursion_limit` / AutoGen `max_consecutive_auto_reply` / cc-haha `maxTurns`)。人人做,但**不是检测打转**——数够就砍,不管你在干活还是空转。
- **第二档·真打转检测(stuck-state)**:识别"无进展"。少数做且**多在演进中**——LangChain 的 progress-aware termination(检测"重复同样 call + 同样 error")**还是 issue #36139、提议中未发布**。

检测手段(简单→高级):① `(tool,args)` 哈希+计数+工具幂等 ② no-progress(重复调用+重复错误) ③ 语义状态相似度(embedding,前沿,未见标配)。**治本**:工具描述清晰 + 返回明确成功信号(实测:含糊工具→14 次重复调用 vs 清晰→0)。**整个业界第二档都不成熟,所以大厂用硬上限兜底——cc-haha 的 maxTurns 是工程判断,不是偷懒。**

### 17. 自我纠正②：代码行是 activity,不是 ROI

前文 §9 一度把"代码增删行"当"产出/ROI"——**不严谨,纠正**。代码行数是公认的坏指标:**古德哈特**(成 KPI 就被刷量)、**删代码悖论**(删 500 行清债是好事却算"负产出")、**5 行精准 fix 常胜 500 行啰嗦码**。所以它是 **activity(活动量),不是 value(价值)**。三层要分清:

| 层 | 例子 | 性质 |
|---|---|---|
| **activity** | token / 轮数 / **代码行** / 时长 | 好量化,**不代表价值**,易 gaming |
| **output** | 生成了文件 / PR / 答案 | 有东西,不知好坏 |
| **outcome** | 测试绿 / eval 过 / PR 被合并 / 被采纳 | **才是真 ROI**,难量化 |

cost-tracker 记代码行,合理定位是**"活动规模 / 异常信号"**(突然改 5000 行=可能 agent 失控),**不是 ROI**。真 ROI 看 outcome——**D8 的 eval 通过率比代码行更接近价值**。

### 18. 三套计费,一个 token 底座

cost-tracker 算的"美元"对订阅用户是**影子价**——计费有三套,底层都是 token:
- **API 按量**:token × 单价 = 美元,无上限(cost-tracker 算这个)。
- **订阅 Pro/Max**:固定月费 + **用量限额**(不按 token 付钱)。限额 = **5 小时滚动窗口**(从首个 prompt 起算)+ **周限额**(分总额 + 高端模型专项);Max 5x/20x = 限额 ×5/×20;计量单位仍是 token。
- **mini-cc 连智谱**:智谱自己的 API 单价(又一套)。

→ **省 token 在 API 省的是钱、在订阅省的是限额额度**——同一个优化(caching/压缩/少废话)两种收益,呼应 D4/D8。(限额政策会变,以 support.claude.com 为准。)

### 19. eval 代表性的局限（诚实）

D8 的 5 个 case 是**工具覆盖的冒烟集(smoke test),代表性弱**,**不是**能力基准(benchmark)——"覆盖每个工具" ≠ "代表真实任务分布"。它代表不了:难度分层(全单步)、负样本(无该拒/无解/需澄清)、n>1(每能力 1 个)、pass@k(只跑一次)、判据偏松(可假阳性)。**代表性怎么来**:从**真实使用日志采样** / 用**公开 benchmark**(SWE-bench)+ 难度分层 + 负样本 + pass@k。本版目标是打通 eval **机制**,代表性是下一步用真实分布填的。面试被问"怎么证明代表性"——**承认是冒烟集 + 知道怎么补,远比硬吹强**。

## 第六部分：面试金句

1. **"eval 是什么 / 为什么"** —— agent 版单测+性能基准:固定任务+成功判据,统计通过率/轮数/token。它把"改 prompt/换模型"从凭感觉变成有数字,是防退化的地基。没 eval = 没做过生产。

2. **"agent 评估难在哪"** —— 非确定性 / 多种正确答案 / 多轮副作用。所以**判达成不判过程**(查最终文件状态/输出,宽松匹配),不像传统单测判确定输出。我 mini-cc 的 check 就是这么写的。

3. **"程序化 vs LLM-judge"** —— 程序化断言确定/免费/可 CI,但只判"有没有";LLM-judge 能判"好不好"(开放质量),代价慢/贵/judge 也会错。我 T1 两种都跑了做对照。

4. **"评估要分层"(最高级)** —— 确定的东西(代码/覆盖率)用程序化硬门;**非确定的 agent 决策不该进 CI 硬门**(会 flaky 到团队麻木),改用运行时可观测+离线eval(容忍flaky)+人工。真 CC 就是这么分的:quality-gate 管确定性、cost-tracker 管可观测,**故意不拿 agent 任务通过率当 CI 门**。

5. **"可观测的工业版长什么样"** —— 不止数 token:按模型单价把 usage 换算成 USD 实时累计、跨会话持久化、连代码增删行和时长都追踪(cc-haha cost-tracker + modelCost 分档单价表)。而且 cache read 是 input 的 1/10 价——可观测数据正好喂回 D4 的 cache 优化决策。

6. **"flaky 测试怎么治"** —— 不删不无脑 skip,登记隔离区,每条带 owner/复查期/退出标准(cc-haha quarantine)。有问责、有期限,技术债不腐烂。

7. **(主动暴露边界)** —— mini-cc 的 eval 是离线手动跑、判据宽松、judge 只演示一个;没做 cc-haha 那种成本→USD 换算、跨会话持久化、CI 集成、覆盖率基线。这些是规模化/生产化特性,不是 eval 的内核。

8. **"可观测三支柱"** —— Metrics(成本/延迟/token)/ Logs(事件)/ Traces(调用链),成本只是 Metrics 一项。agent 特有三块:trace(行为轨迹)、缓存健康、循环健康。真 CC 用 OTel 搭了 interaction→llm_request→tool 三层 span 树,连思考和重试都记。

9. **"遥测的隐私分层"** —— 默认只报元数据(token/时长/tool 名/成败),prompt/工具内容/输出要 `OTEL_LOG_*` 逐项 opt-in 且截断。"默认你的代码一个字不上报"是企业敢开遥测的前提。回放走标准 OTLP→Jaeger/Tempo,不锁厂商。

10. **"防 agent 打转分两档"** —— 硬上限(步数封顶,人人做,LangGraph recursion_limit / cc-haha maxTurns)vs 真 stuck-state 检测(识别无进展,少数做、LangChain 还在 issue 阶段)。手段:(tool,args) 哈希 / no-progress / 语义相似度;治本是工具清晰+明确成功信号。**业界第二档还不成熟,大厂用硬上限兜底——这是成熟度判断,不是偷懒。**

11. **"activity ≠ output ≠ outcome"** —— 代码行/token/轮数是 activity(可观测、防失控,但易 gaming,**别当生产力**);真 ROI 看 outcome(任务达成/被采纳)。说"按代码行衡量产出"是新手坑(古德哈特+删代码悖论)。

12. **"三套计费,一个 token 底座"** —— API 按 token 付美元(无上限)/ 订阅固定费+用量限额(5h 窗口+周限额)/ 第三方端点各自单价。cost-tracker 的美元对订阅用户是影子价。**省 token 在 API 省钱、在订阅省限额**——同一优化两种收益。
