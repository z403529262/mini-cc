# D11.5-① · LLM 通识补强卡(口头准备,五张卡)

> 定位:这五个是"第一轮筛人题"——便宜、高频、答不上很难看。每张卡 = 30 秒版 + 要点 + 追问防守。
> 与 D 系列的关系:D4/D8/D9 已经把"应用层"打穿了,这里补的是往下一层的"机制层"——让已有的应用层答案多一句"底层原因是…"。

---

## 卡 1 · 采样参数(temperature / top_p / top_k)

**30 秒版**:解码时下一个 token 是从概率分布里**采样**出来的。temperature 缩放分布(→0 趋近贪心取最大,>1 更随机);top_p 核采样按概率降序累加到 p 截断;top_k 固定取前 k 个。agent 场景经验值:工具调用、代码、结构化输出用低温(0~0.3)求确定;创意发散才升温。一般只动 temperature 和 top_p 之一。

**要点**:
- 机制:logits 除以 T 再 softmax。T 压低时高概率 token 更占优,T 升高时分布拉平。
- 诚实自省(主动说):mini-cc 的 demo 没显式设采样参数 = 用 provider 默认。反思是 **eval 报告要可比,采样参数就该固定并写进报告**——这是 D8 的一个待补项,我知道它在哪。
- 工具调用的参数值也是 token 生成出来的,同样受温度影响——低温对 tool 参数的稳定性有实际意义。

**追问防守**:
- *"temperature=0 就完全确定了吗?"* —— 不保证:GPU 并行浮点加法非结合性、MoE 的专家路由受 batch 影响、投机解码,都会引入非确定。所以 D8 说 agent eval 要"判达成不判过程",温度调零也救不了逐字复现。
- *"top_p 和 top_k 有什么差别?"* —— top_p 自适应(分布尖时候选少、平时候选多),top_k 固定截断;top_p 更常用。

## 卡 2 · 训练管线(pretrain → SFT → RLHF/DPO → RLVR)

**30 秒版**:pretrain 在海量语料上做 next-token 预测,得到"会接龙但不会好好聊天"的 base model;SFT 用少量高质量指令-回答对教会"格式与服从";偏好对齐让输出更符合人类偏好——RLHF 是先训 reward model 再 PPO,DPO 跳过 RM 直接在偏好对上优化(更稳更便宜);RLVR(可验证奖励,数学/代码可自动判对错)是推理模型的核心训法。

**要点(和 agent 工程的三条连接,这是差异化)**:
1. **工具调用能力是训出来的**(SFT/RL 阶段专门训过工具调用格式)——所以不同模型的 function calling 可靠性差异巨大,**换模型必须重跑 eval**(D8 的机制层理由)。
2. **幻觉的机制根源**:next-token 预测的目标是"像",没有真值锚——所以 grounding 手段(RAG 限定资料、工具结果当 ground truth)是在补目标函数天生的缺口(T2 的机制层理由)。
3. system prompt 之所以被服从,是 SFT/RLHF 教出来的指令层级,不是天然的——这也是 prompt injection 能得手的原因之一:层级是学出来的软约束。

**追问防守**:
- *"DPO 和 PPO 一句话差别?"* —— PPO 要单独训 reward model 再做 RL,链路长且不稳;DPO 把偏好直接写进损失函数,一步监督式训练。
- *"为什么 RLVR 造出了推理模型?"* —— 奖励可自动验证(代码跑测试/数学对答案)→ 不需要人标就能大规模 RL → 模型学会"多想几步换正确率"。

## 卡 3 · 推理模型 / test-time compute

**30 秒版**:推理模型在输出前生成思考 token,用**推理时算力换答案质量**,由 RLVR 训出来。agent 选型:规划、复杂 debug、长程任务开思考;流水线执行、分类、格式转换不开——思考预算是继"模型档位"之后的**第二条成本轴**。

**要点**:
- API 形态:Claude 是 thinking block(可 interleaved——工具调用之间也思考);OpenAI 的 Responses API 下 reasoning token 可跨轮保留(GPT-5 起,核验于 2026-07)。
- 实证锚点:OSWorld 2.0 长程任务榜首(Opus 4.8)的配置就是 **max thinking + 批量工具调用**——思考深度直接影响长程任务成绩。
- 和 D8 的连接:该不该开思考 = eval 说了算——对比"通过率提升 vs token 成本增幅",不是拍脑袋。

**追问防守**:
- *"thinking token 计费吗?"* —— 计,按 output 价——所以"要不要思考、思考多深"本质是成本问题;常见做法是主 agent 规划开思考、子 agent 执行不开(接 D7 分工)。
- *"思考内容能看到吗?"* —— 各家策略不同:有的给摘要、有的隐藏;工程上**别把业务逻辑依赖在思考内容上**,它不是稳定接口。

## 卡 4 · KV cache(把 prompt caching 捅到推理层)

**30 秒版**:推理分两阶段——prefill(并行算完整个 prompt 的 attention,得到每个位置的 K/V)和 decode(逐 token 生成,每步要 attend 全部前文)。KV cache 就是把算过的 K/V 存显存,decode 不重算。**prompt caching 的真相 = 把前缀的 KV 跨请求持久化,同前缀请求直接跳过 prefill**——0.1× 的价格是"省掉了 prefill 算力"的让利。

**要点(这张卡把 D4 的三个应用层规则一次捅穿)**:
- 为什么必须**前缀逐字节一致**才命中:causal attention 下,位置 i 的 K/V 依赖 0..i 的全部内容——中间改一个字,后面所有 KV 全变。D4 说"改一处炸它及之后",物理层就是这个。
- 为什么长上下文贵:prefill 是 O(N²) 的 attention 计算,decode 阶段 KV 占显存 O(N) 还一直占着。
- 为什么 output 比 input 贵:decode 逐 token 串行,每个 token 都要过全部权重、占着 KV 显存和算力时间;prefill 可以并行吞。
- 为什么 cache write 1.25×:持久化 KV 有存储成本;为什么 read 0.1× 还赚:换回来的推理机时远值这个价。

**追问防守**:
- *"lost in the middle 为什么发生?"* —— 训练数据里关键信息多在首尾 + attention 分布特性,长上下文中段利用率天然低——这是"长上下文 ≠ 全塞"的机制层理由(接 D9 三路对照)。
- *"KV cache 和 prompt caching 是一回事吗?"* —— KV cache 是单次请求内的推理优化(必有);prompt caching 是把它跨请求持久化的产品化(可选、计价)。

## 卡 5 · LoRA / 微调方法一句话

**30 秒版**:LoRA 冻结原权重,在旁路加低秩分解矩阵 ΔW=BA,只训这不到 1% 的参数;QLoRA 再把底模量化到 4bit 训练,单卡就能微调。部署时可合并回权重,也可以多个 adapter 热切换(一底模服务多租户)。

**要点(应用工程师的立场)**:
- LoRA 把**训练成本**打下来了,但微调的真成本在**数据管线 + eval + 回归防退化**——这正是 T3 说"90% 的想微调其实是 prompt 或检索没做好"的原因:贵的不是 GPU,是数据和验证闭环。
- 判断顺序(接 T3):缺知识 → RAG;缺风格/格式/领域语言 → 微调;缺行为能力 → prompt + 工具编排先行。

**追问防守**:
- *"微调会伤通用能力吗?"* —— 会(灾难性遗忘),所以要混通用数据回放 + 微调后跑通用 eval;又一个"eval 是安全网"的实例。
- *"你微调过吗?"* —— 诚实:没有生产级微调经验;但我知道决策框架和成本结构,并且我的判断是当前业务场景先把 prompt/RAG/工具的杠杆拉满(能讲 T3 全套理由)。
