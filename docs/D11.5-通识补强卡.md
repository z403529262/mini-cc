## 卡 1 · 采样参数(temperature / top_p / top_k)

**30 秒版**:解码时下一个 token 是从概率分布里**采样**出来的。temperature 缩放分布(→0 趋近贪心取最大,>1 更随机);top_p 核采样按概率降序累加到 p 截断;top_k 固定取前 k 个。机制:logits 除以 T 再 softmax。业界主流不再拧这些旋钮——**按调用角色分温**,推理模型直接锁死。

**机制要点**:
- logits/T → softmax:T 低时高概率 token 更占优,T 高时分布拉平。
- **T=0 也不确定**——GPU 浮点非结合、MoE 专家路由受 batch 影响、投机解码,都会引入非确定性。D8 "判达成不判过程"的机制层理由。

**业界主流:按调用角色分温(一手证据齐)**

| 调用角色 | 温度 | 依据 |
|---|---|---|
| 主 agent loop(生成/探索/自纠错) | 默认或 1 | cc-haha `claude.ts:1755-1759` 原话:"Only send temperature when thinking is disabled — the API requires temperature: 1 when thinking is enabled" |
| 判定类辅助(分类/评分/路由) | 0 | cc-haha `yoloClassifier.ts` 三处 `temperature: 0`;判定摇摆本身就是 bug |
| 批量确定性加工(摘要/上下文增强) | 0 | Anthropic cookbook contextual retrieval `temperature: 0.0` |
| 推理模型(思考模式) | 无旋钮 | GPT-5 系 `reasoning_effort != none` 时**直接拒收 temperature**;Anthropic 思考模式**强制 1**(cc-haha 干脆不发这个字段) |

**三阵营(其他家)**:
- **低温派**:Aider `send_with_retries()` 默认 `temperature=0`——**单次生成求确定,拿采样保稳**。
- **默认温度派**:Claude Code 主 loop 敢用 t=1——**有自我纠错回路**(工具报错回填 → 换方案),多样性帮它跳出重复失败;确定性由别处提供(tool schema 校验、edit 唯一性契约、测试)。**"单次生成"vs"带反馈的循环"的本质差异**。
- **无旋钮派(趋势)**:推理模型;下一节讲为什么锁。

**mini-cc 自省**:全程没设温度 = 智谱默认(≈1),歪打正着和真 CC 主 loop 一致。但 **D8 的 LLM-judge 是评分调用,按规律应显式 t=0**——评分摇摆让 eval 结果不可比,这是待改进项。

**top_p / top_k 现状**:cc-haha 全库 top_k 零处;top_p 只出现在 OpenAI 兼容代理的透传层——生产 agent 根本不碰这两个。top_p 自适应(分布尖时候选少、平时候选多),top_k 固定截断;若真要调**只动其一**。

---

**采样参数 vs 推理预算(为什么现在都不改参数了)**

它们是**两代"用推理时算力换答案质量"的接口**:

- **上一代:横向多样性**。self-consistency / best-of-N:t=0.7~1 采 N 条不同 CoT,投票选优。温度不是噪声,是**多样性引擎**——t=0 时 best-of-N 直接失效(N 条一模一样)。
- **这一代:纵向深度**。RLVR 把探索**内化**进单链——学会"多想几步"、自我检查、回溯("wait 我再想想")。旋钮从"N × 温度"变成"想多少 token"(thinking budget / reasoning effort)。
- 类比:旧世界外挂 N 个线程投票;新世界把并行搜索+选优**内联进函数体**,对外只暴露迭代预算。

**为什么思考模式锁死采样(重到轻)**:
1. **训练分布匹配**——思考行为是 RL 在**某个固定采样配置**(通常 t≈1)下训出来的。改温度=policy 扔进没被训过的分布:低温掐死探索(长贪心易复读),高温摧毁长链连贯性。**思考需要随机性,但只在被训练的配置下工作**。
2. **答案只是"读结论"**——质量几乎全部由思考决定,后面正文是誊写。温度对最终质量的影响被边缘化。
3. **温度是坏旋钮,预算是好旋钮**——温度-质量非单调(事实题基本平/负相关,创意题倒 U 型),用户长期误用;预算-质量单调递增边际递减,还能**直接映射账单**(思考 token 按 output 计费)。
4. **砍支持矩阵**——每个旋钮×每次发布=组合爆炸;锁死 = 行为可预期。

**两个防翻车 nuance**:
- **采样没消失,只是不归你管了**。t 钉在 1,模型**内部照样随机采样**——这也是推理模型仍非确定的原因之一。被拿走的是"用户控制权",不是"随机性"。
- **横向并行也没死,只是搬进了厂商黑盒**。各家 pro/heavy 档本质仍是"多链 + 聚合",只是编排从开发者移到 provider 侧。**test-time compute 两个维度(横 N × 纵深度)都在,接口收敛成"预算/档位"**。

**追问防守**:
- *"那 temperature 会彻底消失吗?"* —— 不会。非推理调用(分类/抽取/judge)大量存在,那里 t=0 仍是正解;消失的是"用温度调质量"这个用法。
- *"推理模型怎么做确定性?"* —— 做不到逐字确定(内部采样+浮点非结合)。工程上还是 D8 那句:**判达成不判过程,确定性去 verifier 层找,别去采样层找**。
- *"top_p 和 top_k 差别?"* —— top_p 自适应,top_k 固定截断;top_p 更常用。生产 agent 都不碰。

## 卡 2 · 训练管线(pretrain → SFT → RLHF/DPO → RLVR)

**30 秒版**:一条"每一阶段都在补上一阶段缺口"的链——能力来自 pretrain,行为来自后面所有阶段。SFT 教格式,模仿学不会分寸 → RLHF/DPO 从偏好比较里学分寸,但会被 reward hacking(谄媚)→ RLVR 用可自动验证的对错,免标注、防刷分,推理模型和现代 agent 能力都是它训出来的。

**英文全称表(面试能随口展开是加分项)**:

| 缩写 | 英文 | 直译 | 出处 |
|---|---|---|---|
| Pretrain | **Pre-training**(self-supervised) | 预训练 | — |
| SFT | **Supervised Fine-Tuning** | 监督微调 | InstructGPT(2022)定型 |
| RLHF | **Reinforcement Learning from Human Feedback** | 人类反馈强化学习 | Christiano et al. 2017 → InstructGPT 带火 |
| DPO | **Direct Preference Optimization** | 直接偏好优化 | Rafailov et al. 2023;副标题"Your LM is Secretly a Reward Model" |
| RLVR | **Reinforcement Learning with Verifiable Rewards** | 可验证奖励强化学习 | AI2 Tülu 3(2024)定名;DeepSeek-R1 带爆 |

周边缩写:**PPO** Proximal Policy Optimization(Schulman et al. 2017,OpenAI)|**GRPO** Group Relative Policy Optimization(DeepSeekMath 2024,组内相对得分当 baseline 省 value model)|**RM** Reward Model|**KL 惩罚** Kullback-Leibler divergence|**RLAIF** RL from AI Feedback(Anthropic Constitutional AI 代表作)。

**五阶段深挖**:

1. **Pretrain**:万亿 token 上 next-token 预测,自监督(数据即标签)。反例秒懂 base model 为什么不能直接用——问它"中国的首都是哪?",它接的很可能是"这是小学地理常见考题。类似的还有:美国的首都是哪?…"——**它在做"最像语料的延续",不是回答你**。**99%+ 算力和钱在这一步,知识和智力天花板在这里定死,后面全是"怎么引出来"**。

2. **SFT**:几万到几百万条指令-回答对,继续 next-token(通常只对回答部分算 loss)= 模仿学习。教会得比想象具体:**对话格式(chat template/role)、工具调用格式(tool_use JSON)、拒答的样子**。→ 这就是**换模型必须重跑 eval** 的根源:function calling 可靠性是训出来的。缺口:模仿教得会格式,教不会分寸;示范里标注员的自信腔调 → 学出"自信的样子"而非"知道才自信"(SFT 后一本正经胡说的部分根源)。**关键转折句:好回答写不出来,但认得出来** → 引出偏好学习。

3. **RLHF**:两步。**① 训 RM**:同 prompt 采 N 答,人做比较(A 比 B 好)训"给任意回答打分"的模型——用比较不用打分,因为人打绝对分极不一致。**② PPO 优化**:RM 当奖励,同时加 **KL 惩罚**拴住 policy 别离 SFT 太远。**反例(不拴的后果)**:reward hacking——回答越写越长、疯狂 bullet、无脑赞同用户。**谄媚(sycophancy)是 RLHF 的职业病**,因为标注员真的更喜欢被认同的感觉。**代价**:四个模型副本(policy/ref/RM/value),链路长、超参敏感、贵。

4. **DPO**:2023 数学结果——RLHF 的优化目标**推导等价**于一个直接在偏好对上的监督式损失,把 policy 自己当隐式 RM,一步到位。loss 直觉:相对参考模型抬 chosen、压 rejected。**得**:不训 RM、不跑 RL,像普通微调一样稳定便宜,**开源社区绝对主力**。**失**:离线(off-policy),只学静态偏好对,不像 PPO 在线采样"当前输出"持续纠偏,一般认为上限略低,前沿大厂主力仍是 online RL。**定位**:RLHF 是目标,DPO 是捷径。

5. **RLVR**:奖励换成**自动可验证的对错**(数学对答案/代码跑测试/格式过校验)。**为什么引爆推理模型**:① 免人工 → rollout 规模不受标注瓶颈;② 客观 → **骗不了单元测试**,hacking 空间小;③ 最神奇的——模型 RL 中**自发涌现**"多想几步换正确率"、自我检查、发现错了回溯(DeepSeek-R1 论文的 **aha moment**)。**GRPO** = PPO 变体,用组内 rollout 相对得分当 baseline 省掉 value model。**边界**:只有能自动判分的域(数学/代码/逻辑/agent 任务)适用,写作/聊天还得靠 RLHF。**现代管线是混合**:SFT 之后,开放域走偏好对齐、推理域走 RLVR。

**和 agent 工程的四条连接(差异化)**:
1. **工具调用能力是训出来的**——不同模型 function calling 可靠性差异巨大,**换模型必重跑 eval**(D8 机制层理由)。
2. **幻觉的机制根源**:next-token 预测目标是"像"没有真值锚 → grounding 手段是补目标函数天生的缺口(T2 机制层理由)。
3. **system prompt 服从是学出来的软约束**(SFT/RLHF 教出的指令层级,不是天然的) → prompt injection 能得手的部分原因。
4. **agentic RL 是 RLVR 的直接延伸**——奖励=任务完成(patch 过测试/GUI 任务达成)。2025-26 agent 能力暴涨不是 prompt 写得更好,是**模型被 RL 直接对着"任务完成"优化过了**,SWE-bench 既是考卷也是训练场。

**verifier 质量 = 能力上限定律**:RLVR 会不会被 hack?会——测试写得弱,模型学会硬编码特判过测试、甚至改测试。所以 verifier 的质量就是能力上限。**这和 D9"召回质量是答案上限"、D8"判据设计决定 eval 可信度"是同一条定律的三个化身:你能验证什么,决定你能得到什么**。

**追问防守**:
- *"为什么说后训练不加知识?"* —— 数据量差 4~6 个数量级(万亿 vs 百万级),知识容量由 pretrain 定;后训练改的是"取用方式",硬灌新知识反而灾难性遗忘。
- *"DPO 和 PPO 怎么选?"* —— 资源有限、快速迭代 → DPO;追上限、有基建 → online RL。
- *"RLVR 就不会被 hack 了吗?"* —— 会,但空间比 RLHF 小;上限由 verifier 决定,同"你能验证什么就得到什么"定律。

## 卡 3 · 推理模型 / test-time compute

**30 秒版**:推理模型在输出前生成思考 token,用**推理时算力换答案质量**,由 RLVR 训出来。agent 选型:规划、复杂 debug、长程任务开思考;流水线执行、分类、格式转换不开——思考预算是继"模型档位"之后的**第二条成本轴**。

**要点**:
- API 形态:Claude 是 thinking block(可 interleaved——工具调用之间也思考);OpenAI 的 Responses API 下 reasoning token 可跨轮保留(GPT-5 起,核验于 2026-07)。
- 实证锚点:OSWorld 2.0 长程任务榜首(Opus 4.8)的配置就是 **max thinking + 批量工具调用**——思考深度直接影响长程任务成绩。
- 和 D8 的连接:该不该开思考 = eval 说了算——对比"通过率提升 vs token 成本增幅",不是拍脑袋。

**追问防守**:
- *"thinking token 计费吗?"* —— 计,按 output 价——所以"要不要思考、思考多深"本质是成本问题;常见做法是主 agent 规划开思考、子 agent 执行不开(接 D7 分工)。
- *"思考内容能看到吗?"* —— 各家策略不同:有的给摘要、有的隐藏;工程上**别把业务逻辑依赖在思考内容上**,它不是稳定接口。

## 卡 4 · KV cache(把 prompt caching 捅到推理层)

**30 秒版**:推理分两阶段——prefill(并行算完整个 prompt 的 attention,得到每个位置的 K/V)和 decode(逐 token 生成,每步要 attend 全部前文)。KV cache 就是把算过的 K/V 存显存,decode 不重算。**prompt caching 的真相 = 把前缀的 KV 跨请求持久化,同前缀请求直接跳过 prefill**——0.1× 的价格是"省掉了 prefill 算力"的让利。

**要点(这张卡把 D4 的三个应用层规则一次捅穿)**:
- 为什么必须**前缀逐字节一致**才命中:causal attention 下,位置 i 的 K/V 依赖 0..i 的全部内容——中间改一个字,后面所有 KV 全变。D4 说"改一处炸它及之后",物理层就是这个。
- 为什么长上下文贵:prefill 是 O(N²) 的 attention 计算,decode 阶段 KV 占显存 O(N) 还一直占着。
- 为什么 output 比 input 贵:decode 逐 token 串行,每个 token 都要过全部权重、占着 KV 显存和算力时间;prefill 可以并行吞。
- 为什么 cache write 1.25×:持久化 KV 有存储成本;为什么 read 0.1× 还赚:换回来的推理机时远值这个价。

**追问防守**:
- *"lost in the middle 为什么发生?"* —— 训练数据里关键信息多在首尾 + attention 分布特性,长上下文中段利用率天然低——这是"长上下文 ≠ 全塞"的机制层理由(接 D9 三路对照)。
- *"KV cache 和 prompt caching 是一回事吗?"* —— KV cache 是单次请求内的推理优化(必有);prompt caching 是把它跨请求持久化的产品化(可选、计价)。

## 卡 5 · LoRA / 微调方法(业界主流详解)

**30 秒版**:应用层 95% 场景就是 **LoRA/QLoRA**——连托管微调 API 底下也是 LoRA + SFT。真正工程量不在训练在**数据**(主流是"用强模型蒸馏出小模型能顶的数据")。工具链国内 LLaMA-Factory、单卡 Unsloth、底座 TRL;部署侧 vLLM **multi-LoRA** 一底模服务 N 租户。2026 增量是 RFT/RLVR 下放成产品——自定义 grader 调推理模型。**微调是省钱手段,不是救命稻草**——顺序永远是 prompt/RAG/工具先拉满,规模上来再微调小模型做成本优化。

**先分层:业界微调是三个不同世界**

1. **模型厂商 post-training**(OpenAI/Anthropic/DeepSeek):SFT+RLHF+RLVR 全家桶、千卡级——这叫**造模型**不叫微调,别混进答案。
2. **应用企业的微调**(面试问的"业界主流"就是这层):拿开源底模做参数高效微调。**2025-26 默认底模基本是 Qwen 系**,其次 Llama/DeepSeek 蒸馏版。
3. **微调即服务**:不碰 GPU 调 API——OpenAI fine-tuning API、国内百炼/方舟/千帆;2025 底新物种 **Tinker**(Thinking Machines)"你写训练循环、它管 GPU 集群",LoRA 权重可下载——**介于全托管黑盒和自攒卡之间**(实践者对全托管的真实抱怨:太黑盒、太受限)。

**技术主流:LoRA/QLoRA 是事实标准**

- LoRA 原理:冻结原权重,旁路加低秩分解 **ΔW = BA**,只训 <1% 参数。
- **一个硬证据**:托管微调 API(OpenAI 这类)**底下也是跑 LoRA + SFT**——不是全参。
- 惯例超参:**rank 8~64、挂在所有 linear 层**(早年只挂 attention q/v,现在共识 all-linear 更稳)、学习率比全参大约一个数量级。
- **"LoRA 是妥协"已过时**:Thinking Machines《LoRA Without Regret》(2025)结论——中小数据量后训练场景,**配置得当的 LoRA 能追平全参**。
- **QLoRA** = 底模量化到 4bit(NF4)+ LoRA,单张 24G 消费卡调 30B 级——把门槛打到个人开发者水平。
- **全参微调只剩三个场景**:厂商 post-training / 小模型(<10B)+大数据量 / 追极致不差钱。

**真正的工程量在数据(教科书 vs 实践差别最大的地方)**:

- **最普遍的真实用法 = teacher-student 蒸馏**:强模型(GPT-5/Claude)批量生成或标注领域数据,微调 7B~30B 小模型顶班。动机不是"能力不够"是**成本**——窄任务(分类/抽取/客服/审核)推理成本降 **10~100 倍**。
- **质量 >> 数量**:LIMA 老教训至今成立,**千条精数据打赢五万条杂数据**;数据管线(收集→清洗→去重→难度配比)吃掉整个项目 **70% 的时间**。
- **防灾难性遗忘**:领域数据里混 5~10 倍通用数据回放,微调完必须跑通用能力回归——又是"eval 是安全网"。

| 工具 | 定位 |

| **Unsloth** | 单卡效率之王(2-5× 提速省显存),个人/小团队首选 |

**2026 增量:RFT/RLVR 下放到应用层**

上次(卡 2)讲 RLVR 是厂商造推理模型的训法——**现在作为产品下放了**:OpenAI Reinforcement Fine-Tuning 让你"用自定义 grader 调他家推理模型"。

**一手样本(2026-03)**:Tinker 上对 Qwen3-4B 做 RLVR——40 步 SFT 热身教输出格式 → GRPO 式 RL 10 轮(每题采 8 答/精确匹配给分/组内标准化优势)、**LoRA rank 32**;reward 设计:对 +1.0 / 格式干净 +0.1 / 缺格式 -0.2。**结果:GSM8K 89.2% → 90.0%,+0.8%**。

**+0.8% 比配方本身更值钱的读数**:好底模上、常规任务的 RFT 收益是**边际的**。RFT 真正值钱的位置是:任务有**可验证判据**、但底模确实做不好的**窄域**(独特工具调用格式、专业领域判定)。**这个"诚实读数字"和 D9 批判 benchmark 是同一块肌肉**。

**给 agent 工程的钩子**:**RFT 的 grader 设计 = D8 的 eval 判据设计**,字面上是同一个技能——"你能验证什么,决定你能训出什么"(接卡 2 verifier 上限定律)。

**部署:multi-LoRA serving 是多租户成本利器**

LoRA adapter 两条路:
- **合并**回底模权重(单一场景,零额外延迟);
- **运行时挂载**——vLLM / SGLang 支持一个底模热插拔 N 个 adapter,**一份底模显存服务 N 个客户的定制模型**。

**SaaS 多租户定制的标准答案就是它**,面试聊"多租户怎么做定制"直接甩这个。

**端到端流程(业界正经团队的顺序)**

**先立 eval**(没有 eval 的微调是盲飞)→ 底模选型(尺寸按延迟/成本倒推)→ **数据管线(蒸馏+清洗+配比,主工程量)**→ SFT 起步(LoRA)→ 需要偏好/判定质量再加 DPO 或 RFT → 领域 eval + 通用回归**双跑** → 部署(merge 或 multi-LoRA)→ **线上 bad case 回流数据,飞轮迭代**。

**反面清单(什么不该用微调)**:
- 注入**时效性知识** → 用 RAG(微调改的是**行为**不是知识库);
- 提升**通用能力** → 你没那个数据量;
- 需求还在**天天变**的阶段 → prompt 当天改,微调一轮一周。

**立场话(接 T3)**:判断顺序=缺知识 → RAG / 缺风格/格式/领域语言 → 微调 / 缺行为能力 → prompt + 工具编排先行。真成本在数据管线 + eval + 回归防退化——**贵的不是 GPU 是数据和验证闭环**。

**追问防守**:
- *"LoRA 的 rank 怎么选?"* —— 从 16/32 起步,数据越多任务越偏离底模分布 rank 越大;**超过 64 通常先怀疑数据而不是加 rank**。
- *"怎么知道该 SFT 还是 RFT?"* —— 有标准答案示范 → SFT;没有示范但有可验证判据 → RFT;只有偏好比较 → DPO。**按"你手里有什么监督信号"选,不按流行度选**。
- *"微调会伤通用能力吗?"* —— 会(灾难性遗忘),混通用数据回放 + 微调后跑通用 eval 兜底。
- *"你微调过吗?"* —— 诚实:没有生产级微调经验;但我知道决策框架、成本结构、工具链版图、数据管线才是主战场;并且我的判断是当前业务先把 prompt/RAG/工具的杠杆拉满。
