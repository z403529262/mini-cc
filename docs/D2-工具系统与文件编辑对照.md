## 0. D2 一句话

给 agent 装上**能改代码的手**：多工具（read/write/edit/glob/grep/bash）+ 统一抽象 +
注册表分发；并实测**并行工具调用**（一轮多个 tool_use 一起回填）。

---

## 1. 工具系统抽象

### 我的实现（`src/tools.ts`）
```ts
interface Tool { name; description; input_schema; execute(input): string }
const tools: Tool[] = [read_file, write_file, edit_file, glob, grep, bash]
const toolMap = new Map(tools.map(t => [t.name, t]))
```
loop 按 name 查表调 `execute`，**加工具不改 loop**（开闭原则）：
```ts
const tool = toolMap.get(b.name)
const out = tool ? tool.execute(b.input) : `[未知工具] ${b.name}`
```

### cc-haha 对照（`buildTool({...}) satisfies ToolDef`）
它的 `ToolDef` 不是两件套，是**十几个钩子**：
`name` / `prompt()`(描述) / `inputSchema`(zod) / `validateInput()` / `call()`(执行) /
`checkPermissions()` / `mapToolResultToToolResultBlockParam()`(结果转 tool_result) /
`renderToolUseMessage()`(TUI 渲染) / `inputsEquivalent()` / 一堆埋点。

**但骨架同构**：name + schema（给模型）+ execute/call（给代码）。我多出来没有的，是
权限、输入校验、结果渲染、等价判断、埋点——都是"外壳"，内核一致。

### 面试点
工具 = **声明（schema 给模型看）/ 实现（execute 给代码跑）分离**（像 interface/SPI vs impl）；
注册表按 name 分发 = 开闭原则，加工具零改动主循环。

## 2. 并行工具调用（实测 ✅）

任务："同时读取 src/m0.ts、src/m1.ts、src/tools.ts，各用一句话说明。"

实测结果：
- **第 1 轮**：模型一次返回 **3 个 `read_file` 的 tool_use 块**
- loop `for (const b of res.content)` 遍历**全部执行**，收进 `results` 数组
- **3 个 tool_result 放进同一条 user 消息**一起回填
- **第 2 轮**：模型基于 3 份结果一次性总结 → end_turn

要点：
- "并行"指**一轮内有多个 tool_use**，不是多线程；执行可串行可并发，
  但**回填必须配齐、且在同一条 user 消息里**（D1 三铁律之一）。
- 这就是 Claude Code 同时读多个文件、跑多条命令时一屏刷出多个调用的底层形态。

## 3. edit_file 深度对照（D2 重点）

体量：我的 `edit_file` **23 行**；cc-haha 文件编辑工具
`FileEditTool.ts(625) + utils.ts(775) + prompt.ts + types.ts ≈ 1500+ 行`。

### 3.1 核心算法：我和官方一模一样
```ts
// 我：tools.ts
const hits = content.split(old_string).length - 1
if (hits === 0) return "没找到"
if (hits > 1)  return "不唯一，请加上下文"
writeFileSync(path, content.replace(old_string, new_string))

// 官方：FileEditTool.ts validateInput
const matches = file.split(actualOldString).length - 1
if (matches > 1 && !replace_all) { /* 报错：加上下文 或 设 replace_all */ }
```
**同一个 `split(x).length - 1` 计数法。** 差别仅：官方有 `replace_all` 逃生口
（不唯一时可全替换），报错给两条路；我只能拒绝逼模型加上下文。

### 3.2 官方在内核外包的「六层洋葱」
最内核（我有）：**定位唯一 → replace → 写盘**。往外，cc-haha 一层层包：

1. **schema 层**（`types.ts`）：zod `strictObject`（多余字段报错）+ `semanticBoolean`
   （容忍模型把 bool 传成字符串 `"true"`）+ 多一个 `replace_all` 参数。我是手写 JSON Schema。
2. **description 层**（`prompt.ts`）：动态生成，含——先读才能改 / **缩进要保留**（警告：
   Read 输出有"行号+tab"前缀，别把前缀写进 old_string）/ 优先改勿新建 / 唯一失败用
   replace_all / 内部用户额外提示"最小唯一 2-4 行"。我只有一句"必须唯一"。
3. **校验层**（`validateInput`，~13 道）：old==new 拒绝、权限 deny、UNC 路径（防
   Windows NTLM 凭据泄露）、>1GiB 防 OOM、编码检测（BOM→utf16le）+ CRLF 归一、
   文件不存在给"**你是不是想改 XXX**"建议（findSimilarFile）、.ipynb 引导去 NotebookEdit、
   **先读才能改**、**读后被改检测**、settings 文件模拟校验。我：try/catch 兜底。
4. **容错层**（`utils.ts`）：弯↔直引号归一化匹配 + 替换后**保留文件原引号风格**
   （连缩写 don't 的撇号都判）、行尾空白 strip（`.md/.mdx` 跳过，因两尾空格=硬换行）、
   **desanitize**、删除时去尾换行（避免留空行）。我：完全没有。
5. **原子写层**（`call`）：mkdir 父目录（✅ 我也有 mkdirSync recursive）、fileHistory
   **改前备份**（可撤销）、临界区**不 await 保原子**（防并发编辑交错）、写后更新时间戳。
6. **IDE 集成层**：LSP `didChange`/`didSave`（触发 TS 服务器重新报红线）、通知 VSCode
   刷 diff、`structuredPatch` 给 TUI 画 diff。

### 3.3 回给模型的是「一句话」不是「全文」
官方 `mapToolResultToToolResultBlockParam` 回的是
`The file X has been updated successfully.` ——**不回文件全文，省 token**。
我的 `已编辑 path（替换 1 处）` **思路完全一致**。这是 token 经济学的小细节。

### 3.4 三个最硬核的暗坑（面试加分项）
1. **先读才能改 + 读后被改检测（TOCTOU/并发安全）**：edit 前必须有 Read 记录
   （`readFileState` 时间戳），且文件 mtime 不能晚于读取时刻——否则可能基于旧内容
   覆盖掉用户/linter 的改动，直接报错让你重读。我的 edit_file 每次"现读现改"天然规避，
   但也因此**没有"基于你看到的那版"的语义**（多 agent/有人在编辑时会有差异）。
2. **desanitize**：模型 serving 层会把 `<function_results>`、`<name>` 等特殊标签
   sanitize 成 `<fnr>`、`<n>` 再喂给模型，模型输出的也是短码——工具必须**还原**
   才能和真实文件匹配。这是"**模型看到的文本 ≠ 磁盘上的真实文本**"的暗坑。
3. **弯引号**：模型几乎打不出弯引号（' ' " "），但文件（尤其文档/md）里常有；
   要弯直归一化才能匹配，替换完还要把风格还原回去。

## 4. 面试金句速取
- "edit_file 比 `sed` 安全在**唯一性约束**：old_string 必须唯一命中、否则拒绝，逼模型
  带足上下文，杜绝子串误伤。这个 `split().length-1` 计数法，我和 Claude Code 官方实现一致。"
- "官方在这内核外包了**六层**：schema / 描述 / 校验 / 容错 / 原子写 / IDE 集成。最见
  工程深度的是 **TOCTOU 防护**（先读才能改+读后被改检测）和 **desanitize**（模型看到的
  文本经过 sanitize，要还原才能匹配真实文件）。"
- "工具回给模型的是'已更新成功'一句话、不是文件全文——**省 token**；这点我的 mini 版思路一致。"
- "并行工具调用的本质是**一轮内多个 tool_use**，回填时所有 tool_result 必须配齐、
  放进同一条 user 消息——不是多线程，是协议要求。"
