## 1. 工具系统抽象

**我的实现**（`src/tools.ts`）：
```ts
interface Tool { name; description; input_schema; execute(input): string }
const tools: Tool[] = [read_file, write_file, edit_file, glob, grep, bash]
const toolMap = new Map(tools.map(t => [t.name, t]))
```
loop 按 name 查表调 execute，**加工具不改 loop**（开闭原则，`src/m2.ts`）：
```ts
const tool = toolMap.get(b.name)
const out = tool ? tool.execute(b.input) : `[未知工具] ${b.name}`
```

**cc-haha 对照**：`buildTool({...}) satisfies ToolDef`，不是两件套而是**十几个钩子**——
`name` / `prompt()`(描述) / `inputSchema`(zod) / `validateInput()` / `call()`(执行) /
`checkPermissions()` / `isConcurrencySafe()` / `isReadOnly()` / `mapToolResultToToolResultBlockParam()`(结果转 tool_result) /
`renderToolUseMessage()`(TUI) / `inputsEquivalent()` / 一堆埋点。**骨架同构**：name + schema(给模型) + execute/call(给代码)。我没有的是外壳：权限、校验、并发标志、渲染、埋点。

**面试点**：工具 = **声明(schema 给模型看) / 实现(execute 给代码跑) 分离**（像 interface/SPI vs impl）；注册表按 name 分发 = 开闭，加工具零改主循环。

## 2. 并行工具调用（协议层）

模型一轮 response 里可以一次吐出**多个 tool_use 块**。处理代码就是朴素三件套（`src/m2.ts:44-54`）：
```ts
const results = []                       // ① 数组，装得下多个
for (const b of res.content) {           // ② 遍历这一轮全部 block
  if (b.type === "tool_use") {
    const out = toolMap.get(b.name).execute(b.input)
    results.push({ type: "tool_result", tool_use_id: b.id, content: out })  // 用 id 配对
  }
}
messages.push({ role: "user", content: results })   // ③ 全部塞进同一条 user 消息
```

**三铁律**（D1 来的，这里复用）：
- assistant 回合要原样回传完整 content；
- 一轮可有多个 tool_use；
- **所有 tool_result 必须配齐(用 tool_use_id 配对) + 放进同一条 user 消息**。

**两种一写就废的反例**：
- `res.content.find(b => b.type==="tool_use")` 只取第一个 → 其余 tool_use 没有配对 result → **API 直接报错**。
- 每个 tool_use 各 `messages.push` 一条 user 消息 → 结构错乱。

> "并行"在代码里**没有任何特殊处理**——一个 for + 一个数组 + 一次 push。模型负责"一轮发多个"，你只要"全收齐、配好 id、一条消息回去"。

实测：任务"同时读 m0/m1/tools.ts" → 第 1 轮模型返回 3 个 read_file → loop 收成 3 个 tool_result 一条回填 → 第 2 轮总结。

## 3. 并行 / 并发 / 多线程辨析（执行层）★ 易混

**关键纠偏**：上面 m2.ts 那段是**纯同步、串行**的，没有多线程也没有并发。
- `tool.execute()` 里是 `readFileSync`/`execSync`，**同步阻塞**；`for` 是同步循环。
- JS/JSC 是**单线程**：for 没跑完，走不到 `messages.push`，更走不到下一轮请求。
- 所以"所有工具执行完才发下一个请求"是**单线程同步天然保证的**，不需要任何同步原语。

**"并行"这个词的两层（别混）**：
| 层 | 谁 | 含义 |
|---|---|---|
| 并行**请求** | 模型 | 一轮 response 里一次给出多个 tool_use |
| **执行** | 你的代码 | for 一个个串行同步跑，**没有并发** |

**想真正并发执行**才需要 `Promise.all`（execute 得先改 async）：
```ts
const results = await Promise.all(
  toolUses.map(async b => ({ type:"tool_result", tool_use_id:b.id,
    content: await toolMap.get(b.name).execute(b.input) }))
)
```
辨析（Kotlin/Android 直接映射）：
- 当前 for + 同步 execute ≈ 主线程顺序调阻塞函数
- `Promise.all` + async ≈ Kotlin `listOf(...).map{ async{} }.awaitAll()` — **并发挂起、仍单线程**（事件循环在 I/O 等待时切换）
- 真·多线程 ≈ Worker / 线程池

> **JS 的"并发"≈ 协程，不是线程。Promise.all 是 concurrent，不是 parallel，更不是 thread。**

**局限 → D3 动机**：mini-cc 同步串行，一个 bash 死循环会**卡死整个 loop，Esc 都打不断**（单线程被那行同步调用占着）。D3 要把 execute 改 async + `AbortController` 可中断。

## 4. cc-haha 工具调度（分批混合）★

**串行还是并行？都不是——按"并发安全性"分批**：连续只读并发、写工具串行、批间严格保序。源码 `services/tools/toolOrchestration.ts`。

**① 每个工具自报能否并发**（`Tool.ts:402` `isConcurrencySafe(input)`，`Tool.ts:759` 默认 `false`）：
- 只读工具硬编码 true：`FileReadTool.ts:373` / GrepTool / GlobTool `return true`
- 写工具(edit/write) **不实现** → 吃默认 `false`
- **BashTool 按 input 动态判**（`BashTool.tsx:434`）：`isConcurrencySafe = isReadOnly(input)`，而 isReadOnly **解析命令**——`git status/ls` → true，`rm/写` → false

**② `partitionToolCalls`（`toolOrchestration.ts:91`）切批**：连续只读合并成一批；每个写工具自己单独一批；**保持原始顺序**，写工具会打断并发批。

**③ `runTools`（:19）按批跑**：只读批→`runToolsConcurrently`（:152，`all(tasks, 10)`，并发上限默认 10，可由 `CLAUDE_CODE_MAX_TOOL_USE_CONCURRENCY` 调）；写批→`runToolsSerially`（:118，朴素 for await）；**批间 for 顺序 await，严格串行**。

例：`[read, grep, edit, read, bash]` → 批1`[read,grep]`并发 · 批2`[edit]` · 批3`[read]`(被 edit 打断单独) · 批4`[bash]`，批间串行保序。

**关键事实**：
- **API 下发的 tool_use 不带任何并发标志**，只有 `{id,name,input}`。串/并是**客户端按 name 找到本地 Tool、调 `isConcurrencySafe(input)` 本地算的**。
- `all()`（`utils/generators.ts:32`，`Promise.race` 合并）**自身没有"一个失败取消其他"的逻辑**。

**为什么这么设计**：
- 只读无副作用 → 并发省墙钟时间；
- 写有副作用 → 串行防冲突（两个 edit 改同一文件 / bash 改了影响后面 read）；
- 写工具打断并发批 → **保依赖顺序**（edit 后面的 read 可能要读 edit 结果，不能提前并发）。

**两个工程之美**：
- **fail-safe 默认 `false`**：新工具忘了声明 → 自动串行，绝不误并发（默认拒绝、显式放行）。
- **复用**：bash 的 `isConcurrencySafe = isReadOnly`，而 isReadOnly 又复用**权限系统**的命令解析。一套"这命令是否只读"喂给 权限 / 只读 / 并发 三处。

体量：我的 `edit_file` **23 行**；cc-haha `FileEditTool.ts(625)+utils.ts(775)+prompt.ts+types.ts ≈ 1500+ 行`。

```ts
// 我 tools.ts：       const hits = content.split(old_string).length - 1
```

## 6. 三个硬核暗坑

### 6.1 desanitize —「模型看到的 ≠ 磁盘上的」
**原理**：文件内容塞进 prompt 前，Anthropic 服务端会 **sanitize**——把 Claude 的"协议词汇"替换成短码再喂给模型，防止文件里的字面量假冒协议标记（= prompt 注入）。被替换的（`utils.ts:531` `DESANITIZATIONS`）：
```
<function_results> <result> <output> <error> <system> <name>   ← 工具/结构标签
\n\nHuman:  \n\nAssistant:                                      ← 角色分隔符
<EOT> <META> <SOS>                                              ← 控制 token
```
手法：缩写(`<function_results>`→`<fnr>`)或插空格(`<META>`→`< META >`)。

**怎么坑到 edit**：模型照"它看到的"(短码)写 old_string → 拿去**磁盘原文**(`<function_results>`)匹配 → 找不到。`desanitizeMatchString`(`utils.ts:557`) 在精确匹配失败后，把短码**还原**再匹配；new_string 同步还原（否则把真实标签写成短码、写坏文件）。**先试精确、失败才还原**。

**类比**：HTML 转义——页面显示 `<script>`，源码是 `&lt;script&gt;`；想改"看到的 `<script>`"，工具得知道源码里是转义形态。本质同防 SQL 注入的参数转义。

**对 mini-cc**：你没这层，撞上含协议词的文件会报"找不到"——但概率极低，**知道即可，不用补**。

### 6.2 TOCTOU 乐观锁 —「先读才能改 + 读后被改检测」
**本质**：给文件编辑加乐观锁，**mtime 当版本号**，`readFileState` 当"我读到哪个版本"的记录本（= 数据库 version 字段 / HTTP ETag）。

**钥匙**：存的 timestamp 是**读取那刻文件的 mtime**，不是墙钟时间。三段闭环：
```
① Read 写入(FileReadTool.ts:1032)
   readFileState[path] = { content: 快照, timestamp: floor(mtimeMs) }   ← 记下"我读的版本"
   ⟨中间 用户/linter/别的 agent 可能改文件 → 磁盘 mtime 变新⟩
② Edit 校验(FileEditTool.ts)
   ├ 没记录 或 isPartialView(只读了局部)  → 拒:"先读再改" (:275)
   ├ 当前 mtime > 记录 timestamp           → 被动过 (:290)
   │    ├ 全量读 且 内容仍相同?            → 放行(mtime 假变，见下)
   │    └ 否则                            → 拒:"读后被改，请重读"
   ├ 写盘前再验一次(validate→write 还有窗口，throw) (:451)
   └ 写完: readFileState[path]={content:新, timestamp:新 mtime} (:519)  ← 更新版本号
```
`getFileModificationTime`(`utils/file.ts:66`) = `floor(statSync.mtimeMs)`，与写入端同量纲。

**四个精妙点**：
1. mtime 当版本号 = 乐观锁；
2. **内容兜底**（mtime 变但全量读+内容相同则放行）——修正 mtime 不可靠（Windows 云同步/杀毒会乱改 mtime）；
3. validate 和 call **各查一次**（两步间也是 TOCTOU 窗口），call 里那次是 throw（更硬）；
4. 写完更新记录，否则自己改完被自己的检查拦住。

**对 mini-cc**：你"现读现改"（execute 内当场 readFileSync→replace→write）**天然规避 TOCTOU**，但**没有"版本"概念**——若模型基于旧版本构造 old_string、文件已被改，你要么匹配失败、要么在新内容上误改且**毫不察觉**。这是正确性 vs 复杂度的取舍。

### 6.3 弯引号归一
模型几乎打不出弯引号(`'' ""`)，文档/md 里却常有。匹配时两边归一成直引号(`findActualString` `utils.ts:73`)，替换后把 new_string 还原成文件原引号风格(`preserveQuoteStyle` :104，连缩写 don't 的撇号都判)。

## 7. zod（schema 层为什么不手写）

**zod 是什么**：TS 的**运行时校验 + 类型推导**库。补的是 TS 的根本缺陷——**类型只活在编译期，运行时被擦除**，所以 `interface` 管不了外部数据（模型输出 / HTTP body 都是 `any`）。zod 的 schema 对象**运行时还活着**，一处声明三用：
1. 运行时校验 `schema.safeParse(unknown)`；
2. 编译期类型 `z.infer<typeof schema>`（不用手写 interface）；
3. 导出 JSON Schema 发给模型（`z.toJSONSchema`）。

**Kotlin 类比**：≈ `@Serializable data class` + kotlinx.serialization——一处声明，编译期类型 + 反序列化校验。TS 原生没有，zod 补上。

**对比**：
| 写法 | 问题 |
|---|---|
| ❌ mini-cc 现状：手写 JSON Schema(发模型) + `execute(input:any)` 裸用 | 零校验、零类型，无条件信任模型 |
| ⚠️ 手写三份(schema+interface+if 校验) | 三份割裂，加字段要改三处、漏一处出 bug |
| ✅ cc-haha：zod 一处定义 | 改一处，类型/校验/发模型自动同步 |

**两个关键 modifier**：
- `strictObject`：模型幻觉出多余字段 → **直接报错**（普通 `object` 默默忽略 → 静默 bug）。
- `semanticBoolean`（`utils/semanticBoolean.ts`，真身就 5 行）：
  ```ts
  z.preprocess(v => (v === 'true' ? true : v === 'false' ? false : v), z.boolean())
  ```
  模型常把 bool 写成字符串 `"false"`。三方对比：
  - `z.boolean()`：`"false"` → 校验失败（太严，模型常触发）
  - `z.coerce.boolean()`：`"false"` → **true**（JS truthiness 陷阱，更糟）
  - `semanticBoolean`：显式判 `"false"`→false。且 preprocess 导出给 API 的 schema 仍是 `{"type":"boolean"}`——**字符串容错是客户端隐形兜底，不写进 schema**（不鼓励模型乱传）。

**灾难复现**（不归一）：`if ("false")` → 非空字符串恒为真 → "只改一处"变"全部替换"。

> 金句：`execute(input:any)` 是无条件信任模型；zod 是在工具边界给模型输出上**运行时类型闸**——**LLM 输出本质是不可信外部输入**，该当外部数据校验。

## 8. 权限审批 × 并发（交互边界）★

场景：一轮里有个工具需审批，用户拒绝，后续怎样？源码 `services/tools/toolExecution.ts`。

**审批在执行之前**（`streamedCheckPermissionsAndCallTool`，先审后执）：
```ts
// toolExecution.ts:995  权限决策出来
if (permissionDecision.behavior !== 'allow') { ...push is_error rejection...; return }  // :1103
// ...
const result = await tool.call(...)   // :1207  真正执行在 return 之后
```
→ **被拒工具的逻辑(tool.call)绝不会执行**。并发的是多个「审批→执行」流水线，不是"先执行后审批"。

**拒绝触发 abort（条件性）**（`PermissionContext.ts:166`）：
```ts
if (isAbort || (!feedback && !contentBlocks?.length && !sub)) toolUseContext.abortController.abort()  // :170
```
→ 纯拒绝(No)/Esc → abort → 后续批次每个工具在 `runToolUse:415` 看到 aborted → CANCEL 不执行。（带反馈的拒绝走 ask，不一定 abort。）

**三种命运**（`[read_A, bash_B(需审批), edit_C]`）：
| 工具 | 位置 | 结果 |
|---|---|---|
| 被拒 bash_B 自己 | — | **逻辑绝不执行**（审批前置） |
| 同并发批的只读工具 | 拒绝前已抢跑 | 可能已执行完，**但无副作用、无害** |
| 拒绝之后的批 (edit_C) | abort 之后 | **CANCEL，不执行** |

**金句 / 只读才并发的第三个理由**：`all()` 不取消其他，所以审批期间同批只读工具会"抢跑"。**只有无副作用的操作容忍得起这种抢跑**——若让写工具并发，"你还在审批 B、edit_A 已落盘且不可逆"。所以**写必须串行成批、逐个过审批，杜绝未批先改**。这是正确性+安全的硬约束，不是性能优化。

补：cc-haha 里只读 bash 通常被 `isReadOnly→allow` 自动放行不弹窗，所以"只读 bash 还需审批"多半不成立；但需审批工具的语义如上。

## 9. headless core + 多 host 架构

**cc-haha = 一份核心 + 多个外壳**（三个 package.json）：
| package | 是什么 | 证据 |
|---|---|---|
| `claude-code-local`(root) | **CLI = 核心** | `bin: claude-haha`，src/ 里 query.ts/工具/权限全在这 |
| `claude-code-desktop`(desktop/) | **Tauri 桌面 App** | `@tauri-apps/*` + `xterm` + react-diff-viewer；`build:sidecars` 把 CLI 当 **sidecar** 打包 |
| `claude-code-im-adapters`(adapters/) | **IM 入口** | telegram/feishu/wechat/dingtalk |

**VSCode 不是独立插件**：repo 里关于 VSCode 只有一根 MCP 桥 `services/mcp/vscodeSdkMcp.ts`：
```ts
// :39 改完文件后，核心经 MCP 朝外部 VSCode 插件喊一嗓子
notifyVscodeFileUpdated(...) {
  if (process.env.USER_TYPE !== 'ant' || !vscodeMcpClient) return   // :44 门控内部用户
  vscodeMcpClient.client.notification({ method:'file_updated', params:{...} })
}
```
→ VSCode 插件是**外部独立项目**，通过 **MCP 双向**接成核心的一个宿主（agent 调编辑器 + 编辑器收文件变更通知）。**用的就是同一份核心**。


> 金句：Claude Code 同时支持 CLI/桌面/IM/IDE，靠的不是多份实现，而是 **headless core + 多 host**。IDE 集成尤其巧——把编辑器当 **MCP peer**，双向通信。说明 MCP 不只"给 agent 接工具"，也能"给 agent 接宿主环境"。⑥ 那层因此该叫"向宿主广播变更的钩子"，核心里没有 IDE 实现。

## 10. 面试金句总集
- **工具系统**："工具 = 声明(schema)/实现(execute)分离 + 注册表分发，加工具不改主循环（开闭）。"
- **并行 vs 并发**："并行工具调用是协议层概念——模型一轮发多个 tool_use；执行层我是同步串行。真要并发用 Promise.all，那是单线程事件循环的 concurrency，不是多线程。"
- **调度**："cc-haha 按并发安全性分批：连续只读并发(上限10)、写工具串行、批间保序。依据不是快慢，是**有没有副作用 + 依赖顺序**。并发安全 API 不下发，是客户端按 input 本地判，默认 false 是 fail-safe。"
- **edit 唯一性**："`split().length-1` 计数法我和官方一致；官方外包六层(schema/描述/校验/容错/原子写/宿主广播)，内核之外全是工程化。"
- **desanitize**："模型看到的文本经 serving 层 sanitize(`<function_results>`→`<fnr>`)，和磁盘不一致，严谨 edit 工具要 desanitize 反解码。"
- **TOCTOU**："文件编辑加了乐观锁——mtime 当版本号、readFileState 当读取快照，改前校验版本，防并发丢更新。"
- **zod**："LLM 输出是不可信外部输入，该在工具边界做运行时校验；zod 一处定义=类型+校验+发模型三用。"
- **权限×并发**："只读才能进并发批的深层原因是权限——审批期间同批工具会抢跑，只有无副作用的操作容忍得起；写工具必须串行逐个过审批，杜绝未批先改。"
- **架构**："headless core + 多 host，IDE 当 MCP peer 双向接入——一次构建、到处接入。"
