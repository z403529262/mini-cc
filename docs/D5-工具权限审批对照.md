# D5 — 工具权限审批（mini-cc M5 × cc-haha 对照）

> 给自主 agent 装刹车：在「模型决定调工具」和「工具真执行」之间插一道权限门。What/Why 优先（见 `docs/讲解约定.md`）。

## 0. 这一关是什么、为什么

**是什么**：在 loop 的「拿到 `tool_use` → `execute`」之间，插一道判定——该放行 / 问用户 / 拒绝。

**为什么**：D1-D4 把 agent 武装成能自主循环、读写文件、跑 bash 的自主体，却一直没有刹车（`m4.ts:102` 模型说调啥就 `execute` 啥）。**能力即风险**：prompt 注入（恶意文件内容让它 `rm`）、幻觉（自以为该删）、误判（理解错范围）都可能让它毁掉你不想动的东西。权限门 = human-in-the-loop，是自主性与安全性的平衡阀。**没有这道门的 agent 不能交给用户跑。**

# 第一部分：M5 我们怎么做（最小版）

## 1. 三态判定，不是开关

**是什么**：结果是 `allow | ask | deny` 三态，不是"准/不准"二元。allow=只读/安全自动放行；ask=有副作用问一次；deny=明确危险直接拒。

**为什么**：全自动=危险，每步都问=烦到没法用。三态在"安全"与"可用"间找平衡。
（验证：真 CC 的 `PermissionBehavior` 也正是 `'allow' | 'deny' | 'ask'`，一字不差。）

## 2. 危险在参数，不在工具名

**是什么**：判定对象是「工具 + 这次的参数」，不是「工具」。
**为什么**：`bash` 既能 `ls`（无害）也能 `rm -rf ~`（灾难）。光看工具名一刀切是错的，必须解析具体输入。

## 3. 门必须在 execute 之前（+ 权限 × 并发坑）

**是什么**：判定在 `execute` 之前，拒绝就根本不执行。
**为什么**：D2 的"只读才并发"伏笔在此兑现——写操作要审批，审批必须在执行前完成。若并发把审批和执行穿插，会**未批先改**。规则：只读可并发自动放行，写操作串行 + 先批后行。

## 4. M5 怎么做（从简）

- `tools.ts`：`Tool` 接口加 `readOnly`，6 个工具自报（read/glob/grep=true）
- `permission.ts`：`checkPermission(tool, input)` → 三态；bash 用 SAFE/DANGER 正则分级
- `m5.ts`：loop 在 `execute` 前插门；deny 回填 `is_error`、ask 问 y/n、allow 执行
- 验证：`demo/permission-check.ts`（判定 13/0）+ `demo/permission-flow-check.ts`（后果 6/0）

⚠️ **诚实的局限**：正则不是真安全，挡不住 `r''m -rf`、变量拼接 `X=rm;$X -rf`、base64。真安全做不到纯静态判定——真 CC 的答案见第二部分（§10 LLM 分类器）。

# 第二部分：解剖 cc-haha（真实权限系统）

> `utils/permissions/` 25 个文件、`bashPermissions.ts` 单文件 2621 行。我们的 ~50 行是它的骨架雏形。

## 5. 判定优先级链（核心）

`hasPermissionsToUseToolInner`（permissions.ts:1158）从上往下短路：

| 顺序 | 检查 | 命中结果 |
|---|---|---|
| 1a | deny 规则（整个工具） | **deny**（最高优先级） |
| 1b | ask 规则（整个工具） | ask |
| 1c | 工具自查 `checkPermissions`（bash 子命令规则） | 拿到结果 |
| 1d | 工具实现返回 deny | deny |
| 1f | 内容级 ask 规则（如 `Bash(npm publish:*)`） | ask（**bypass 也尊重**） |
| 1g | safetyCheck（`.git/`、`.claude/`、shell 配置） | ask（**bypass-immune**） |
| 2a | mode = bypassPermissions | allow |
| 2b | always allow 规则 | allow |
| 3 | 都没命中 → passthrough | **默认 ask** |

**两个硬结论**：① **deny 最高、bypass 越不过**（1a/1d 在 2a 之前）；② **默认兜底是 ask 不是 deny**（步骤 3）。

## 6. 坐实「硬 deny vs ask」

mini-cc 把危险命令硬 `deny`，真 CC 不是——解剖坐实：
- 真 CC **没有内置的 `rm -rf → deny`**。`rm -rf` 走 bash 自查 → 没匹配 allow 规则 → passthrough → **默认 ask（弹确认）**。
- **deny 只来自用户显式配置**的 deny 规则（`getDenyRuleForTool` 从 `alwaysDenyRules` 取，源自 settings.json）。
- **设计哲学**：系统不替用户做"绝对禁止"的决定；拿不准就问，把最终决定权交给人。硬编码黑名单会**误伤、会过期、给虚假安全感**。
- 我的 mini-cc 硬 deny 是"教学演示 + 强保护"的取舍；真 CC = ask + 可配 deny + LLM 分类器。

## 7. dangerousPatterns 的真相

`dangerousPatterns.ts` 里的 python/node/eval/sudo/curl… **不是命令黑名单**，是**「危险的 allow 规则前缀」黑名单**：
- 一条 `Bash(python:*)` = 允许任意 python = 任意代码执行 = 绕过分类器；
- 进 auto 模式时 CC **strip（剥离）** 这种过宽 allow 规则（存进 `strippedDangerousRules`），逼命令回落到分类器/ask；
- 一句话：**危险的不是命令，是给它的宽泛通行证。**

## 8. 权限模式（5 + 2）

| 模式 | 语义 |
|---|---|
| `default` | 按规则判，没命中 → ask（最常用） |
| `plan` | 计划模式，只读探索不执行写（**怎么做见 §16**） |
| `acceptEdits` | 自动放行文件编辑（CWD 内），危险命令仍查 |
| `bypassPermissions` | YOLO 全放行（但 deny / 内容级 ask / safetyCheck 仍 immune） |
| `dontAsk` | 不问 |
| `auto` / `bubble` | ant-only，用 LLM transcript 分类器 |

## 9. 规则是配置驱动的 + 前缀匹配

**规则结构**：`{ source, ruleBehavior, ruleValue:{toolName, ruleContent} }`
- **来源**（source）：userSettings / projectSettings / localSettings / cliArg / session…多源叠加
- **三组**：`alwaysAllowRules` / `alwaysDenyRules` / `alwaysAskRules`

**前缀匹配**（shellRuleMatching.ts）规则三形态：
- `exact`：`Bash(npm run test)` 精确整条
- `prefix`：`Bash(npm:*)` → 前缀 `npm`，匹配所有 npm 开头（legacy `:*` 语法）
- `wildcard`：`Bash(git * --force)` → `*` 转 `.*` 正则；`git *` 让尾部参数可选（也匹配裸 `git`）；`\*` 转义字面星；dotAll 匹配 heredoc 换行

## 10. 正则 → LLM 分类器（静态判定的尽头）

我标注"正则不是真安全"，真 CC 的答案 = **`yoloClassifier`：用 LLM 当分类器**判命令危险性。
- `YoloClassifierResult.shouldBlock` + thinking + **2-stage**（fast 快筛 → thinking 深思）
- 输入含命令 + cwd + 描述，**理解语义和上下文**（正则做不到）
- 完整 telemetry（token / duration / request_id）
- `classifierDecision.ts` 的 `SAFE_YOLO_ALLOWLISTED_TOOLS`：read/grep/glob/task 等只读工具**免分类器**（省 API 调用）

**洞察**：静态规则的尽头，是让另一个模型理解语义来判断。但分类器也非 100%——最终仍有"人审批"兜底。

> 深挖（2-stage 是不是额外的"思考循环"、thinking 到底是什么、`sideQuery` 怎么回事、换 provider 能不能用）见 **§17–18**。

## 11. bypass-immune：YOLO 也有底线

即使 `bypassPermissions`（YOLO），几类强制 ask：
- 内容级 ask 规则（用户显式配的 `Bash(npm publish:*)`）
- safetyCheck：写 `.git/`、`.claude/`、`.vscode/`、shell 配置 → 强制问

**为什么**：有些操作的破坏性/不可逆性，不该被一个全局开关豁免。

# 第三部分：对照与面试

## 12. mini-cc vs 真 CC

| 维度 | mini-cc M5 | 真 CC |
|---|---|---|
| 三态 | allow/ask/deny | 同（+ passthrough 兜底） |
| 优先级 | deny>ask>allow | deny>ask规则>工具查>bypass>allow>默认ask |
| 危险命令 | 内置正则硬 deny | 默认 ask；deny 靠用户规则；危险靠 LLM 分类器 |
| 只读判定 | `Tool.readOnly` | `SAFE_YOLO_ALLOWLISTED`（分免检 / 需分类器） |
| 规则 | 硬编码 | 配置驱动（多源 × 三组 × 三形态） |
| 模式 | 无 | default/plan/acceptEdits/bypass/dontAsk/auto |
| 命令判定 | 正则 | LLM 分类器（2-stage 语义理解） |
| bypass | 无 | 有，但敏感操作 immune |

## 13. 面试金句

- "权限门是三态 allow/ask/deny，核心是**优先级链：deny 最高、连 bypass 越不过；默认兜底是 ask 不是 deny**——系统不替用户做绝对禁止，拿不准就问，决定权交人。"
- "危险命令不靠硬编码黑名单（会误伤、过期、给虚假安全感），靠 **LLM 分类器理解语义 + 人审批兜底**。静态正则只是第一道粗筛。"
- "dangerousPatterns 防的不是命令，是**给解释器的宽泛 allow 通行证**（`Bash(python:*)` = 任意代码），auto 模式会 strip 掉。"
- "YOLO 模式也有底线：写 `.git/`/`.claude/` 等敏感路径 **bypass-immune**，强制问。"
- "判定对象是「工具 + 参数」不是「工具」——同一个 bash，`ls` 和 `rm -rf` 天差地别。"

# 第四部分：深挖答疑（plan 模式 + LLM 分类器 + sideQuery）

> 封板前几轮深问的沉淀：plan 模式怎么锁只读、classifier 是不是额外"思考循环"、`sideQuery` 到底是什么。仍 What/Why 优先。

## 16. plan 模式怎么做的

**是什么**：6 种模式里最特别的——只读探索 + 规划，产出计划给用户批，批了才动手。

**为什么（和权限门的分工）**：权限门管**单个危险操作**（"这条 `rm` 能不能跑"）；plan 模式管**整体方向**。agent 对复杂任务直接动手，方向错了已改一堆文件、回滚巨贵。plan 模式把"对齐方案"提到"写第一行代码之前"——human-in-the-loop 从"操作级"升到"方案级"。

**怎么做（4 机制）**：
1. **进入**：`EnterPlanMode`（或 shift+tab）→ `mode='plan'`；`prepareContextForPlanMode`(permissionSetup.ts:1462) 把当前模式存进 **`prePlanMode`**（为退出恢复）。
2. **约束只读 = 软约束，不是硬牢笼**（最反直觉）：
   - Prompt 主导：EnterPlanMode prompt + plan_mode attachment 强引导"探索→设计→提计划，别实现"；
   - 权限兜底：plan 非 bypass/acceptEdits → 写操作不自动放行 → 落到默认 ask；
   - ⚠️ **坐实**：grep 三个写工具（FileEdit/FileWrite/Bash）**无一处 plan 特判**，全库无"plan 禁止写"硬拦话术。**只读靠 prompt 让模型自觉 + 默认 ask 兜底，没有硬开关。**
3. **产出 = 写到 plan 文件，不是聊天**：模型把计划写进 `getPlanFilePath`；ExitPlanMode prompt 明说"不接受 plan 作参数，从你写的文件读"。
4. **退出 = 唯一正门**：`ExitPlanMode` → checkPermissions 返回 `ask("Exit plan mode?")` → 用户批 → 恢复 `prePlanMode` → 注入 `plan_mode_exit` attachment → 开干。

```
default ──EnterPlanMode──▶ plan(只读+写计划文件) ──ExitPlanMode+你批准──▶ 恢复 prePlanMode ──▶ 开干
   └─ 存 prePlanMode ────────────────────────────────────────────────┘
```

**精妙设计（why）**：
- ExitPlanMode **不传计划内容、从文件读**：避免参数里被截断/重复；文件还能被 Read/VerifyPlanExecution 复查、被 web UI 编辑。
- ExitPlanMode **本身就是个 `ask` 权限**：把"提交计划求批准"工具化，复用 D5 的审批机制。
- prompt 明令"别用 AskUserQuestion 问'计划行吗'"——那正是 ExitPlanMode 的职责。
- `prePlanMode` 记忆：acceptEdits 时进 plan，退出回 acceptEdits，不打断节奏。
- **为什么软约束而非硬拦每个写工具**？硬拦要在每个工具塞 plan 判断（散乱、易漏、新工具易忘）；"强 prompt + 默认 ask + 单一正门"更集中，贴合"决定权交人"哲学。

锚点：`EnterPlanModeTool/prompt.ts`、`ExitPlanModeTool/ExitPlanModeV2Tool.ts`(:221 checkPermissions=ask、:243 call 读文件)、`permissionSetup.ts:1462`(prePlanMode)、`attachments.ts:1268`(plan_mode_exit)。

## 17. yoloClassifier 是不是额外的"思考循环"

**问题**：auto 模式调工具前那个 2-stage（fast/thinking）是不是又加了几轮循环？

**不是主 loop 的循环，是旁路单次调用**：classifier 走 `sideQuery`（见 §18），独立于主对话。主 agent 吐 `tool_use` 后，权限层"背着它"问一个模型"这命令危险吗"，主 agent 的轮次/上下文不受影响。

**"thinking" 是分类器自己的 XML CoT，不是主 agent 的、也不是 API extended thinking**：
- API 自带的 extended thinking 在这里**特意关掉**（`thinking: disableThinking`，yoloClassifier.ts:785）——为省 token；
- 改用 XML 标签 `<thinking>…</thinking>` 让**分类器模型**自己写一段轻量推理，再给 `<block>`/`<reason>`。

**几次调用**（2-stage 'both' 默认）：

| Stage | max_tokens | 行为 |
|---|---|---|
| 1 fast | **64** + `stop_sequences:['</block>']` | 催它立刻吐 allow/block，一吐 `</block>` 就停 |
| → allow | — | **立即返回(:807)，stage 2 不跑** → 共 **1 次** |
| → block | — | 才升级 stage 2 thinking 深思（减误杀）→ 共 **2 次** |

绝大多数安全命令 = +1 次极轻调用（64 token 秒回），少数可疑才 +2。

**最关键：只在 auto 模式发生**。default 模式（日常）压根不跑 classifier——直接弹 `Allow? y/n` 问人。只读工具（allowlist）也免跑。**不存在"每次工具都被偷加 LLM"。**

**为什么这么设计**：auto 模式要"自动放行不打扰人"，代价是用模型调用替人判断。于是拼命压成本：fast 只 64 token + 一拿结论就停、2 stage 共享 system+content 挂 cache_control(1h)、只对可疑的才深思。本质是 **default 省 token 费人力（问你） vs auto 省人力费 token（问模型）** 的权衡。

## 18. sideQuery 到底是什么

**是什么**：单次裸 API 调用的轻封装（`sideQuery.ts:109`，注释自称 "Lightweight API wrapper for side queries outside the main conversation loop"）。包一层 `client.beta.messages.create`——发一个请求、拿一个 `BetaMessage`，**无循环、无 tool_result 回填、无多轮**。统一处理 OAuth fingerprint / attribution / betas / model 归一化，是所有"私下问模型一句"的统一出口（权限解释器、会话搜索、模型校验、classifier 都用它）。

**和 sub-agent 的区别**（别混）：

| | 本质 | 循环 | 工具 | 上下文 |
|---|---|---|---|---|
| sub-agent(AgentTool/Task) | 完整 agent | 自己的 loop | 自己的工具集 | 隔离完整窗口 |
| sideQuestion(用户 side Q) | 轻 query | `maxTurns:1` | 禁用 | 走 query 骨架 |
| **sideQuery**(classifier 用) | **裸单次调用** | 无 | 可选(不带) | 调用方传什么就什么 |

**换模型么**：classifier 的 model = `getClassifierModel()`(yoloClassifier.ts:1334)，**默认兜底 = `getMainLoopModel()`（主模型）**，可被 env/GrowthBook 配成专用小模型降本。默认不换。

**叠加上下文么——不**：sideQuery 不继承主历史，只用 `opts.messages`。classifier 传 `buildTranscriptForClassifier` 构造的精简 `<transcript>`（判危险必需的只读快照）；并带 **`skipSystemPromptPrefix:true`** 跳过 CC 主系统提示，只挂自己的 system + attribution header。结果 `shouldBlock` 不回主对话。**看必需快照，不叠 CC 系统提示，不污染主线。**

**provider：换其他家能用么**：
- **机制可移植**：标准参数（model/messages/max_tokens/temperature/stop_sequences）任何 Anthropic 兼容端点通用（CC 的 client 能指智谱，同 mini-cc）。
- **优化绑 Anthropic**：`betas`、`output_format`(structured outputs beta)、`thinking`、`cache_control`、OAuth fingerprint——换非兼容家降级/失效。
- **漂亮的连带印证**：classifier 用 XML(`<thinking><block><reason>`)而非 JSON structured output，正是为**跨模型稳健**——structured output 是 Anthropic beta、挑端点(:135 `modelSupportsStructuredOutputs`)，XML 谁都能吐。它自己就在为多 provider 做兼容取舍。

锚点：`sideQuery.ts:109`(本体)、:126(getAnthropicClient 可指任意兼容端点)、:135(structured-output 按 model 能力)、:150(systemBlocks=attribution + 可跳过 prefix)、`yoloClassifier.ts:1334`(getClassifierModel 默认主模型)、:434(buildTranscriptForClassifier)、:785(关 API thinking)。
