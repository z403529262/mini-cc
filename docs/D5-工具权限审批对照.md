# D5 — 工具权限审批（mini-cc M5 × cc-haha 对照）

> 给自主 agent 装刹车：在「模型决定调工具」和「工具真执行」之间插一道权限门。What/Why 优先（见 `docs/讲解约定.md`）。

## 0. 这一关是什么、为什么

**是什么**：在 loop 的「拿到 `tool_use` → `execute`」之间，插一道判定——该放行 / 问用户 / 拒绝。

**为什么**：D1-D4 把 agent 武装成能自主循环、读写文件、跑 bash 的自主体，却一直没有刹车（`m4.ts:102` 模型说调啥就 `execute` 啥）。**能力即风险**：prompt 注入（恶意文件内容让它 `rm`）、幻觉（自以为该删）、误判（理解错范围）都可能让它毁掉你不想动的东西。权限门 = human-in-the-loop，是自主性与安全性的平衡阀。**没有这道门的 agent 不能交给用户跑。**

# 第一部分：M5 我们怎么做（最小版）

## 1. 三态判定，不是开关

**是什么**：结果是 `allow | ask | deny` 三态，不是"准/不准"二元。allow=只读/安全自动放行；ask=有副作用问一次；deny=明确危险直接拒。

**为什么**：全自动=危险，每步都问=烦到没法用。三态在"安全"与"可用"间找平衡。
（验证：真 CC 的 `PermissionBehavior` 也正是 `'allow' | 'deny' | 'ask'`，一字不差。）

## 2. 危险在参数，不在工具名

**是什么**：判定对象是「工具 + 这次的参数」，不是「工具」。
**为什么**：`bash` 既能 `ls`（无害）也能 `rm -rf ~`（灾难）。光看工具名一刀切是错的，必须解析具体输入。

## 3. 门必须在 execute 之前（+ 权限 × 并发坑）

**是什么**：判定在 `execute` 之前，拒绝就根本不执行。
**为什么**：D2 的"只读才并发"伏笔在此兑现——写操作要审批，审批必须在执行前完成。若并发把审批和执行穿插，会**未批先改**。规则：只读可并发自动放行，写操作串行 + 先批后行。

## 4. M5 怎么做（从简）

- `tools.ts`：`Tool` 接口加 `readOnly`，6 个工具自报（read/glob/grep=true）
- `permission.ts`：`checkPermission(tool, input)` → 三态；bash 用 SAFE/DANGER 正则分级
- `m5.ts`：loop 在 `execute` 前插门；deny 回填 `is_error`、ask 问 y/n、allow 执行
- 验证：`demo/permission-check.ts`（判定 13/0）+ `demo/permission-flow-check.ts`（后果 6/0）

⚠️ **诚实的局限**：正则不是真安全，挡不住 `r''m -rf`、变量拼接 `X=rm;$X -rf`、base64。真安全做不到纯静态判定——真 CC 的答案见第二部分（§10 LLM 分类器）。

# 第二部分：解剖 cc-haha（真实权限系统）

> `utils/permissions/` 25 个文件、`bashPermissions.ts` 单文件 2621 行。我们的 ~50 行是它的骨架雏形。

## 5. 判定优先级链（核心）

`hasPermissionsToUseToolInner`（permissions.ts:1158）从上往下短路：

| 顺序 | 检查 | 命中结果 |
|---|---|---|
| 1a | deny 规则（整个工具） | **deny**（最高优先级） |
| 1b | ask 规则（整个工具） | ask |
| 1c | 工具自查 `checkPermissions`（bash 子命令规则） | 拿到结果 |
| 1d | 工具实现返回 deny | deny |
| 1f | 内容级 ask 规则（如 `Bash(npm publish:*)`） | ask（**bypass 也尊重**） |
| 1g | safetyCheck（`.git/`、`.claude/`、shell 配置） | ask（**bypass-immune**） |
| 2a | mode = bypassPermissions | allow |
| 2b | always allow 规则 | allow |
| 3 | 都没命中 → passthrough | **默认 ask** |

**两个硬结论**：① **deny 最高、bypass 越不过**（1a/1d 在 2a 之前）；② **默认兜底是 ask 不是 deny**（步骤 3）。

## 6. 坐实「硬 deny vs ask」

mini-cc 把危险命令硬 `deny`，真 CC 不是——解剖坐实：
- 真 CC **没有内置的 `rm -rf → deny`**。`rm -rf` 走 bash 自查 → 没匹配 allow 规则 → passthrough → **默认 ask（弹确认）**。
- **deny 只来自用户显式配置**的 deny 规则（`getDenyRuleForTool` 从 `alwaysDenyRules` 取，源自 settings.json）。
- **设计哲学**：系统不替用户做"绝对禁止"的决定；拿不准就问，把最终决定权交给人。硬编码黑名单会**误伤、会过期、给虚假安全感**。
- 我的 mini-cc 硬 deny 是"教学演示 + 强保护"的取舍；真 CC = ask + 可配 deny + LLM 分类器。

## 7. dangerousPatterns 的真相

`dangerousPatterns.ts` 里的 python/node/eval/sudo/curl… **不是命令黑名单**，是**「危险的 allow 规则前缀」黑名单**：
- 一条 `Bash(python:*)` = 允许任意 python = 任意代码执行 = 绕过分类器；
- 进 auto 模式时 CC **strip（剥离）** 这种过宽 allow 规则（存进 `strippedDangerousRules`），逼命令回落到分类器/ask；
- 一句话：**危险的不是命令，是给它的宽泛通行证。**

## 8. 权限模式（5 + 2）

| 模式 | 语义 |
|---|---|
| `default` | 按规则判，没命中 → ask（最常用） |
| `plan` | 计划模式，只读探索不执行写 |
| `acceptEdits` | 自动放行文件编辑（CWD 内），危险命令仍查 |
| `bypassPermissions` | YOLO 全放行（但 deny / 内容级 ask / safetyCheck 仍 immune） |
| `dontAsk` | 不问 |
| `auto` / `bubble` | ant-only，用 LLM transcript 分类器 |

## 9. 规则是配置驱动的 + 前缀匹配

**规则结构**：`{ source, ruleBehavior, ruleValue:{toolName, ruleContent} }`
- **来源**（source）：userSettings / projectSettings / localSettings / cliArg / session…多源叠加
- **三组**：`alwaysAllowRules` / `alwaysDenyRules` / `alwaysAskRules`

**前缀匹配**（shellRuleMatching.ts）规则三形态：
- `exact`：`Bash(npm run test)` 精确整条
- `prefix`：`Bash(npm:*)` → 前缀 `npm`，匹配所有 npm 开头（legacy `:*` 语法）
- `wildcard`：`Bash(git * --force)` → `*` 转 `.*` 正则；`git *` 让尾部参数可选（也匹配裸 `git`）；`\*` 转义字面星；dotAll 匹配 heredoc 换行

## 10. 正则 → LLM 分类器（静态判定的尽头）

我标注"正则不是真安全"，真 CC 的答案 = **`yoloClassifier`：用 LLM 当分类器**判命令危险性。
- `YoloClassifierResult.shouldBlock` + thinking + **2-stage**（fast 快筛 → thinking 深思）
- 输入含命令 + cwd + 描述，**理解语义和上下文**（正则做不到）
- 完整 telemetry（token / duration / request_id）
- `classifierDecision.ts` 的 `SAFE_YOLO_ALLOWLISTED_TOOLS`：read/grep/glob/task 等只读工具**免分类器**（省 API 调用）

**洞察**：静态规则的尽头，是让另一个模型理解语义来判断。但分类器也非 100%——最终仍有"人审批"兜底。

## 11. bypass-immune：YOLO 也有底线

即使 `bypassPermissions`（YOLO），几类强制 ask：
- 内容级 ask 规则（用户显式配的 `Bash(npm publish:*)`）
- safetyCheck：写 `.git/`、`.claude/`、`.vscode/`、shell 配置 → 强制问

**为什么**：有些操作的破坏性/不可逆性，不该被一个全局开关豁免。

# 第三部分：对照与面试

## 12. mini-cc vs 真 CC

| 维度 | mini-cc M5 | 真 CC |
|---|---|---|
| 三态 | allow/ask/deny | 同（+ passthrough 兜底） |
| 优先级 | deny>ask>allow | deny>ask规则>工具查>bypass>allow>默认ask |
| 危险命令 | 内置正则硬 deny | 默认 ask；deny 靠用户规则；危险靠 LLM 分类器 |
| 只读判定 | `Tool.readOnly` | `SAFE_YOLO_ALLOWLISTED`（分免检 / 需分类器） |
| 规则 | 硬编码 | 配置驱动（多源 × 三组 × 三形态） |
| 模式 | 无 | default/plan/acceptEdits/bypass/dontAsk/auto |
| 命令判定 | 正则 | LLM 分类器（2-stage 语义理解） |
| bypass | 无 | 有，但敏感操作 immune |

## 13. 面试金句

- "权限门是三态 allow/ask/deny，核心是**优先级链：deny 最高、连 bypass 越不过；默认兜底是 ask 不是 deny**——系统不替用户做绝对禁止，拿不准就问，决定权交人。"
- "危险命令不靠硬编码黑名单（会误伤、过期、给虚假安全感），靠 **LLM 分类器理解语义 + 人审批兜底**。静态正则只是第一道粗筛。"
- "dangerousPatterns 防的不是命令，是**给解释器的宽泛 allow 通行证**（`Bash(python:*)` = 任意代码），auto 模式会 strip 掉。"
- "YOLO 模式也有底线：写 `.git/`/`.claude/` 等敏感路径 **bypass-immune**，强制问。"
- "判定对象是「工具 + 参数」不是「工具」——同一个 bash，`ls` 和 `rm -rf` 天差地别。"
