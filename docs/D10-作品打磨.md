# D10 · 作品打磨 — 测试策略、CI 与「5 分钟走读」

> 产出:`tests/` 六件套(67 测试 / 174 断言 / ~0.4s / 零 API key)+ GitHub Actions CI + README 重构(架构图 + 模块地图 + docs 索引)。
> 考点:工程素养展示 —— 面试官只给你的 repo 2-5 分钟,打磨 = 优化这条扫描路径的信息密度。

---

## 第一部分 · 是什么/为什么:打磨打磨的到底是什么

**面试官看 repo 的真实方式**:扫 README(30 秒)→ 看目录结构(30 秒)→ 点开一两个源文件(2 分钟)→ 有测试/CI 吗(10 秒瞥徽章)。没人会 clone 下来跑。所以打磨的对象不是代码本身(代码 D1-D9 已经写完了),而是**这条 2-5 分钟扫描路径上每一眼的信息密度**:

| 扫描点 | 打磨前 | 打磨后 |
|---|---|---|
| README 第一屏 | 文字描述 | CI 徽章 + Mermaid 架构图(GitHub 原生渲染) |
| 「这人懂工程吗」 | 只有 demo/*-check.ts 手工脚本 | 正式 `bun test` 套件 + typecheck + CI |
| 「学的深不深」 | docs/ 十篇笔记藏在目录里 | README 索引表,每篇一句钩子 |
| 「代码质量」 | tsc 有 10 个类型错误(demo) | `tsc --noEmit` 全绿,进 CI |

**为什么测试对 agent 项目特别值得讲**:LLM 输出非确定,直觉上「agent 没法单测」。但 D8 已经画过那条分界线 —— **协议层 + 逻辑层是确定的**(模型回了 tool_use,loop 该执行、该回填、该拦截,这些行为和模型说什么话无关),**输出质量才是非确定的**(归 eval 管)。D10 把左半边做实:凡是「模型回 X,代码该做 Y」的契约,全部单测覆盖。

一句话:**eval 测模型,单测测你自己写的代码。混着测 = 又慢又贵又 flaky;分开测 = 单测上 CI、eval 本地跑。**

## 第二部分 · 测试策略(本次的核心取舍)

### 测什么、不测什么

| 层 | 测不测 | 怎么测 | 归谁 |
|---|---|---|---|
| loop 状态机(执行/回填/拦截/熔断/中断) | ✅ | fake client 注入脚本 | `tests/agent.test.ts` |
| 工具行为契约(edit 唯一性/错误回填/进程组击杀) | ✅ | 真文件系统 + 真子进程 | `tests/tools.test.ts` |
| 压缩切割不变量 | ✅ | 纯逻辑 property 遍历 | `tests/compact.test.ts` |
| 权限三态 + 已知绕过的兜底 | ✅ | 表驱动 | `tests/permission.test.ts` |
| MCP 协议全流程 | ✅ | 起真 server 子进程 | `tests/mcp.test.ts` |
| 子 agent 隔离/递归防护 | ✅ | fake client + 结构断言 | `tests/task.test.ts` |
| LLM 输出质量(答对没/答好没) | ❌ 不进单测 | 任务集 + 判据统计通过率 | `demo/eval.ts`(D8) |

### 三个策略决定

**① fake client 注入 —— `runAgent` 依赖注入的第三次兑现。** D7 抽 `runAgent` 时把 `client` 做成参数,当时的直接动机是子 agent 嵌套调用;D8 批量 eval 是第二次复用;D10 测试是第三次:把「模型会怎么回」写成消息脚本(`asstToolUse(...)` → `asstText(...)`),注入假 client,loop 的每条分支就都能确定性触发了 —— 包括真模型很难稳定复现的分支(未知工具、恒返回 tool_use 触发熔断、危险命令触发 deny)。

**② 不 mock 文件系统、不 mock MCP server。** 工具的价值就在「与真实世界交互的边界处理」:`edit_file` 命中 2 次该拒绝、grep 退出码 1 不是错误、bash 中断要杀整个进程组 —— mock 掉 fs/子进程,这些就全没测到,剩下的只是在测 mock 本身。临时目录(`mkdtempSync`)隔离副作用,MCP 测试直接 spawn 真 server(`demo/mcp-server-calc.ts`)走完握手/发现/调用/关闭。**该 fake 的是「贵且非确定」的 LLM,不该 fake 的是「免费且确定」的本地世界。**

**③ 零 API key 是设计约束,不是碰巧。** 测试离线 → CI 不需要 secrets → 任何人 fork 下来 CI 都绿 → 徽章可信。反过来,如果有一个测试要 key,整条 CI 链就得配 secrets、就会因配额/网络 flaky、徽章就不再等于「代码没坏」。

### 顺手修的:typecheck 全绿

`tsc --noEmit` 暴露 `demo/rag-mini.ts` 10 处严格索引访问错误(`noUncheckedIndexedAccess` 下 `a[i]` 是 `T | undefined`)。修法与仓库既有风格一致:等长数组循环内用非空断言 `a[i]!`。修完 typecheck 进 CI —— 类型检查不进 CI 等于没有。

## 第三部分 · 每个测试文件在守什么(关键不变量)

| 文件 | 守的东西 | 最重要的那一条 |
|---|---|---|
| `compact.test.ts` | 切割安全 | ★ 对**任意** keepRecent,保留段内每个 tool_result 的配对 tool_use 也在保留段内 —— 劈开配对 = API 400,agent 当场崩 |
| `permission.test.ts` | 三级分级 | ★ 诚实边界:`X=rm; $X -rf /` 绕过 DANGER 正则后落进 **ask** 而非 allow —— 静态检测漏了还有人兜底 |
| `tools.test.ts` | 工具契约 | ★ `sleep 5` 在 100ms 内被中断且 2s 内返回 —— 验证的是「杀整个进程组」,只杀 bash 会留孤儿进程拖死管道(D3 的坑) |
| `agent.test.ts` | loop 状态机 | ★ deny 时 `approve` 回调**根本不被调用**(防「手滑按了 y」);maxTurns 熔断挡住无限工具循环 |
| `task.test.ts` | 子 agent 性质 | ★ 递归防护:task 自身 readOnly:true,不显式剔除就会被只读筛选选中 → 无限套娃 |
| `mcp.test.ts` | 协议实现 | ★ 错误二分:参数非法 = `result.isError`(模型可见可重试),未知工具 = JSON-RPC error(协议级)—— spec 的关键设计 |

写测试时的选择性:**每个文件都测「文档里吹过的那个点」**。文档说 edit 比 sed 可靠 → 测唯一性拒绝;文档说正则不是真安全 → 测绕过后的兜底;文档说压缩配对安全 → property 遍历所有切割参数。测试是文档主张的可执行证据,面试官抽查任何一句都有对应断言。

## 第四部分 · CI(.github/workflows/test.yml)

```yaml
on: push(main) / pull_request
steps: checkout → setup-bun → bun install --frozen-lockfile → bun run typecheck → bun test
```

- `--frozen-lockfile`:锁死依赖,CI 不悄悄升包
- 零 secrets:见第二部分③,这是测试离线设计的直接回报
- typecheck 在 test 之前:类型错误比测试失败更早、更便宜地暴露

## 第五部分 · 5 分钟走读路线(带面试官看 repo 的话术)

1. **README 架构图(30 秒)**:「三层 —— 壳只管交互,内核是可嵌套的 runAgent,工具层是个数组。两条虚线是灵魂:执行指向工具层,而工具层的 task 又递归指回内核 —— 子 agent 就是"包装成工具的整个 loop"。」
2. **src/agent.ts(90 秒)**:「runAgent 的参数表就是全部设计:client 可注入所以能离线测,approve/onText/signal 是回调所以内核不碰终端。loop 本体 = 流式取回复 → 权限门 → 执行 → 回填 → 压缩 → 循环,80 行。」
3. **src/task.ts(60 秒)**:「子 agent = 在工具的 execute 里再跑一遍 runAgent,用全新的 messages —— 上下文隔离不是什么机制,就是"新数组"。筛工具那行的 `!== "task"` 是递归防护。」
4. **tests/(60 秒)**:「测试测协议层不测模型:fake client 把模型回复写成脚本,loop 每条分支确定性触发。最得意的是 compact 那条 property 测试 —— 任意切割参数下配对不被劈开。」
5. **docs/(30 秒)**:「十篇按天笔记,每篇都是 what/why + 解剖真 Claude Code 对照。这个 repo 同时是作品和学习方法的证据。」

**兜底 Q&A —— 「这和真 Claude Code 差在哪?」**(主动说,别等问):单 provider 无降级(demo 里有 withFallback 演示)、权限是正则演示级(真 CC 有沙箱 + 更复杂的策略引擎)、无 IDE/LSP 集成、无 hooks/插件体系、压缩是单级摘要(真 CC 有 8 段结构化模板)。**能一口气说出差距清单,比吹实现了多少更能证明你看懂了真家伙。**

## 第六部分 · 面试金句

1. 「LLM 应用的测试分两层:**协议层逻辑层用单测**(确定/免费/上 CI),**输出质量用 eval**(非确定/花钱/看统计)。分界线是:这段代码的对错,依不依赖模型的自由发挥。」
2. 「runAgent 把 client 参数化,一个决定换来三次复用:子 agent 嵌套(D7)、批量 eval(D8)、离线测试(D10)。依赖注入在 agent 工程里不是教条,是**省钱按钮**。」
3. 「压缩最值得测的不是摘要质量,是一个不变量:**任意切割参数下 tool_use/tool_result 配对绝不被劈开** —— 劈开就是 API 400。」
4. 「权限测试里我特意测了绕过:变量拼接骗过正则后,它落进 ask 而不是 allow。**测自己的局限,比假装没有局限更工程。**」
5. 「CI 零 secrets 不是省事,是设计结果:测试全离线才可能零 secrets,零 secrets 徽章才可信。」
6. 「该 fake 的是贵且非确定的 LLM,不该 fake 的是免费且确定的本地世界 —— mock 掉文件系统,你测的就只是 mock 本身。」
