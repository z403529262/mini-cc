# D3 — 流式输出与可中断（mini-cc M3 × cc-haha 对照）

> 双线螺旋第三关：先自写最小版（`src/m3.ts` + `src/tools.ts` 异步化），再解剖 cc-haha 的流式主循环与中断传播，最后提炼面试话术。
> 配套：`docs/D1-深度问答笔记.md`、`docs/D2-工具系统与文件编辑对照.md`。

---

## 0. 这一关解决什么

D2 留下的**局限 1**：`tools.ts` 的 `execute` 全是**同步**的，`bash` 用 `execSync` 死占 event loop。一个 `while true; do :; done` 就能把整个 agent loop 冻住，Esc 根本插不进来。

D3 把整条链路改成**异步 + 可中断**，三件事：

| # | 做什么 | 关键 API | 解决的痛 |
|---|--------|----------|----------|
| ① 流式吐字 | `.create()` → `.stream()`，边生成边打印 | `stream.on("text", …)` | 不再"空白等很久 → 一下全出来" |
| ② 攒齐 content | 流式碎片重新拼成完整 message 塞回 loop | `await stream.finalMessage()` | 流式下 tool_use 的 `input_json_delta` 碎片要拼回完整 input |
| ③ 双点中断 | 一把 `AbortController` 同时管模型流 + 工具 | `{ signal }` 传给 stream 和 execute | Esc 既能掐模型生成，也能杀 bash 子进程 |

## 1. mini-cc 的改造落点

| 文件 | 改动 |
|------|------|
| `src/tools.ts` | `execute(input): string` → `execute(input, signal?): Promise<string>`（工具的正确形态：异步不阻塞、signal 可中断） |
| `src/tools.ts` 的 `bash` | `execSync` → `node:child_process` 的 `spawn({detached:true})` + `process.kill(-pid)` |
| `src/m3.ts` | `.create()` → `.stream()` + `finalMessage()` + Esc/Ctrl+C 监听 + 每轮 `new AbortController()` |
| `src/m2.ts` | 跟着加一个 `await`（execute 现在返回 Promise） |

骨架（`m3.ts`）：

```ts
let ac = new AbortController()
process.stdin.on("keypress", (_, key) => {
  if (key?.ctrl && key?.name === "c") process.exit(0)   // Ctrl+C 退出
  if (key?.name === "escape") ac.abort()                 // Esc 打断当前轮
})

while (true) {
  ac = new AbortController()                             // 每轮一把新锁（一次性）
  const stream = client.messages.stream({...}, { signal: ac.signal })  // ③ 给模型流
  stream.on("text", (d) => process.stdout.write(d))     // ① 边吐字
  let res
  try { res = await stream.finalMessage() }             // ② 攒齐完整 content
  catch (e) { if (ac.signal.aborted) break; throw e }   // 模型流被掐
  messages.push({ role: "assistant", content: res.content })
  if (res.stop_reason !== "tool_use") break
  for (const b of res.content) if (b.type === "tool_use") {
    const out = await tool.execute(b.input, ac.signal)   // ③ 同一把 signal 给工具
    results.push({ type: "tool_result", tool_use_id: b.id, content: out })
  }
  if (ac.signal.aborted) break                           // 工具阶段被掐
  messages.push({ role: "user", content: results })
}
```

## 2. 流式三件事

### 2.1 流式吐字
`client.messages.stream()` 返回 SDK 的 `MessageStream`（= `BetaMessageStream`），它把底层 SSE 事件流封装成易用接口：`stream.on("text", delta => …)` 拿到文本增量边吐边打。tool_use 的参数走的是 `input_json_delta` 事件、不触发 `"text"`，所以攒齐后再统一打印 `执行｜xxx`。

### 2.2 finalMessage() 攒齐 content
agent loop 需要的是**完整** `content` 数组（含 tool_use 的完整 input）才能塞回 `messages`。`await stream.finalMessage()` 把一地碎片（`text_delta` + `input_json_delta`）重新拼回一条完整 message——text 拼好、tool_use 的 partial JSON 也 parse 好。**这是流式与 agent loop 结合的关键**：流式只改"怎么收"，不改"塞什么回去"。

### 2.3 为什么流式下还要"攒"
模型的一次响应在线路上是被切成无数 SSE chunk 的：`message_start` → `content_block_start` →（`content_block_delta` × N）→ `content_block_stop` → `message_delta` → `message_stop`。tool_use 的参数 JSON 被切成 `input_json_delta` 的 `partial_json` 片段，必须按 block index 拼接、最后 parse 成对象。SDK 的 `finalMessage()` 替你做了这件事；不用它就得自己累积。

## 3. AbortController 机制（本关重点）

> **一句话本质：`AbortController` 自己什么都不停。** 它是一根**取消信号总线**；`ac.abort()` 不去关 socket、也不去杀进程——真正的取消动作，是模型流和 bash **各自**接在这根总线上、自己执行的。这叫**协作式取消（cooperative cancellation）**。

### 3.1 两个零件

| 零件 | 角色 | 关键成员 |
|------|------|----------|
| `AbortController` | **发射端**（一次性开关） | `.abort(reason?)`、`.signal` |
| `AbortSignal` | **接收端**（广播给所有监听者） | `.aborted`（布尔标志）、`.reason`、`addEventListener('abort')`、`throwIfAborted()` |

一个 controller 配一个 signal；signal 可被**任意多个**消费者同时监听——这就是"一把 signal 管两处"的物理基础。

### 3.2 abort() 实际只做三件事

```
ac.abort(reason)
  ├─ 1. signal.aborted = true      // 置标志位
  ├─ 2. signal.reason  = reason    // 记原因
  └─ 3. 向所有 addEventListener('abort') 的监听者派发一次 'abort' 事件
```
到此为止。没有任何 HTTP 请求被关、没有任何进程被杀。下面两条链才是"生效"的真相。

### 3.3 路径一：模型流（HTTP）——库帮我监听

```
ac.abort()
  → signal 派发 'abort'
  → SDK 把 signal 透传给底层 fetch(url, { signal })
  → fetch 原生支持 AbortSignal：监听到 'abort' → 关闭底层 TCP socket
  → 正在传输的 SSE 流立即断开
  → finalMessage() reject 一个 AbortError（cc-haha 里叫 APIUserAbortError）
  → 我的 catch 里 if (ac.signal.aborted) → break
```
我只做了一件事：**把 signal 递进去**。关 socket 是 `fetch` 自己干的（它原生 opt-in 了 AbortSignal）。

### 3.4 路径二：bash 子进程——我手动监听

```ts
const onAbort = () => {
  process.kill(-proc.pid!, "SIGKILL")  // 杀整个进程组
  resolve("[bash 已中断]")             // 立即收场，不等管道 EOF
}
signal?.addEventListener("abort", onAbort)
```
```
ac.abort() → signal 派发 'abort' → 我注册的 onAbort 触发 → kill(-pid) + resolve
```
fetch 那套"原生支持"在这里不存在，得我**自己** `addEventListener('abort')` 把取消动作接上去。

### 3.5 两条路径，本质是同一件事

| | 谁在监听 signal | 取消动作是谁实现的 |
|---|---|---|
| 模型流 | `fetch`（库内建） | 库：关 socket |
| bash | **我**（手写 `addEventListener`） | 我：`kill(-pid)` |

都是"消费方监听同一个 signal"，区别只在**库帮你监听**还是**你自己监听**。signal 永远只负责广播"该停了"，停不停、怎么停由监听者决定。

### 3.6 两个配套细节
- **为什么每轮 `new AbortController()`**：它是**一次性**的——abort 过的 signal 永久 `aborted`、不能 reset。每轮换新的，这轮的 Esc 才不污染下一轮。
- **发射端是谁不重要**：`m3.ts` 里是键盘 `keypress → ac.abort()`；`demo/abort-check.ts` 里是 `setTimeout(() => ac.abort())`。两条监听链对"谁拉的闸"无感——这正是能用定时器替代真人按 Esc 来自动化验证的原因。

### 3.7 Kotlin / Android 类比（秒懂版）

`AbortSignal` 就是 **JS 版的 `CancellationToken`**，机制和 Kotlin 协程取消一模一样：

| Web API | Kotlin / Android |
|---------|------------------|
| `new AbortController()` | `Job()` / `CoroutineScope` |
| `ac.abort()` | `job.cancel()` |
| `signal.aborted`（轮询） | `isActive` 检查 |
| `signal.addEventListener('abort')`（事件） | `invokeOnCancellation { }` |
| `fetch(url, { signal })` | OkHttp，取消时 `call.cancel()` |
| 手写 `addEventListener('abort', () => kill())` | `invokeOnCancellation { process.destroy() }` |

**协作式取消**完全相通：Kotlin 的 `cancel()` 也只是设标志 + 抛 `CancellationException`，挂起函数要在 `ensureActive()` / `isActive` / `suspendCancellableCoroutine` 这些**协作点**才真正停。`cancel()` 不抢占式 kill 线程——和 `abort()` 不魔法般停东西，是同一个哲学。

## 4. bash bug：杀父不杀孙（本关最值钱的坑）

**现象**：第一版 `bash` 用 `Bun.spawn` + `proc.kill()`，abort 后实测**傻等满 10s** 才返回（虽然最后确实返回了 `[bash 已中断]`）。

**根因**：`proc.kill()` 只给**直接子进程** `bash` 发 SIGTERM；但 `bash -c "sleep 10"` 里的孙子进程 `sleep` 继承了 stdout **管道的写端**、且没被杀 → 管道不 EOF → `new Response(proc.stdout).text()` 一直 await 到 sleep 自然结束。**杀父不杀孙，管道不关，await 不返回。**

**修法（两招叠加）**：
1. **杀整个进程组**：`spawn(..., { detached: true })` 让 bash 自成进程组（pgid === pid），再 `process.kill(-pid, "SIGKILL")`——负 pid = 杀整组，bash + 所有后代一锅端。
2. **abort 即 resolve**：不再苦等管道 EOF，`onAbort` 里直接 `resolve("[bash 已中断]")`。

**额外坑**：`node spawn` 其实**也**支持 `{ signal }` 参数，但它的实现只 SIGTERM 直接子进程、杀不到进程组——**就是这个 bug**。所以没用它的内建 signal，而是手动 `addEventListener('abort')` 接 `kill(-pid)`。

→ 面试问"你写 agent 踩过什么坑"的标准答案。

## 5. cc-haha 对照

**结论：mini-cc 的 M3 和 cc-haha 在骨架上完全同构，差距在 6 处生产级加固。**

### 5.1 骨架同构（我做对的方向）

| 维度 | mini-cc (`m3.ts`) | cc-haha |
|------|------|---------|
| 一把 signal 管全程 | `ac.signal` 同传 stream + execute | `abortController.signal` 同传 `callModel`（`query.ts:664`）+ 工具（`:1471`） |
| 两个独立中断点 | 模型流 catch / 工具后 check | `aborted_streaming`（`query.ts:1015`）/ `aborted_tools`（`:1485`） |
| signal 透传 SDK | `finalMessage()` 抛错被 catch | SDK 抛 `APIUserAbortError`，识别为"非错误"直接 re-throw（`claude.ts:879`） |
| 流式攒齐 content | `await stream.finalMessage()` | 手动累积 `contentBlocks` |

### 5.2 cc-haha 多做的 6 处（mini-cc 待补）

1. **裸流 vs 封装**：cc-haha 用 `.create({stream:true})` 自己消费裸 SSE，**故意避开** `BetaMessageStream`——因为后者对每个 `input_json_delta` 都 `partialParse()` 一次整段 partial JSON = **O(n²)**；它不需要"边流边看半截 tool input"，攒齐再 parse 一次 = O(n)。我用 `.stream()` 封装（mini 够用）。`claude.ts:1883-1889`
2. **中断后补齐 `tool_result`**：`yieldMissingToolResultBlocks(…, 'Interrupted by user')` 给半截 tool_use 补配对结果——否则下一轮 API 报"tool_use 缺 tool_result"。**这是 D2 配对铁律在中断场景的延伸**。我直接 `break` 丢弃整轮所以不需要。`query.ts:1019/1025`
3. **`abort(reason)` 语义**：`signal.reason === 'interrupt'` 区分"用户主动中断"vs 其他 abort，决定要不要插一条 interruption 消息。我的 `ac.abort()` 没带 reason。`query.ts:1046`
4. **流式失败降级 + 墓碑**：streaming → non-streaming fallback（`claude.ts:867`，也透传 `{ signal }`）；半截 assistant 消息（尤其 thinking block 签名失效）要 `tombstone` 删掉，否则污染下一轮。`query.ts:712`
5. **idle watchdog**：90s 无新 chunk 主动 abort——SDK 的 timeout 只覆盖初始 `fetch()`、不覆盖流式 body，连接静默掉线会无限挂死。`claude.ts:1933`
6. **边流边执行工具**：`StreamingToolExecutor`（`src/services/tools/StreamingToolExecutor.ts:40`，在 `query.ts:562/735` 实例化）——流没结束工具就开跑，省延迟；我是"流完 → 执行"串行。

> 主流式函数 = `queryModelWithStreaming`（`query/deps.ts:35` 注入为 `callModel`），是个 async generator，`query.ts:659` 用 `for await` 消费。

## 7. 面试金句总集

- **"流式 agent loop 最反直觉的坑不是怎么吐字，是怎么收尾。"** 中断/降级时，半截的 `tool_use` 必须补 `tool_result`、半截的 thinking 必须删——否则下一轮请求直接 422。
- **"`AbortController` 是取消的协议层，不是执行层。"** 它只广播"该停了"，真正的取消散落在每个消费方——和 D2"并行是协议层不是执行层"同构。
- **"传了 signal ≠ 能取消；被取消方必须主动 opt-in 监听它才行。"** fetch 能取消是因为它监听了；spawn 杀不干净是因为它监听得不够。
- **"一次 `abort()`，两个监听者（HTTP / 子进程）同时收到同一个事件，各杀各的。"**
- **"杀子进程要杀整个进程组。"** 杀父不杀孙 → 孙子攥着管道写端 → 管道不 EOF → await 永不返回。
- **"SDK 的 timeout 不覆盖流式 body。"** 所以要自己加 idle watchdog，这是连接悄悄掉线时不挂死的唯一防线。
- **"流式我用封装、生产用裸流。"** `.stream()` 的 `BetaMessageStream` 对每个 `input_json_delta` 都 `partialParse()` = O(n²)；不需要半截 tool input 就降到 `.create({stream:true})` 自己攒。

## 8. mini-cc 待补清单（对照 §5.2，按性价比）

- [ ] **中断后补齐 tool_result**（性价比最高，直接对应"配对铁律"）：abort 时给未完成的 tool_use 补一条 `is_error` 的 `tool_result`，让对话能继续而非整轮丢弃。
- [ ] **abort(reason)**：`ac.abort('interrupt')`，按 reason 区分中断来源。
- [ ] **idle watchdog**：流式期间 setTimeout 看门狗，超时无 chunk 自动 abort。
- [ ] **边流边执行工具**（最难、最有高级感）：流式到 `content_block_stop` 且该 block 是 tool_use 时立即开跑，不等整条流结束。
- [ ] 裸流消费（`.create({stream:true})` 自己处理 SSE）/ 流式降级（低优先，mini 够用）。
