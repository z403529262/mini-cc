# D4 — 上下文压缩与 Prompt Caching（mini-cc M4 × cc-haha 对照）

> 本文遵循 `docs/讲解约定.md`：每个知识点按 **是什么 → 为什么（配"不这样会怎样"）→ 怎么办（从简）** 组织，What/Why 是主体，How 点到为止。
> 配套：D1 深度问答 / D2 工具系统 / D3 流式与中断。

---

## 0. 这一关是什么、为什么

**是什么**：D4 解决"对话越滚越长"带来的两个问题，用两件武器——**Prompt Caching（省钱）** + **上下文压缩（省窗口）**。

**为什么**：agent loop 是**无状态全量重发**的（D1 学过）——每轮把**整个历史**重新发一遍。第 k 轮发送的 token 量 ∝ k，N 轮累计 ≈ 1+2+…+N = **O(N²)**。这带来两个必然恶果：
- **贵**：重复发送是平方级浪费，前面那段历史每轮都在重新计费、重新 prefill。
- **终将爆**：历史无限增长,迟早撞上 context window 上限(如 200K)。

**不这样会怎样(反例)**：一个 50 轮的 agent 任务,不做任何优化——光重复 prefill 同一段历史就烧掉天文数字 token;且跑到某轮历史超 200K,直接 `400 prompt too long`,任务**崩在半路**。

两件武器各打一个恶果:**caching 砍掉"重复计费"(把重复段降到 0.1× 价)**,**压缩砍掉"无限增长"(真正缩短历史)**。它们之间有张力(见 §12),这正是 D4 的灵魂。

# 第一部分:Prompt Caching(治"贵")

## 1. caching 是什么、为什么

**是什么**:把"重复发送的前缀"的计算结果(KV cache,即注意力的键值)缓存在服务端,下次命中就**不重算**、直接复用。

**为什么**:agent 每轮重发整个历史,但历史**前面那段**(tools / system / 老对话)每轮**字节完全一样**——对它重复 prefill 是纯浪费。caching 让"每轮都一样的部分"只算一次。

**不这样会怎样**:10 轮对话,把同一个 10K 的 `system+tools` 完整 prefill 了 **10 遍**,其中 9 遍是花钱重算一模一样的东西。

---

## 2. cache_control 是什么、为什么

**是什么**:请求 JSON 里挂在某个 content block 上的一个**"缓存断点"标记**——`{"type":"ephemeral"}`。它**不是缓存本身**:缓存实体是服务端那段 KV,以"前缀字节 hash"为 key,TTL 到期自动驱逐。

**为什么**:得有个东西告诉服务端"**缓存从开头到这个 block 为止的前缀**"。cache_control 就是这枚图钉。

**不这样会怎样 / 易混点**:把它当成"缓存数据"就错了。它是**图钉,不是仓库**——响应里根本不回 cache_control,只回 `usage.cache_read_input_tokens` 等账单字段。类比 Spring 的 `@Cacheable` 注解:注解是标记,缓存表在别处,key 是入参(这里换成前缀字节)。

**怎么办**:`{type:"ephemeral", ttl?:"1h"}` 挂在 block 上;最多 4 个断点/请求。

---

## 3. 前缀匹配 —— caching 的唯一不变量

**是什么**:**缓存是前缀匹配。前缀里任何一个字节变了,它之后的全部失效。** cache key = 渲染后 prompt 到断点的精确字节 hash。渲染顺序固定:`tools → system → messages`。

**为什么**:这一条推出**全部**最佳实践——稳定的内容放前面,易变的放后面。get 对了,大部分缓存自动生效;get 错了,打再多 cache_control 也没用。

**不这样会怎样(经典反例)**:在 system 头部插一个 `Date.now()` 时间戳 → 每次请求前缀第一段就变 → 整段缓存每次 miss,`cache_control` 形同虚设。

---

## 4. 三层缓存(tools / system / messages)是什么

**是什么**:它们**不是三块独立缓存**,而是同一条字节序列上**三个一层套一层的前缀**:

```
[   tools 段   ][  system 段  ][   messages 段   ]
 tools 缓存 = [tools]
 system 缓存 = [tools+system]        ← 含 tools
 messages 缓存 = [tools+system+messages] ← 含前两者
```

**为什么(解释"改一处只炸它及之后")**:前缀匹配从头逐字节比。改 system:`[tools]` 段字节没动 → "到 tools 末尾"的 hash 不变 → ✅ 仍命中;但 `[tools+system]` 因 system 变 → ❌;messages 在更后 → ❌。**你改动的位置之后全炸,之前安然无恙。**

| 改动 | tools | system | messages |
|------|:--:|:--:|:--:|
| 改 tool 定义 / 切 model | ❌ | ❌ | ❌ |
| 改 system | ✅ | ❌ | ❌ |
| 改 message | ✅ | ✅ | ❌ |

**类比(Docker 镜像分层)**:`tools`=底层(base image)、`system`=中层、`messages`=顶层(你的代码)。改底层→上面全重建;改顶层→底层缓存照复用。Docker 教你"不变的写底层、常变的写顶层"——prompt 同理。

---

## 5. 静默失效 —— 最坑的地方

**是什么**:缓存失效但**不报错**,`cache_read` 默默归零,你还以为它在生效。

**为什么坑**:报错你会去修;静默失效你根本不知道在漏钱。根因都是同一个——**不该变的字节溜进了 prefix 还每次都在变**。

**清单(每条都是"为什么炸")**:

| 坑 | 为什么炸 |
|----|----------|
| `Date.now()`/时间戳进 system | 每请求前缀都变 |
| `JSON.stringify` 不排序 / 迭代 Set | 逻辑相同但**字节顺序**不定 |
| session/user ID 进 system | 每用户不同前缀、跨用户不共享 |
| 条件 system 段(`if flag`) | 每种 flag 组合是不同前缀 |
| TTL 翻转(5min↔1h) | `cache_control` 字节变 → 整段失效 |

**怎么办(从简)**:进 prefix 的东西必须**确定且稳定**;易变的(时间/状态/user id)一律挪到 messages 末端;序列化排序键;TTL 决策会话级锁死。

---

## 6. 经济学 —— 为什么值得做

**是什么(前提:三种 token 三种价)**:开缓存后 input token 按命运分三类计价(以普通价为 1× 基准):

| 命运 | 价格 |
|------|------|
| fresh(没缓存) | 1× |
| **cache write**(写入) | **贵**:1.25×(5min)/ 2×(1h) |
| **cache read**(命中) | **便宜**:0.1× |

**为什么这样定价**:缓存是"**先花钱存(write 贵),后省钱读(read 便宜 90%)**"的买卖。

**5min vs 1h 是什么**:TTL = 缓存存活时间。5min(默认,写便宜)适合**请求密集**(每次命中刷新、缓存一直活);1h(写贵一倍)适合**请求稀疏/突发**(扛得住长间隔不掉)。

**回本要几次(为什么"单次用缓存反而亏")**:第一次必然 write(付溢价),要复用够多次,read 省的才盖过 write 溢价:
- 5min:1 次 = 1.25(亏);**2 次** = 1.25+0.1 = 1.35 < 2 ✅ 回本
- 1h:2 次 = 2.1 > 2(还亏);**3 次** = 2+0.2 = 2.2 < 3 ✅ 回本

**算账例子**:`tools+system` = 10000 token,连续 5 轮:
- 不缓存:5 × 10000 = 50000
- 5min 缓存:write 12500 + 4×read 4000 = **16500** → **省 67%**

**连实测**:mini-cc 实测命中率 **83.8%** = 发出去的 token 里 83.8% 走 0.1× 价,这部分只花 1/10 的钱。

> 倍数是 Anthropic 官方定价;glm 端点数字可能不同,机制一样。

---

## 7. mini-cc 怎么做(How,从简)

`m4.ts` 三步(详见源码):
1. **SYSTEM block 打断点**(注释①):system 升级成 `TextBlockParam[]`、挂 `cache_control` → 缓存 `tools+system`(最稳定段);
2. **`withRollingCache`**(注释②):每轮给"最后一条 message 的最后一个 block"打滚动断点,缓存"到此刻为止的全部历史";
3. **副本防污染**:只在发送的副本上打,真实 `messages` 永远干净——否则 `cache_control` 逐轮残留 → 超 4 断点 → 400(路径复制,等同 Kotlin `data class.copy()`,别 mutate 共享状态)。

实测命中率 83.8%。

## 8. cc-haha 的 6 处生产级增强(What/Why 一句,How 略)

朴素 caching(打个断点)只是起点;生产级是一整套**稳定性 + 可观测性工程**:

1. **1h TTL + latch 锁定**(`should1hCacheTTL`):把影响 `cache_control` 字节的变量(资格/allowlist)会话级**锁死**,防中途翻转炸缓存(~20K token 损失)。教科书级静默失效——"用量超额"竟能顺着 `超额→TTL翻转→字节变→失效` 暗链炸掉缓存。
2. **跳过 thinking block**:thinking 带签名、不能挂 cache_control,打断点要避开。
3. **分模型开关**(`DISABLE_PROMPT_CACHING_HAIKU/SONNET/OPUS`):不同模型缓存经济性/最小阈值不同(Opus 最小 4096 token、Sonnet 1024),能单独关。
4. **scope: global/org**:把"对所有用户字节相同的大段 system"提升成**跨会话/组织共享**缓存(需 system 干净——呼应 §5 坑3)。⚠️ global **不是**"三层前缀"之一,而是**段内容寻址**的特殊层,且 tools 因人而异(MCP)时会被关掉——准确机制见**进阶篇 B**。
5. **cache_edits 增量编辑**:见 §9。
6. **promptCacheBreakDetection**:记录上次请求全套 hash,自动 diff 出"谁破了前缀"(连 `perToolHashes` 定位到具体哪个 tool 描述变了),把 §5 那句"cache_read 不涨就 diff 字节"**工程化**。

## 9. cache_edits 增量编辑 + 各家对比(前沿)

**是什么**:从"已缓存的前缀"里**外科手术式删掉指定几块**,不重建整段缓存。它把缓存前缀从"不可变整块"变成"可增量删除的结构"。

**为什么需要**:普通 caching 只有"命中/整段失效"两态。压缩老历史会改前缀 → 整段缓存炸(见 §12)。**反例**:10 轮对话第 3 轮有个 2 万 token 的大 tool_result 想删——普通压缩会连累第 4–10 轮缓存全失效(为删 1 块炸 7 轮);cache_edits 只发 `delete t3`、其余原地保留。**它让"压缩省窗口"和"保住缓存"不再二选一。**

**怎么办(从简)**:给前缀内 tool_result 打 `cache_reference`(=tool_use_id)→ 插 `{type:"delete", cache_reference}` → 服务端摘块 → pin 持续重发 + dedup。

**它现在只有 Anthropic 有吗**(2026-06 查证):
- **是,且还是非公开 beta**(cc-haha 里带 `cachedMCEnabled` gated header;公开 prompt-caching 文档完全没提它)。
- **关键区分**:"删整个缓存对象/TTL 驱逐"各家都有;"**前缀内增量摘块**"只有 Anthropic。
- **为什么别家没有(范式决定)**:增量编辑要求开发者能**引用缓存里的某个块**,这需要"**显式 + block 级**"控制。OpenAI/DeepSeek 是**全自动隐式**(连断点都标不了,没有挂载点);Gemini 是**对象级**(粗粒度,只能删整个 context cache 对象)。只有 Anthropic 的显式 block 断点范式能自然长出 block 级增量编辑。
- **趋势**:行业方向是"自动化降门槛"(与精细手动控制相反),短期大概率仍 Anthropic 独有。

| 厂商 | 范式 | 粒度 | 前缀内增量删块 |
|------|------|------|:--:|
| Anthropic | 显式 cache_control | block | ✅ beta |
| OpenAI / DeepSeek | 自动隐式 | 无控制 | ❌ |
| Gemini | 显式 cache 对象 | 对象级 | ❌ |

# 第二部分:上下文压缩(治"爆")

## 10. 压缩是什么、为什么

**是什么**:对话逼近 context window 上限时,把**老历史用一次 LLM 调用总结成一段摘要**、替换掉,从而缩短 messages。

**为什么**:caching 只治"贵"、**不治"爆"**——历史再便宜也还占着 context window,迟早撞上限。压缩是**唯一能真正缩短历史**的手段。

**不这样会怎样**:只做 caching 不压缩,第 X 轮历史超 200K → `400 prompt too long`,缓存再省钱也**发不出去**,任务崩在半路。

**怎么办(从简)**:`compact.ts` —— `estimateTokens` 估规模(字符/3),超阈值就把 `messages[0..cut]` 总结成一条 `user` 摘要 + 保留最近几条。

---

## 11. 配对安全 —— 压缩的核心难点

**是什么**:压缩切割历史时,**不能把 `tool_use` 和它的 `tool_result` 劈开**。

**为什么**:D2 的配对铁律——每个 `tool_use` 必须有配对的 `tool_result`。**反例**:切割点落在某个 `tool_result` 上、把它的 `tool_use` 划进被压缩段 → 保留段出现**孤儿 tool_result** → 下一轮 `400`。

**怎么办(从简)**:`findCutPoint` 让**保留段从 assistant 开头**——这样压缩段自然以"配对完整的 tool_result"收尾,两段各自自洽。实测 7 条→5 条,全程不报错。

---

## 12. 压缩 vs 缓存的张力 —— D4 的灵魂

**是什么**:压缩**改写了历史前缀**,而 caching 靠前缀稳定 → **压缩会炸掉一段缓存**。

**为什么(实测铁证)**:mini-cc 第 4 轮压缩后,新 input 从几十**暴涨到 305**(messages 段缓存炸了),但 `tools+system`(在 messages 前面、没动)仍命中 **704**——精确对应 §4 "改 message:前两层 ✅✅、末层 ❌"。

**这个张力怎么破**:常规压缩只能认栽(炸 messages 段缓存);Anthropic 的 **cache_edits(§9)** 是高级解——外科手术删个别块、不炸整段。**"压缩省窗口"与"保 cache 命中"的对赌,cache_edits 是目前唯一让你两头都要的工具。**

# 第三部分:进阶答疑(封板前深挖)

> 封板前一长串深问的沉淀,专治几个"看起来矛盾"的点。仍 What/Why 优先。

## A. cache_control 再澄清:图钉 + "删不删"是伪命题

**它不进 token 流**:cache_control 是请求的**元数据标记**,在解析阶段处理/丢弃,**不会被 tokenize 进模型输入**。模型和缓存看到的只有 A/B/C 的文本。

**每轮要删上一轮的 cache_control 吗**——伪命题。`withRollingCache` 用副本:真实 `messages` 永远不带 cache_control,每轮临时克隆、在副本最后一条打标记、发完即弃。**要删,恰恰说明你原地改了历史(污染)**;原地改的下场=cache_control 逐轮残留→超 4 断点→400。

## B. global 的真相 —— 修正"三层前缀"的简化

**修正**:§4/§8 把 global 当"三层前缀之一"是简化过头。**global 不是线性前缀缓存,是段内容寻址的特殊层。**

**证据**:渲染顺序里 attribution header(含指纹、因人而异、scope=null)排在 global static 段**前面**。若 global 靠线性前缀,前缀早分叉、永不命中;它还能 global → 说明 key 只看"段自身内容 + 命名空间",不看前面。

**两层模型**:命名层(段内容当 key,同前缀下跨请求复用)+ 物理层(KV 数值仍依赖前缀,causal attention)。跨用户复用要求前缀(尤其 tools)也全局一致。

**为什么 MCP 一来就关 global**(`needsToolBasedCacheMarker`→`skipGlobalCacheForSystemPrompt`):KV 是一条链,**system 段的 KV 物理上长在 tools 上面**(每 token 的 K/V 依赖前面所有 token);MCP 是 per-user → tools 因人而异 → system 段 KV 也因人而异(哪怕 system 文本没改)→"全网共享一份 system KV"成空头支票 → 关掉退 org。**skip global ≠ 不缓存**:退会话内/org,同一用户连续请求(tools 不变)照常命中(83.8% 就是会话内这层)。

**ToolSearch defer 救 global**:`!willDefer(t)` —— MCP 工具被 `defer_loading`(配合 ToolSearch 延迟加载)就不实际渲染进 tools 段 → 不触发 skip。**defer 既省 token 又保住 global。**

## C. system 内部不是一块 —— section + boundary + 三 scope

**是什么**:tools/system/messages 是 API 粗段;**system 内部**拼成十几个 section,中间插 `SYSTEM_PROMPT_DYNAMIC_BOUNDARY`,再压成 ≤4 个缓存 block、每块一个 scope。

- **静态段**(boundary 前,可 global):CC 核心 ~7 段(Intro/核心/Actions/ToolUsage/Tone/Efficiency);
- **BOUNDARY**;
- **动态段**(boundary 后,scope=null 不缓存):memory(你的 CLAUDE.md) / env_info(cwd/git/**date**) / skill 清单(session_guidance) / output_style / mcp_instructions。

**boundary 的 why(损失隔离)**:把"全网一样的 CC 内核"和"你的 CLAUDE.md/环境"隔开——你改 CLAUDE.md 只炸动态段,**不连累全网共享的内核缓存**。

**对号入座**:CC 核心=静态段(global);用户自定义=memory(动态 null);组织=org;skills=SkillTool 入口+清单在动态段;插件=工具进 tools/指令进 section;soul=`/buddy` 桌宠彩蛋(不在主 prompt),你说的"人格"其实是 output_style + Tone。**date 哪来的**:env_info 注入(模型有知识截止、不知今天几号),放动态段不缓存(天天变,缓存会天天炸)。

## D. tools 段到底装什么

**三类来源**:① 内置工具(~50个,固定/全局一致):FileRead/Write/Edit/Bash/Glob/Grep/Agent/Task*/Web*/Plan*…;② **MCP 工具**(`isMcp`,per-user,每个独立顶层 tool,`mcp__server__tool`)——**就是它破 global**;③ 插件工具(`source=plugin`)。

**skill 不是独立 tool**:tools 段里只有一个 `SkillTool` 入口;几百个 skill 的清单在 **system 动态段(session_guidance)**,不在 tools 段。

**两种"动态"粒度不同**:MCP 增删=tools 段 **tool 数量/结构**变(破 global);skill 增删=**system 动态段**变(本就不缓存);子 agent 增删=**AgentTool 单个 tool 的 description** 变(`perToolHashes` 抓的就是这个)。

## E. 跨 provider:cache_control 不是标准

**不是事实标准**:Anthropic 私有字段;聚合层(OpenRouter/LiteLLM)借用当准标准;OpenAI/DeepSeek 自动隐式不认、Gemini 用 cache 对象。

**关键差异**:**Anthropic opt-in(不加 cache_control 就不缓存)** vs **OpenAI/DeepSeek 默认开(不加也自动)**。

**双向验证(因 cache_control 不进 token 流)**:
- **DeepSeek 跑**(不认):cache_control 被忽略、对其自动缓存**零影响**(按 token 内容前缀匹配,A B 照命中);它 KV 落硬盘、64-token chunk、默认开。
- **Codex 打 Anthropic**(不发):全链路无 cache_control → Anthropic 默认不缓存 → 命中不了(除非转换层替你注入)。

**"为通用不加 cache_control"?反了**:它对不认的家无害、对 Anthropic 是缓存唯一开关——不加只在 Anthropic 上白丢缓存。**通用正解=加 + 兼容层/分支兜底;换端点必验 `cache_read` 真涨。**

## F. break detection 怎么做到自动归因

**两阶段**:pre-call `recordPromptState`(记录全套 hash + 算"什么变了",不报警)/ post-call(按 `cache_read` 是否意外下跌触发)。

**多维 hash**(定位"谁破的"):systemHash/toolsHash + **cacheControlHash**(抓 TTL/scope 翻转)+ **perToolHashes**(定位具体哪个 tool 变,77% 的 tool break 是 AgentTool 嵌动态列表)。

**智能去噪**(不误报):<2000 token 不报 / 排除 TTL 过期 / `notifyCompaction`+`notifyCacheDeletion`(主动压缩删除不算 bug)/ 排除 haiku / agentId 隔离防并发误报。

**自动到哪、人工到哪**:发现+归因+去噪+上报**全自动**;改代码**人工**。闭环=自动发现 → 人工 latch 修一次 → 自动回归验证(那些"应该不再破了,追踪验证"的字段)。

## 进阶金句

- **"cache_control 是元数据、不进 token 流"**:所以 DeepSeek 忽略它=无影响,Anthropic 没它=不缓存。
- **"global 不靠前缀、靠段寻址;但 system 的 KV 长在 tools 上,MCP 让 tools 因人而异,全网共享就成空头支票"** → skip global 退 org。
- **"Anthropic opt-in、别家默认开"**:同一个"缺 cache_control",后果相反。
- **"skill 不是 tool,是 SkillTool 一个入口 + 清单塞进 system 动态段"**:所以 MCP 增删破 global、skill 增删只动不缓存的动态段。
- **"失效归因要自动侦查、人工整改"**:能精确到"哪个 tool 变了",但改不改是人的判断。

## 13. 面试金句总集

- **caching**:"缓存是前缀匹配——稳定的放前面、易变的放后面,这一条推出全部最佳实践。"
- **cache_control**:"它是图钉不是仓库——标记断点,缓存实体(KV)在服务端、按前缀 hash 索引、TTL 自管。"
- **三层**:"tools/system/messages 不是三块缓存,是一条线上三个嵌套前缀;改动只炸它及之后。"(Docker 分层)
- **静默失效**:"最危险的缓存 bug 不报错——`cache_read` 恒为 0 就是有静默失效。"
- **经济学**:"缓存是先贵写后便宜读;单次用缓存反而亏,要复用够次数才回本(5min 两次/1h 三次)。"
- **压缩**:"caching 治贵、压缩治爆;压缩最反直觉的坑是配对——切割点劈开 tool_use/tool_result 就 400。"
- **张力**:"压缩省窗口的代价是炸缓存;cache_edits 是让两头都要的唯一解,但它还是 Anthropic 独有 beta。"
