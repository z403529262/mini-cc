# D11 · 十道高频面试题答题骨架(+压轴题+四道通识)

> 用法:每题三段式 —— **30 秒版**(电梯答案,先给结论)→ **展开版**(2 分钟,带 mini-cc 实证与真 CC 对照)→ **追问防守**(面试官大概率追的问题和接法)。
> 总原则:**每个论点都落到自己写过的代码或实测过的数字上**。"我实现过,数据是 X" 比任何背诵都可信。
> 深挖细节按题回查对应 D 系列笔记 —— 这份骨架是检索钩子,不是全文。

## Q1 描述 agent 的工作循环 / ReAct

**30 秒版**:agent loop 本质是一个以 `stop_reason` 为状态的 while 循环:把 messages + tools 发给模型,模型回 `tool_use` 就执行工具、把结果作为 `tool_result` 回填再循环,回 `end_turn` 就收工。ReAct 的"思考→行动→观察"在 API 层就是 assistant(text + tool_use)→ tool_result 的一来一回。

**展开版**:
- 我的 M1 用 20 行写完这个 loop,实测一句「列出最大 3 个文件并推测项目用途」自主循环 11 轮完成。
- 两条铁律:① assistant 的 content 必须**原样** push 回历史(模型要看到自己说过什么);② 每个 `tool_use` 必须配一个同 id 的 `tool_result`,劈开配对 = API 400 —— 这条贯穿到后面压缩的切割设计。
- API 是无状态的:每轮全量重发历史,"记忆"完全靠 messages 数组 —— 这是后面 caching / 压缩两个问题的根源。
- 真 CC 的 query.ts 主循环与此同构,多的是流式、权限门、压缩、子 agent 这些"外挂",内核不变。

**追问防守**:
- *"tool_result 为什么放 user role?"* —— 协议如此:模型的输出 = assistant,环境的反馈 = user。工具结果是"世界对模型说话"。
- *"并行工具调用怎么处理?"* —— 一条 assistant 可含多个 tool_use,结果全放**同一条** user 消息回填(D2 实测一轮 3 个 read_file)。
- *"loop 怎么防失控?"* —— 三层:maxTurns 熔断、执行前权限门、AbortSignal 中断。三个我都有测试覆盖。

## Q2 tool calling 底层怎么 work?schema 怎么设计?

**30 秒版**:工具 = 两部分 —— 给模型看的**说明书**(name / description / JSON Schema)+ 给代码用的**执行体**。模型只见说明书;所谓"调用"只是模型输出一个结构化 `tool_use` block,**执行永远发生在我方代码里**。schema 设计的核心是把 description 当 prompt 写:何时用、参数含义、约束。

**展开版**:
- 我的 `Tool` 接口就是这两部分合一个对象。最好的例子是 `edit_file`:description 写明「old_string 必须在文件中唯一出现,否则报错」—— 这句话直接塑造模型行为,它会自带上下文来保证唯一。核心算法 `split(old).length - 1` 与真 CC 完全一致。
- **错误也是接口的一部分**:工具失败不抛异常,回填 `[xx 失败] 原因`(必要时 `is_error`),模型看到原因会自愈换方案 —— 我的 loop 里未知工具、权限拒绝、执行失败全走这条路,有测试覆盖。
- 结构化输出与 tool calling 是**同一机制**:`tool_choice: {type:"tool", name}` 强制模型本轮必须按 schema 出参。我实测对比过:裸 prompt 求 JSON 会被裹 markdown 围栏导致 parse 失败,tool 强制直接拿到校验过的对象。

**追问防守**:
- *"schema 写不好会怎样?"* —— 模型选错工具、传错参。description 是隐形 prompt,真 CC 把每个工具的 description 独立成 prompt.ts 文件维护,当 prompt 工程做。
- *"工具参数会被注入怎么办?"* —— 执行侧防:我的 grep 用 `execFileSync` 逐参传入不走 shell,query 再恶意也只是个搜索串。
- *"模型怎么知道该不该调工具?"* —— 全靠说明书 + system 引导;真 CC 的 GrepTool prompt 直接写 "ALWAYS use Grep, NEVER bash grep"。

## Q3 context window 满了怎么办?(高频)

**30 秒版**:四板斧,按序:**压缩**(把老历史 LLM 摘要成一条)、**截断**(超大工具结果剪掉)、**隔离**(读 100 个文件的脏活派子 agent,只回结论)、**检索**(海量资料不进上下文,按需取相关片段)。再加 caching 解决"还没满但每轮全量重发太贵"。

**展开版**:
- 我的 compact.ts:估 token 超阈值就把 `[0..cut)` 摘要成一条 user 消息。**难点不在摘要在切割**:保留段首条必须是 assistant,否则孤儿 tool_result = 400。我为此写了条 property 测试 —— 任意切割参数下配对不被劈开。
- 真 CC 的深度:触发线是**窗口减 13K 绝对值**(不是百分比),warning/auto/blocking 三道线;摘要用固定 9 段模板,取舍逻辑 = **能重新生成的(模型输出)压,不能重建的(用户原话、接续点、关键代码)顶着压缩本能保真**;还有 full/micro/snip 多级压缩。
- 压缩与缓存有张力:压缩改了前缀会炸 messages 段缓存(我实测压缩后那轮新 input 暴涨、tools+system 段仍命中)。
- lost in the middle:长上下文中段信息易被忽略,所以"窗口大了就全塞"不成立(接 Q3½ 的 RAG 话题)。

**追问防守**:
- *"什么时候该 RAG 而不是压缩?"* —— 资料**本来就不该进**上下文时:海量文档、低迭代预算。压缩管"对话历史",RAG 管"外部知识",不是竞品。
- *"摘要丢了关键信息怎么办?"* —— 真 CC 的做法:摘要后**重注入**压缩前读过的文件/计划;9 段里 user messages 逐条列全。
- *"为什么不等满了再压?"* —— blocking 硬限前要留摘要本身的生成空间,所以提前 13K。

## Q4 prompt caching 是什么,怎么省钱提速?

**30 秒版**:`cache_control` 是打在 content block 上的**元数据图钉,不进 token 流**;唯一不变量是**前缀字节一致**才命中。三种价:fresh 1×、写缓存 1.25×、读缓存 **0.1×** —— 5 分钟内同前缀 2 次就回本。设计原则:不变的放前面,变的放后面。

**展开版**:
- 我的实现两个断点:SYSTEM 段固定断点 + `withRollingCache` 给最后一条消息打滚动断点。坑:必须在**发送副本**上打,不能污染原 messages —— 否则 cache_control 逐轮残留,超 4 个断点 = 400。实测智谱端点命中率 83.8%。
- 深一层,缓存是嵌套前缀(类比 Docker 分层):tools → system → messages,改上游炸下游。真 CC 的 global 缓存层曾被 MCP 破掉 —— MCP 工具 per-user,tools 段因人而异,system 的 KV 物理上长在 tools 上(causal attention),global 只好关掉;ToolSearch 的 defer_loading 才救回来。
- 真 CC 还有 break detection:按 cache_read 突跌自动侦测缓存破裂,多维 hash 定位到具体哪个 tool 变了。

**追问防守**:
- *"为什么 Anthropic 要显式打点,OpenAI 默认开?"* —— 设计哲学差异;跨 provider 时 cache_control 不是事实标准(DeepSeek 忽略它无害),换端点必须实测 cache_read 验证。
- *"压缩之后缓存全失效怎么办?"* —— 事实如此(前缀变了),cache_edits(beta)是唯一两全解;工程上接受"压缩那轮贵一次"。
- *"1h TTL 什么时候用?"* —— 写 2× 成本,3 次以上复用才划算,适合低频长会话。

## Q5 多 agent 架构怎么设计?何时拆 sub-agent?

**30 秒版**:核心动机**不是"多个脑子",是 context 隔离** —— 子 agent 干"读 100 个文件"的脏活,中间过程留在它自己的上下文里,只把结论带回主线。拆的判断三条:子任务独立自包含、产出可压缩成结论、需要并行。

**展开版**:
- 我的实现说破魔法:子 agent = 在工具的 execute 里**再跑一遍 runAgent,用全新的 messages 数组** —— 隔离不是什么机制,就是"新数组"。跑完即弃,只 return 最终文本。
- 隔离铁证(实测):子 agent input 3268 token 含文件全文,主 agent 全程 input 2700 **不含**。
- 三个安全设计:① 子工具集只读 → 全自动跑不卡审批(子 agent 没有终端可弹 y/n);② 递归防护 = 筛工具时显式剔除 task 自己(它 readOnly:true 会被只读条件误选 → 无限套娃);③ 继承主 signal,主中断子也停。
- 真 CC 的进阶:Explore 子 agent 用 **haiku** 压成本 + omitClaudeMd 精简上下文;还有反向的 fork 模式 —— 继承父上下文,消息**字节对齐**为的是命中父的缓存。

**追问防守**:
- *"agent 之间怎么通信?"* —— 我的是最简单向:prompt 下发、结论回收。真 CC 有 teammate 模式(SendMessage 互发),但默认仍是单向 —— 通信越多,隔离红利越少。
- *"为什么不默认多 agent?"* —— 贵(各自全量上下文)、慢(串行等待)、结论有损耗。单 loop 够用就单,拆是为了隔离不是为了酷。
- *"orchestrator 怎么防子 agent 跑偏?"* —— prompt 自包含(子看不到主对话)+ maxTurns + 只读工具集,三重限界。

## Q6 MCP 是什么,解决什么,和 function calling 的区别?(必问)

**30 秒版**:function calling 是**模型和工具对话的语法**;MCP 是**工具分发的物流** —— 把工具打包成独立 server 进程,任何 client 走标准握手就能发现并调用,解耦复用,USB-C 类比。两者不是竞品:MCP 工具最终还是以 function calling 的形式喂给模型。

**展开版**:
- 我手写过 client(~130 行,不用 SDK):协议三件套 = **stdio 换行分帧**(一行一条 JSON,日志必须走 stderr 不能污染 stdout)、**三步握手**(initialize → initialized 通知 → tools/list)、**tools/call**。请求靠 id 配对,pending Map + 超时兜底。
- 关键设计:`toTools()` 把 MCP 工具**包装成本地 Tool 形状**,塞进工具注册表 —— loop 和权限门零改动。这验证了注册表抽象的正交性:后来子 agent 也是同样方式接入的。
- **错误二分**(spec 的关键设计):参数非法 = `result.isError`(工具执行错误,回填给模型让它换参重试);未知工具 = JSON-RPC error(协议级,reject)。我的测试两条路都覆盖。
- 安全:server 自称 `readOnlyHint:true` 我也不信(spec 明说 annotations untrusted),一律过 ask 权限门。

**追问防守**:
- *"MCP 的安全风险?"* —— 第三方 server 不可信:工具描述可藏注入、行为可越权。防线 = 权限门管行为 + 不信 annotations + 用户显式接入才算数。
- *"除了 stdio 还有什么传输?"* —— 真 CC 支持 5 种:stdio / SSE / StreamableHTTP / WebSocket / InProcess。
- *"你的实现和生产版差在哪?"* —— 主动说:没做工具名非法字符 normalize(Anthropic API 对 tool name 有字符限制)、没做重连和批量 reject、单 server —— 知道简化在哪。

## Q7 怎么保证 agent 安全?

**30 秒版**:分层防御:① 执行前**权限门**,判定对象是"工具+参数"而非工具名(同一个 bash,`ls` 和 `rm -rf` 天差地别);② 三态 allow/ask/deny,**拿不准一律 ask 交给人**;③ 静态规则(正则/前缀)不是真安全 —— 能被变量拼接绕过 —— 所以尽力检测 + 人审兜底;④ prompt injection:不信任何工具返回内容里的"指令"。

**展开版**:
- 我的 permission.ts:只读工具永远放行;bash 按命令内容分级(确信安全 allow / 确信危险 deny / 其余 ask);deny 直接拒**连问都不问** —— 防"手滑按了 y"造成不可逆后果,我有测试断言 deny 时 approve 回调根本不被调用。
- 我特意写了"诚实边界"测试:`X=rm; $X -rf /` 绕过 DANGER 正则后落进 **ask 而非 allow** —— 静态检测漏了,还有人兜底。**测自己的局限比假装没有局限更工程。**
- 真 CC 的深度:判定优先级链 deny → ask → 工具自查 → bypass → allow,**deny 最高连 bypass 模式都越不过,但兜底是 ask 不是 deny**(哲学:系统不替用户做绝对禁止);正则尽头是 LLM 分类器(yoloClassifier,fast/thinking 两级理解语义);写 .git/.claude 是 bypass-immune 强制问。

**追问防守**:
- *"prompt injection 具体怎么防?"* —— 见通识 T1,核心:指令与数据分信任边界,权限门管的是**行为**(不管模型被谁说服,写文件就要过门)。
- *"为什么真 CC 不内置 rm -rf → deny?"* —— 硬 deny 全来自用户配置;系统的立场是"拿不准就问",绝对禁止是用户的决定。
- *"沙箱呢?"* —— 我没做(诚实),真 CC 有 sandbox 模式;权限门是"逻辑层刹车",沙箱是"物理层护栏",生产两个都要。

## Q8 怎么评估一个 agent 好不好?

**30 秒版**:按确定性分层:**确定的代码走单测/CI 硬门;非确定的 agent 行为走离线 eval**(固定任务集 + 判"达成"不判"过程" + 看通过率/轮数/token 统计),**不当 CI 硬门**;开放质量用 LLM-judge;线上靠成本可观测;人工兜底。

**展开版**:
- agent 评估的三难:非确定(过程每次不同)、多种正确答案、多轮+副作用。解法 = 判最终状态(文件内容/输出关键词,宽松匹配),不卡过程和措辞。
- 我的 mini-eval:5 任务覆盖 write/edit/搜索/MCP/子 agent,每任务独立 scratch 目录隔离,实测 5/5、平均 2.6 轮;另对一个任务加 LLM-judge 按 rubric 打分 —— 程序化判"有没有",judge 判"好不好"。
- 反直觉发现:真 CC **没有把 agent 任务 eval 做成 CI 门** —— 非确定性做硬门必然 flaky,团队会对红灯麻木。它的重心在运行时 cost-tracker(token/成本/时长跨会话持久化)和确定性 quality-gate(覆盖率基线对比防退化 + flaky 隔离区带 owner 和复查期)。
- 我的 D10 测试套就是这套分层的落地:67 个确定性测试上 CI,LLM 行为归 eval 本地跑。

**追问防守**:
- *"flaky 测试怎么办?"* —— 隔离区(quarantine):不删不 skip,带 owner、复查期、退出标准。
- *"LLM-judge 可信吗?"* —— rubric 固定 + 只评开放质量 + 抽样人工对齐;judge 漂移本身也要被 eval。
- *"改了 prompt 怎么防退化?"* —— eval 集当回归测试跑,对比通过率和成本曲线,不看单次。

## Q9 streaming 为什么重要?怎么实现可中断?

**30 秒版**:流式降首 token 延迟、用户能中途打断省钱省时。可中断的核心是理解 **AbortController 只是取消的"协议层",它自己什么都不停** —— 每个持有资源的环节要自己 opt-in:模型流靠 fetch 原生支持,子进程靠手动监听 abort 去 kill。

**展开版**:
- 我的实现:`.stream()` + `on("text")` 边吐字,每轮新建 AbortController,Esc 触发 abort —— 一把 signal 两个监听者。
- 最深的坑是"**杀父不杀孙**":kill 只杀直接子进程 bash,孙子进程(如 sleep)还攥着 stdout 管道写端,管道不 EOF → 读取 await 傻等。修法 = `detached:true` 让 bash 自成进程组 + `process.kill(-pid)` 杀整组 + abort 即 resolve 不等管道。实测工具中断 2.0s、模型流中断 2.5s;D10 有回归测试(sleep 5 在 100ms 中断、2s 内返回)。
- 真 CC 的生产加固(多 6 处):中断后**补 tool_result**(否则孤儿配对害下一轮 400)、idle watchdog、裸流避 O(n²) 重复 parse、边流边执行工具。

**追问防守**:
- *"中断后对话还能继续吗?"* —— 能,前提是把中断点的 tool_use 补上占位 tool_result,配对完整才合法。
- *"SSE 和 WebSocket 怎么选?"* —— 模型输出是单向流,SSE 够;WebSocket 是双向场景(协作、语音)才需要。
- *"流式怎么和工具执行配合?"* —— 我的是攒齐 finalMessage 再执行;真 CC 有 StreamingToolExecutor 边流边执行抢时间。

## Q10 模型选择与成本权衡?

**30 秒版**:按任务分层路由:主对话用强模型,批量/调研/分类用小模型。成本三板斧 = **caching(读 0.1×)+ 压缩截断 + 小模型分流**,再加换模型必跑 eval 回归防退化。

**展开版**:
- 分层路由的三个实证:真 CC 的 Explore 子 agent 对外部用户跑 **haiku**(调研型任务量大、单步简单);权限 LLM 分类器 fast/thinking 两级(先便宜的,拿不准才升级);Anthropic 官方 Contextual Retrieval 用 haiku 批量给 737 个 chunk 生成上下文,配合 caching 把成本从 $9.2 压到 $2.85(省 69%)。
- 成本可观测是前提:真 CC 的 cost-tracker 按模型分档单价累计 token/成本/时长,cache read = input 的 1/10 直接印证 caching 收益。
- 可靠性是同一枚硬币的另一面:我写过 retry-fallback demo —— **可重试判定才是灵魂**(429/5xx/超时才重试,400/401 立即放弃,死磕只会放大故障),指数退避 + jitter 防惊群,overloaded 时 withFallback 切备用模型(SDK 自带重试但换模型降级要自己做)。

**追问防守**:
- *"怎么知道换便宜模型没退化?"* —— eval 集跑回归,看通过率+轮数+成本三条曲线(D8);感觉不算数。
- *"小模型分流的边界?"* —— 单步判断/格式转换/批量生成可下放;多步规划、工具编排别下放 —— 省的钱会在重试和错误里加倍还回来。

## 压轴题:你会怎么设计一个 Claude Code?

**答题路线**(这题的正确姿势是掏出自己的架构图):

1. **画三层**:外壳(只管交互:键盘中断/审批/账单)→ 内核 runAgent(流式 → 权限门 → 执行 → 回填 → 压缩 → 循环)→ 工具层(注册表即数组)。两条关键连线:执行指向工具层;工具层里的 task 又**递归指回内核** —— 子 agent 就是包装成工具的整个 loop。
2. **按数据流走一遍**:用户输入进 messages → 模型回 tool_use → 权限门三态 → 执行(错误也回填)→ 超阈值压缩(配对安全切割)→ 循环到 end_turn。
3. **报增量演进路径**:先 20 行裸 loop 跑通,再按痛点长能力 —— 慢(流式)、贵和爆(caching+压缩)、危险(权限)、封闭(MCP)、上下文污染(子 agent)。每步是被真实痛点逼出来的,不是堆特性。
4. **主动报"生产级还差什么"**:多 provider 降级、沙箱(权限门只是逻辑层刹车)、IDE/LSP 集成、hooks/插件、多级压缩、团队级 eval 与可观测。
5. 收尾一句:"这几块我都写过最小实现,并逐块对照过真 Claude Code 的源码取舍 —— 差距清单就是我下一步的路线图。"

## 四道通识补充(口头准备,不写代码)

### T1 prompt injection 怎么防?

骨架:**信任边界** —— 用户指令是指令,工具返回/网页/文件内容是**数据**,数据里出现"指令样"文本一律不执行。防线三层:① 权限门管**行为**不管说辞(不管模型被谁说服,写文件就要过 ask);② 敏感操作人工确认,且确认通道要防模型自答(真 CC 的 channelPermissions 把审批转发到手机,靠短 ID 回复防自批准);③ 输出侧最小权限(只读工具集、路径限制)。诚实一句:没有 100% 的防注入,只有"注入成功后它能干什么"的爆炸半径控制。

### T2 幻觉 / grounding 怎么治?

骨架:① **约束生成**:system 严格限定"只能用参考资料回答,资料里没有就说不知道" —— 我的 ragAnswer 就这么写,副作用是**召回质量成了答案上限**(garbage in);② **引用溯源**:要求答案标注来源片段,可校验;③ **工具结果当 ground truth**:能查的不要背(算术给计算器,数据给 API);④ 分场景容忍度:创意任务幻觉是特性,事实任务是 bug —— eval 判据要区分。

### T3 微调 vs RAG 怎么选?

骨架:按"缺的是什么"选:缺**知识**(时效性/私有数据)→ RAG,随时更新、可溯源;缺**风格/格式/领域语言** → 微调;缺**行为能力**(用工具/守流程)→ 先 prompt + 工具编排,都不够再微调。成本与迭代速度排序:prompt < RAG < 微调(微调要数据管线+训练+部署+每次更新重来)。经验法则:**90% 的"想微调"其实是 prompt 或检索没做好**;先把便宜的杠杆拉满。

### T4 常见 agent 设计模式?

骨架(每个都指向自己写过的锚点):**ReAct** = 思考→行动→观察循环,就是我的 M1 loop;**orchestrator-worker** = 主派子收结论,我的 M7,动机是 context 隔离;**plan-execute** = 先只读探索出计划、人批准后执行,真 CC 的 plan 模式(软约束只读 + ExitPlanMode 是唯一正门);**Reflection / evaluator-optimizer** = 生成后自评再改,我的 LLM-judge 是雏形,真 CC 的 2-stage 权限分类器(fast 拿不准升 thinking)也是这个形状。选型心法:模式是手段不是目标,**从单 loop 开始,被具体痛点逼一步才加一层**。
