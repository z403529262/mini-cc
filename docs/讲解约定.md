# 讲解约定（What → Why → How）

> 本教程（mini-cc，agent 工程师面试备战）的讲解与笔记规范。既约束口头讲解，也约束所有 D 系列笔记的组织方式。
> 来由：2026-06-17 用户反馈——讲解过于偏重"怎么办"，"是什么/为什么"几乎不主动解释，要追问才补。

## 核心约定

**顺序：是什么（What）→ 为什么（Why）→ 怎么办（How）。**
**重心：What / Why 是主体，主动讲透；How 其次、从简。**

## 为什么这么定（这条约定本身的 Why）

面试考的是**概念理解与动机**，不是代码默写。考官问的是"什么是 prompt caching""为什么需要上下文压缩""AbortController 的本质是什么""为什么压缩会炸缓存"——他要听你**理解了本质和权衡**。所以：

- **What（本质定义）**：这东西到底是什么，先把概念立住，给它在整个体系里的位置。
- **Why（动机）**：它要解决什么问题？不用它会怎样？——面试最爱问、最能体现理解深度的部分。
- **How（实现）**：知道了 what/why，怎么做往往是自然推论，点到为止。

## 反例（要避免）

一上来就抛代码 / "正确做法" / "cc-haha 怎么防"。读者还不知道这是什么、为什么需要，就被灌一堆 how，只能反复追问"没看懂"——这正是这条约定要纠正的。

## 正例

先 **"X 是什么"**（一句话本质 + 它在体系里的位置）→ 再 **"为什么需要 X / 没有它会怎样"**（动机、痛点、一个具体反例）→ 最后 **"怎么实现"**（最小代码 / 关键点，简略）。

每个概念至少配齐三样：**本质定义 + 动机 + 一个"不这样会怎样"的反例**。这三样讲清，how 自然就懂了。
