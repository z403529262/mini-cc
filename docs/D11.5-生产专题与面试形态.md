## 一、记忆系统设计(2026 大热题,重点准备)

**30 秒版**:上下文窗口只是**工作记忆**,会话结束即失忆;产品要"越用越懂用户"就需要跨会话记忆。记忆系统 = 写路径(何时记、记什么、怎么去重)+ 存储(摘要式/向量式/结构化文件式)+ 读路径(何时召回、怎么注入)。**难点在写,不在读**——读只是检索,写要做判断。

**分类框架**(认知科学的词,落到工程):
- working:上下文窗口内(D4 的压缩/缓存管的就是它)
- episodic:历史会话记录,可检索回放
- semantic:沉淀下来的事实与偏好("用户是 Android 转 iOS 背景")
- procedural:做事规则,直接反哺 system prompt("讲解 what/why 优先")

**三条实现路线**:

| 路线 | 机制 | 优劣 |
|---|---|---|
| 摘要式 | 会话结束生成摘要,下次开场注入 | 便宜简单;粗粒度,细节丢失 |
| 向量式 | 记忆切片入向量库,语义检索召回(mem0 一类) | 细粒度;继承 D9 全部 RAG 问题:过期、冲突、召回质量是上限 |
| **结构化文件式** | LLM 自己读写 markdown 文件 + 索引常驻 | Claude Code auto-memory 就是;**我每天在用,可以细讲** |

**我的一手案例(差异化,别人没有)**:Claude Code 的 memory = 一个目录,**一文件一事实**(带 frontmatter:name/description/type),外加一个 MEMORY.md 索引——**索引每次会话常驻上下文(一行一条),正文按需再读**。这个"索引常驻、内容懒加载"的模式,和 MCP 工具的 defer_loading、和 skills 清单机制是同构的——**上下文经济学的通用解:目录放前台,内容放后台**。我的两周备战进度就是靠它跨几十个会话接力的,它工作得如何我有体感。

**写入侧的四个决策(难点所在)**:
1. 何时写:用户显式"记住 X" / 会话结束反思提取 / 关键纠错即时写(用户纠正过的错误最值得记)
2. 去重:写前先查同主题旧记忆,**更新优于新建**
3. 冲突:新事实推翻旧事实要删旧的,不是并存
4. 过期:记忆带时间戳;**用前验证**——我们自己的运维规矩就是"记忆里提到的文件/开关,用前先确认还存在"

**失败模式(说出这些比说功能更显深度)**:
- **记错比不记糟**:过期记忆会自信地误导(我们真实吃过亏:凭旧印象报数被"数字必经实测"铁律纠正)
- 隐私:记忆必须用户可见、可改、可删
- 无限增长:要定期 consolidate(合并/清理),否则索引本身撑爆上下文
- **记忆污染是 prompt injection 的新面**:不可信内容(网页/工具结果)绝不能直接写入记忆——否则一次注入永久生效

**追问防守**:
- *"记忆和 RAG 什么关系?"* —— 记忆是**读写闭环**,RAG 只是读路径的一种实现;记忆的本质难题(何时写/怎么改/何时忘)RAG 根本不管。
- *"多用户怎么隔离?"* —— per-user namespace 是底线;更深一层是"组织记忆 vs 个人记忆"的分层(类比 CC 的 global CLAUDE.md vs 项目 memory)。

## 二、长时任务 / durable execution

**30 秒版**:agent 跑小时级任务,进程崩了/部署重启怎么办?答案的地基是:**messages 数组本身就是完整的状态机状态**——每轮结束落盘即 checkpoint,恢复 = 加载数组重进 loop。真正难的是副作用工具的**幂等性**:恢复重放时,已执行过的 tool_use 不能再执行一遍。

**要点**:
- 一手实证:Claude Code 的会话就是这么存的——jsonl 按行追加,resume = 重放文件。我研究过它的存储结构(会话内容与列表分离、靠 sessionId+cwd 关联),迁移过整机会话。
- 幂等两条路:loop 侧记"已完成 tool_use_id 集合",重放跳过;或工具本身幂等设计(退款接口带唯一单号,重复调用不重复扣)。
- 报菜名:LangGraph checkpointer(把 checkpoint 做成框架一等公民)、Temporal(通用 durable execution 引擎)、CozeLoop(国内对应)。
- 后台 agent(2026 常态:cron 触发、异步跑完通知)的连带要求:**无人值守 = 权限模型必须收紧**——自动批准范围白名单 + 预算熔断(maxTurns/token 上限),这是 D5 权限门和 D10 熔断测试在生产维度的直接延伸。

**追问防守**:
- *"压缩过的历史怎么 checkpoint?"* —— 压缩本身就是状态变更,落盘压缩后的 messages 即可;要审计原始历史就另存全量日志(状态与日志分离)。
- *"并行子 agent 怎么恢复?"* —— 父的 checkpoint 记录"哪个 task 调用未完成",恢复时重派——tool_use_id 就是天然的任务 id(接 D7)。

## 三、Guardrails(和权限门分清楚)

**30 秒版**:权限门管**做什么**(行为:写文件、执行命令),guardrail 管**说什么/进什么**(内容:输入过滤、输出安全)。输入侧:injection 检测、话题分类(离题拦截);输出侧:PII 泄漏、内容安全、格式校验。实现套路 = 规则先行 + 小模型分类器兜底——和真 CC 权限的 yoloClassifier 2-stage(便宜的先跑,拿不准升级)完全同构。

**要点**:
- 立场句:**guardrail 是概率的,权限门是确定的——确定的守底线,概率的降风险**。生产两个都要,谁也替不了谁。
- structured output 也是一种输出侧 guardrail(schema 校验挡住格式漂移)——D9 的 tool 强制 schema demo 就是最小实现。
- OpenAI Agents SDK 把 guardrails 做成一等公民(与主流程并行跑、tripwire 触发即熔断)——框架界已把它当标配。

**追问防守**:
- *"guardrail 误杀正常请求怎么办?"* —— 和权限门同一个哲学:拿不准别硬拦,降级到"标记+人审";误杀率本身要进 eval 指标。

## 四、Tracing / 可观测(报菜名 + 最小版)

**30 秒版**:trace = 一次任务,span = 一轮/一次工具调用/一次 LLM 调用,span 上挂 token/延迟/成本属性;OTel 的 GenAI 语义约定正在标准化这层。工具:LangSmith(LangChain 系托管)、Langfuse(开源可自部署)、CozeLoop(国内)。**agent 特别需要 trace 的原因:非确定 + 多轮,"错在第几轮哪个工具"没有 trace 根本无法还原。**

**我方锚点**:D8 的 usage/turns 聚合是最小版可观测;cc-haha 的 cost-tracker(跨会话成本持久化、按模型分档单价)是产品版——我知道从最小版到产品版之间差什么:span 树、采样策略、成本归因到功能。

## 五、系统设计骨架:"设计一个客服 agent"

> 系统设计轮不一定考"设计 Claude Code"。客服 agent 是最高频变体——用这个骨架,换皮可答"知识库助手/数据分析 agent/运维 agent"。

**第一步永远是澄清需求(2 分钟,不许跳)**:渠道(IM/电话/网页)?量级 QPS?SLA?知识源(FAQ/工单史/产品文档)?允许执行什么操作(只查 vs 能退款)?合规要求(金融/医疗)?

**架构分层(画图讲)**:
1. **入口分流**:小模型意图分类(便宜、快)→ FAQ 直答走 RAG(D9 全套:混合检索+rerank)/ 任务型走 agent loop / 敏感或高危直转人工
2. **agent 层**:工具分级——查订单(只读,自动放行)vs 退款(副作用:小额度内自动+幂等单号,超额度过审批)——**这就是 D5 权限门三态在业务里的样子**;记忆(用户画像+本会话上下文);输出 guardrail
3. **human handoff(必答项)**:触发条件 = 置信度低 / 用户明确要求 / 规则命中(投诉、法务)/ N 轮未解决;**关键设计是上下文打包**——转人工时带摘要+已尝试步骤,不让用户重讲一遍(这是压缩能力的业务复用)
4. **评估闭环**:离线 τ-bench 式(工具调用正确率+政策遵守率,模拟用户跑回归)+ 线上(自助解决率/转人工率/满意度/单会话成本)——D8 分层直接套
5. **成本与降级**:分流已经省了大头;高峰降级链 = 超时 → 缓存答案 → 排队人工

**与"设计 Claude Code"的差异点(面试官在听你有没有生产 sense)**:多租户(数据与成本隔离)、SLA(编码 agent 可以慢,客服不行)、合规审计日志(每条建议可回溯)、知识更新管线(产品改版了 FAQ 怎么同步——RAG 的新鲜度问题,D9 讲过的结构性缺陷在这里是日常)。

**万能收尾**:"先上最小闭环(分流+RAG+转人工),拿线上数据再决定要不要上任务型 agent——**eval 数据驱动演进,不是一次设计到位**。"

## 六、Computer use 一页(2026 热点)

**30 秒版**:模型看截图、输出坐标和动作(点击/键入),loop 与编码 agent 完全同构——**只是工具从 read/write/bash 换成了鼠标键盘,观察从文本换成了截图**。现状(核验 2026-06):短程任务(OSWorld-Verified)头部 ~85% 已可用;**长程任务(OSWorld 2.0,人类中位 1.6 小时)榜首只有 20.6%**——短程能用,长程还早。

**要点**:
- 失败模式四条(引 OSWorld 2.0,也是通用 agent 短板清单):丢约束、漏掉中途到达的新信息、瞎猜而不是问用户、跳过验证。
- 工程要点:一步一截图 → 延迟与 token 成本都高;浏览器场景优先拿 DOM/accessibility tree(结构化、便宜、稳),纯像素是兜底——**和 D9 "有结构就别用模糊匹配"同一哲学**(LSP>grep>向量 的 GUI 版)。
- 安全:不可逆操作(付款/发送/删除)必须确认——D5 的 ask 哲学在 GUI 上更重要,因为误操作面更大。

## 七、面试形态:STAR 故事 × 3 + 学习方法 + 反问

### STAR 1:杀父不杀孙(技术深度型,首选)
- **S**:mini-cc 的 bash 工具加了中断,实测按 Esc 后却要傻等满 10 秒超时。
- **T**:找到根因,做到中断即返回。
- **A**:逐层排查——kill 确实杀掉了 bash,但孙子进程(sleep)还活着,攥着 stdout 管道写端,管道不 EOF,读取的 await 就永不返回。修法:`detached:true` 让 bash 自成进程组 + `kill(-pid)` 杀整组 + abort 立即 resolve 不等管道。
- **R**:中断从 10s 傻等变 2.0s 实测;后来在 D10 写成回归测试(sleep 5 在 100ms 中断、2s 内必须返回)。
- **亮点句**:"这个 bug 教会我:AbortController 只是取消的协议层,它自己什么都不停——每个持有资源的环节都要自己接住信号。"

### STAR 2:cache_control 残留 400(读 API 约束型)
- **S**:做 prompt caching 时多轮对话后突然 API 400。
- **T**:定位为什么只在多轮后炸。
- **A**:发现我直接在原 messages 上打 cache_control,断点逐轮累积,超过 4 个断点上限即 400。修法:每次发送前做**副本**打点,原数组永远干净。
- **R**:实测命中率 83.8%;顺手理解了"断点是滚动的,缓存是嵌套前缀"的模型。
- **亮点句**:"教训是不可变数据的思维——发送视图和持有状态要分离。"

### STAR 3:幻觉数据教训(工程化怀疑型,展示"用 AI 工作"的元能力)
- **S**:用 AI 做存储芯片研究,初版报告里美光 PE 写了 41 倍——是模型凭记忆编的,实测只有 7.3 倍。
- **T**:防止此类错误再次发生,而不是只改这一处。
- **A**:立下硬规矩"任何数字必经工具实测、结论附核验痕迹",并把这条纪律做成 skill 植入本机全部 4 个 AI agent 的全局指令,金融类问题强制触发。
- **R**:此后数据类错误由流程拦截,不靠人肉记性。
- **亮点句**:"对 AI 输出的正确态度不是信或不信,是**给它配验证流程**——这也是我做 agent eval 的动机来源。"

### "你怎么快速学新领域?"(拿 D9 那条真实追问链当模板)
六步:官方一手材料(cookbook)→ clone 源码逐行读 → **实测验证宣称**(jq 抓数据对账,发现"9 个代码库"实为 90 个文件)→ 批判性读 benchmark(发现 Elastic 对照组是稻草人)→ 映射回已有知识框架(contextual retrieval = 经典六步只改 embed 一步)→ 沉淀成文档和测试。一句话版:**"我用两周从零到手写出一个带测试和 CI 的 Claude Code 最小实现——这条流水线本身就是答案。"**
